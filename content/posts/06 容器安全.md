# 容器安全

### 一、容器安全的两个层面
#### 1. **架构层面**

- **容器与宿主机共享 Linux 内核**：容器并非独立操作系统，依赖宿主机的内核，因此内核漏洞可能影响所有容器。
- **Namespace 隔离资源**：通过 Namespace（如 PID、Network、Mount）隔离进程、网络、文件系统等资源。
- **通过 shim 和 runC 启动容器**：容器运行时（如 Docker）通过 shim 和 runC 工具启动容器，架构层面的安全性由运行时实现。
- **用户无法直接修改架构设计**，需通过运行时配置提升安全性。

#### 2.  **容器启动时的权限控制**

用户可通过以下方式主动增强容器安全性：

- **控制容器的 Linux Capabilities**
- **以非 root 用户运行容器**

### 二、控制容器的 Linux Capabilities
#### 1. **Linux Capabilities 的作用**

- 内核 2.2 后引入，将 root 用户的特权细化为多个独立权限（如 `CAP_NET_ADMIN` 管理网络、`CAP_SYS_ADMIN` 管理文件系统）。
- 默认容器内的 root 用户**仅拥有部分** Capabilities，需通过 `--privileged` 参数获取全部权限（不推荐，风险大）。

#### 2. **示例：容器内 root 用户权限受限**

> - **Dockerfile**：
>   ```dockerfile
>   FROM centos:7
>   RUN yum install -y iptables
>   ```
> - **构建镜像并运行容器**：
>   ```bash
>   docker build -t test_cap:v1.0 .
>   docker run -ti --rm --name test test_cap:v1.0 sh
>   ```
> - **执行 `iptables` 命令报错**：
>   ```bash
>   sh-4.2# iptables -L -n
>   Permission denied (you must be root)
>   ```
>   即使容器内用户为 root，仍缺少 `CAP_NET_ADMIN` 权限。

#### 3. **解决方案**

- **赋予特定 Capability**：
  ```bash
  docker run -ti --rm --cap-add NET_ADMIN --name test test_cap:v1.0 sh
  ```
  此时容器内可正常执行 `iptables` 命令。
- **禁用 Capability**（宿主机测试）：
  ```bash
  capsh --keep=1 --user=root --drop=cap_net_admin -- -c "iptables -L"
  ```
  报错提示权限不足，验证 Capabilities 的细粒度控制。

### 三、以非 root 用户启动容器
#### 1. **安全问题**

- **容器内 root 用户的风险**：

  即使与宿主机 root 用户隔离，容器内 root 仍可能修改挂载的宿主机文件（如 `/etc/shadow`）。

- **示例**：
  ```bash
  docker run -ti --rm -v /etc:/etc --name test centos:7 sh
  ```

#### 2. **解决方案**

- **方案一：`-u` 参数指定非 root 用户**
  ```bash
  docker run -ti --rm -u 1000:1000 -v /etc:/etc --name test centos:7 sh
  ```
  - 容器内用户无写权限，操作 `/etc/shadow` 报错。
  - **缺点**：多个容器共享同一 UID 可能导致资源竞争（如文件句柄限制）。

  **补充**：

  在dockerfile中也可以指定非root用户：

  ```dockerfile
  FROM centos:7
  RUM useradd tracy
  USER tracy
  ```

- **方案二：User Namespace 隔离**

  - **原理**：**容器守护进程以宿主机 root 用户运行**，但利用 Linux Kernel 的 `user_namespace`功能，映射容器内 UID 到宿主机非特权 UID 范围（如容器内 0-1999 映射到宿主机 30000-31999）。
  - **Podman 实现**（默认启用 User Namespace，但在此处是用 `--uidmap`/ `--gidmap` 手动控制映射）：

    ```bash
    # 该命令以宿主机 root 用户运行
    podman run --uidmap 0:30000:2000 -ti centos:7 sh
    ```

- **方案三：Rootless 容器**（优先推荐在生产环境使用[尤其在公有云/K8s 场景]）

  - 使用**非 root 用户**启动容器（如 Podman 支持），彻底避免权限提升风险。

    ```bash
    # 以普通用户身份运行
    podman run -ti -v /etc:/etc centos:7 sh
    ```

### 四、总结：容器安全最佳实践
- **最小化 Capabilities**
   - 仅添加必需的 Capabilities（如 `--cap-add NET_ADMIN`），避免使用 `--privileged`。

- **非 root 用户运行**
   - 通过 `-u` 或 Dockerfile 的 `USER` 指令指定非 root 用户。
   - 优先使用 User Namespace 或 Rootless 容器（如 Podman）。

- **限制挂载敏感目录**
   - 避免挂载 `/etc`、`/dev` 等关键目录，或设置只读挂载（`-v /etc:/etc:ro`）。

- **选择安全的容器运行时**
   - 使用支持 User Namespace 的工具（如 Podman），替代传统 Docker 命令。

#### 附：Capabilities 关键点
| Capability         | 权限说明                   |
| ------------------ | -------------------------- |
| `CAP_NET_ADMIN`    | 网络配置（iptables、路由） |
| `CAP_SYS_ADMIN`    | 文件系统挂载、系统调试     |
| `CAP_DAC_OVERRIDE` | 绕过文件权限检查           |

通过精细控制 Capabilities 和用户权限，可显著降低容器逃逸和特权提升风险。
