## 一、容器进程管理

### 1.**容器内的进程组织**

可使用 `docker top` 查看容器内所有**正在运行**的进程（含它们在宿主机中的 pid 和 ppid 等信息）

> 但通过 `nsenter -t` （用于进入另一个进程的命名空间）进入容器启动的进程可能无法查看。因为虽然该进程在容器中显示的 `ppid` 是0（方式2才会是0，方式1的话会是shell的pid），但实际上该0进程是在容器外部的进程（即nsenter的进程），而 `docker top` 命令仅显示容器内的进程信息，它不会显示通过 `nsenter` 在容器外部启动的进程（如果需要在宿主机上查看通过 `nsenter` 启动的进程，应该直接在宿主机上使用 `ps` 或 `top` 等命令）。
>
> 当你通过 `nsenter -t <pid>` 进入容器并启动一个新进程时，新进程的父进程 ID（ppid）取决于你在容器内启动进程的方式。
>
> 方式一：在容器内通过一个 shell 启动新进程（例如，通过 `bash` 或 `sh`），那么新进程的父进程将是那个 shell 进程。
>
> 方式二：如果你直接通过 `nsenter` 启动一个进程（不通过 shell），那么新进程的 ppid 将是 `nsenter` 进程的 pid，前提是 `nsenter` 在容器内没有指定其他进程作为新进程的父进程。

- **0号进程：`containerd-shim`**

   - **角色与功能**
     - 由 `containerd-shim` 调用 `runc` 创建容器，是容器操作系统的**祖宗进程**。
     - 负责接收和处理容器内执行的指令，生命周期与容器绑定：**若 `containerd-shim` 终止，容器立即退出**。
   - **核心特点**
     - **父子关系管理**
        - 容器内所有进程的父进程。
        - 若容器内 **1号进程结束**，`containerd-shim` 会回收整个容器命名空间中的进程。
     - **信号处理与容器终止**
        - `docker stop` 会向容器内的 **1号进程发送 `SIGTERM`（-15）信号**：
          - 若 1 号进程**无信号转发能力**：直接向容器内所有进程发送 `SIGKILL`（-9）强制终止。
          - 若 1 号进程**有信号转发能力**：将 `SIGTERM` 转发给命名空间内所有进程，允许优雅退出。

- **1号进程：容器内的首个用户进程**
   - **角色与功能**
     - 代表容器的生命周期：**1号进程终止，容器终止**。
     - **必须在前台一直运行**
       - 如果1号进程被设置为在后台运行（例如，通过在命令后面添加`&`符号），它将不会阻塞终端，并且容器启动后，终端会立即返回，允许执行其他命令。
       - 在这种情况下，如果1号进程完成了它的任务并退出，由于它是容器内唯一的前台进程（或者是最顶层的进程），容器没有其他前台进程可以关联，因此容器会认为没有进程需要保持运行，从而会停止。

   - **与宿主机的关键差异**：

     - **进程管理局限性**
        - **非全权管理**：
          - 不一定是容器内所有用户进程的祖先。
            - 当容器已经启动后，通过 `exec` 进入容器执行命令产生的新进程都是0号进程的儿子。
          - 孤儿进程（1号进程的后代，但非直接父子关系）会被1号进程**收养**，**但可能无法正确处理**（回收）。
        - **僵尸进程风险**：若 1 号进程未实现回收逻辑，僵尸状态的子进程会累积（可能引发资源泄漏）。
     - **信号处理缺陷** （可能每实现对应的逻辑）
        - 用户自定义的 1 号进程可能**无法转发信号**（如 `SIGTERM`），导致容器无法优雅终止。

   - **理想能力**
     - **僵尸进程回收**：定期调用 `wait/waitpid` 回收终止的子进程。
     - **信号转发**：将接收到的信号（如 `SIGTERM`）传递给子进程。

   - **解决方案**
     - 使用轻量级初始化工具（如 `tini`）作为 1 号进程，由其管理实际应用进程（如 `tini → python app`），实现僵尸回收和信号转发。

       ```dockerfile
       ENTRYPOINT ["tini", "--"]
       CMD ["python", "app.py"]
       ```

### 2.**与宿主机操作系统的对比**

| **对比维度**      | **容器内进程**                                               | **宿主机进程**                                               |
| :---------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **0号进程**       | `containerd-shim`（用户态进程，由 `containerd` 启动，以调用 `runc` 创建容器） | 内核创建的 0 号进程（内核态，负责进程调度和资源管理，是操作系统启动的起点） |
| **1号进程**       | 用户指定（如 `bash`、`tini` 或自定义程序）                   | 系统初始化进程（如 `systemd`、`init`，以**守护进程**形式运行） |
| **生命周期绑定**  | 1号进程终止 → 容器退出；<br />0号进程（`containerd-shim`）终止 → 容器强制退出 | 1号进程终止 → 操作系统崩溃；<br />0号进程终止 → 系统崩溃（理论上不可发生） |
| **进程树结构**    | 1号进程不一定是所有用户进程的祖先（孤儿进程可能被 1 号或 0 号进程收养） | 1号进程是所有用户进程的祖先（严格父子关系）                  |
| **僵尸进程回收**  | 依赖 1 号进程实现（若未实现，僵尸进程会累积；需工具如 `tini` 辅助）<br />若1号进程终止，会由0号进程来收养它的子进程，并回收 | 1号进程自动调用 `wait/waitpid` 回收僵尸进程                  |
| **信号处理**      | 若 1号进程未处理信号： `docker stop` 发送 `SIGTERM` 失败后，直接发送 `SIGKILL` | 1号进程自动转发信号（如 `SIGTERM`）至子进程                  |
| **前台/后台运行** | 1号进程必须在前台运行（否则容器退出）                        | 1号进程以守护进程形式运行（无需前台占用）                    |
| **典型问题**      | - 僵尸进程累积 <br />- 信号未转发导致强制终止 <br />         | - 僵尸进程极少（系统自动回收） <br />- 信号处理链完整        |
| **解决方案**      | 使用 `tini`、`dumb-init` 等工具作为 1号进程，实现信号转发和僵尸回收 | 无需额外工具，依赖操作系统本身的进程管理机制                 |

**关键总结**

1. **容器生命周期绑定**
   - **0号进程**是容器存在的标志（`containerd-shim` 终止 → 容器退出）。
   - **1号进程**是容器内用户空间的起点（1 号终止 → 容器退出）。
2. **僵尸进程风险**
   - 容器内需确保 1 号进程具备**僵尸回收能力**（如通过 `tini` 或代码实现）。
3. **优雅终止依赖信号处理**
   - `docker stop` 的优雅终止需 1 号进程**转发 `SIGTERM`**，否则触发强制终止。

### 3. 使1号进程具备理想能力的解决方案

#### 3.1 容器外限制：配置PID Cgroup

1. **原理**
   - 通过 Linux 的 `Cgroup` 机制限制容器内允许的最大进程数，从根源上避免PID耗尽。

2. **操作步骤**
   - 在宿主机上为容器配置 `pids.max` 参数：
     ```bash
     # 使用 Docker 运行时直接限制
     docker run --pids-limit 100 <image>

     # 手动配置 Cgroup（示例）
     echo 100 > /sys/fs/cgroup/pids/system.slice/docker-<容器ID>.scope/pids.max
     ```
   - **效果**：无论容器内进程是正常创建还是Bug导致，一旦达到阈值，新进程将无法创建。

#### **3.2 容器内优化**

##### **1. 针对开发者：修正代码逻辑（以Python为例）**
- **目标**：确保主进程主动回收子进程。
- **关键点**：
  1. **使用 `join()` 方法**
     ```python
     import multiprocessing as mp

     def worker():
         print("子进程执行...")

     if __name__ == "__main__":
         p = mp.Process(target=worker)
         p.start()
         p.join()  # 等待子进程结束并回收
     ```
     - `p.join()` 调用 `wait()` 系统调用，阻塞主进程直至子进程结束并回收。

  2. **避免主进程阻塞**
     - **错误示例**：主进程未调用 `join()` 且进入阻塞状态（如 `sleep`）：
       ```python
       p.start()
       time.sleep(1000)  # 主进程阻塞，无法回收子进程
       ```
     - **结果**：子进程结束后成为僵尸进程。

  3. **多进程场景的通用方案**
     - 维护子进程列表并逐个调用 `join()`：
       ```python
       processes = []
       for _ in range(10):
           p = mp.Process(target=worker)
           p.start()
           processes.append(p)

       for p in processes:
           p.join()
       ```

##### **2. 针对维护者：引入具备回收能力的1号进程**
- **场景**：主进程未实现僵尸回收逻辑（如遗留代码或第三方软件）。
- **核心思路**：替换容器的1号进程为具备回收能力的工具（如 `bash` 或 `tini`），由其管理实际业务进程。

**方案一：使用 `bash` 作为1号进程**

- **原理**
  - `bash` 会定期调用 `waitpid` 回收子进程（仅限直接子进程）。
- **操作步骤**
  1. **编写启动脚本**（`entrypoint.sh`）：
     ```bash
     #!/bin/bash
     # 启动实际业务进程（作为子进程）
     python app.py
     ```
  2. **在Dockerfile中配置**：
     ```dockerfile
     COPY entrypoint.sh /entrypoint.sh
     ENTRYPOINT ["/bin/bash", "/entrypoint.sh"]
     ```
- **优点**：简单易用，无需额外依赖。
- **缺点**：
  - 仅回收直接子进程（无法处理孙子进程）。
  - **无信号转发能力**：`docker stop` 发送 `SIGTERM` 时，`bash` 不会转发信号给子进程，可能导致强制终止。

**方案二：使用 `tini` 作为1号进程（推荐）**
- **原理**
  - `tini` 是一个轻量级 `init` 系统，专为容器设计，兼具僵尸回收和信号转发能力。
  - **核心功能**：
    1. **定期回收僵尸进程**：调用 `waitpid(..., WNOHANG)` 非阻塞回收所有子进程（包括孙子进程）。
       - `wait()` 总是阻塞的，等待任意一个子进程结束。
       - `waitpid()` 默认是阻塞的，但可以通过 `WNOHANG` 选项使其变为非阻塞，并且可以指定等待哪个子进程。
    2. **信号转发**：将接收到的信号（如 `SIGTERM`）转发给子进程，支持优雅终止。

- **操作步骤**

  针对 k8s 和 docker 低版本：

  1. **安装 `tini`**：
     ```dockerfile
     # 使用官方镜像（已集成 tini）
     ENTRYPOINT ["tini", "--"]

     # 或手动安装（适用于自定义镜像）
     RUN apt-get update && apt-get install -y tini
     ENTRYPOINT ["tini", "--"]
     ```
  2. **启动业务进程**：

     ```dockerfile
     CMD ["python", "app.py"]
     ```
  3. **完整Dockerfile示例**：
     ```dockerfile
     FROM python:3.9
     RUN pip install tini
     COPY app.py /app.py
     ENTRYPOINT ["tini", "--"]
     CMD ["python", "/app.py"]

     # 之后启动容器无需指定--init参数
     ```

  针对 docker1.13+：

  - ```sh
    docker run -d --init --name test1 t1:v1 # 不需要做任何准备，直接指定--init即可，因为docker本身默认集成了tini
    ```

- **优点**：

  - 支持回收所有层级的僵尸进程。

  - 转发信号至子进程，确保优雅终止（如数据库连接释放）。

    - 问题：默认只转发给儿子，其他后代无法接收

    - 解决方法：

      ```dockerfile
      ENV TINI_KILL_PROCESS_GROUP 1
      # 启用进程组杀除，kill(-PGID, SIGTERM) 向整个进程组发送信号，确保所有后代进程都能收到终止信号
      ```

  - 轻量级（仅增加约20KB镜像体积）。

#### **3.3 对比与总结**

| **方案**       | **回收能力** | **信号转发** | **适用场景**       | **缺点**                     |
| -------------- | ------------ | ------------ | ------------------ | ---------------------------- |
| **开发者修正** | 依赖代码实现 | 依赖代码实现 | 可控的自主开发应用 | 对遗留代码或第三方软件无效   |
| **bash脚本**   | 仅直接子进程 | 无           | 快速修复简单场景   | 无法处理孙子进程，无优雅终止 |
| **tini**       | 所有子进程   | 有           | 通用场景（推荐）   | 需额外安装或使用集成镜像     |

### 4. 进程信号

**信号（Signal）** 是Linux系统中进程间通信的一种机制，本质是向目标进程发送通知。当进程收到信号后，会根据信号类型执行预设操作。
- **示例**：使用 `kill` 命令发送信号，如 `kill -9 PID` 强制终止进程。

> **查看信号详细信息**：`man 7 signal` 可获取各信号的默认行为和用途。

#### 4.1 进程收到信号后的三种反应
##### （1） 忽略（Ignore）
- 进程完全无视该信号，不采取任何动作。
- **例外**：特权信号（如 `SIGKILL` 和 `SIGSTOP`）无法被忽略。

##### （2） 捕获（Catch）
- 进程可注册自定义处理函数（Handler），信号到达时触发该函数。
- **示例**：通过Shell的 `trap` 命令捕获信号并执行自定义逻辑：

  ```bash
  trap 'echo "SIGINT received!"' SIGINT
  ```

##### （3） 默认/缺省行为（Default）
- Linux为每个信号定义了默认行为，可通过 `man 7 signal` 查看。
- **常见默认行为**：
  - `SIGTERM`（15）：终止进程。
  - `SIGKILL`（9）：强制终止。
  - `SIGSTOP`（19）：暂停进程运行。

#### 4.2 特权信号

**`SIGKILL` (-9) **：强制杀死进程

**`SIGSTOP` (-19)**：暂停进程的运行

- `SIGCONT`（18）：恢复被 `SIGSTOP` 暂停的进程。

**特权性表现**

- **无法被忽略**：进程不能屏蔽这两个信号。
- **无法被捕获**：无法注册自定义Handler覆盖默认行为。
- **只能执行默认行为**

#### 4.3 “容器内”进程的信号处理特殊性

内核为保护 1 号进程，使其**不受未捕获的、可能导致终止的特权信号的伤害**。规定：

如果 1 号进程没有为某个信号显式注册一个处理函数（handler），那么即使该信号是特权信号（当然，特权信号无法被自定义，永远不会被执行），内核也会忽略该信号，不会将它传递给 1 号进程，也不会执行默认行为。

- **特权信号失效**：

   - `kill -9 1` 无法杀死容器内的1号进程。
   - `kill -19 1` 无法暂停容器内的1号进程。

- **`SIGTERM`（15）可能杀死容器内的1号进程**：

   - 若信号未注册Handler，默认行为不生效（如：没有为 `SIGTERM` 注册任何自定义handler，这个信号实际上会被内核忽略掉，1号进程收不到信号）。

   - 若注册了Handler，进程会执行自定义逻辑。

     > 拓展：
     >
     > - go程序
     >   - 默认注册 `SIGINT`（通常在用户按下Ctrl+C时发送，用于中断程序）、`SIGTERM` （通常用于请求程序终止）信号
     > - bash程序
     >   - 默认注册 `SIGINT`、`SIGTERM`、`SIGHUP`（当终端关闭时发送，通常会导致shell退出）、`SIGQUIT`（通常在用户按下Ctrl+\时发送，会导致前台进程退出并生成核心转储）、`SIGTSTP`（通常在用户按下Ctrl+Z时发送，将前台进程挂起） 信号
     > - Python程序不会自动为信号注册处理函数

- **原因**（了解即可）：

  - **`kill` 命令的执行路径**

    <img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/image-20250328185641781.png" alt="image-20250328185641781" style="zoom: 40%;" />

  - **关键条件判断（`sig_task_ignored()`）**

    <img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/image-20250328191310977.png" alt="image-20250328191310977" style="zoom: 33%;" align='left'/>

    `return true`：忽略该信号，目标进程不会接收到该信号。

    - **`t->signal->flags & SIGNAL_UNKILLABLE`**
      - 容器内1号进程在创建时会被标记为 `SIGNAL_UNKILLABLE`，该条件一定会成立。

    - **`handler == SIG_DFL`**
       - 若进程未注册自定义Handler（使用默认行为 `SIG_DFL`），条件成立。
       - 若注册了Handler，条件不成立。
    - **`!(force && sig_kernel_only(sig))`**
       - `force` 的值：
         - `0`：发送信号的进程与目标进程在同一Namespace。
         - `1`：跨Namespace（如宿主机向容器内进程发信号）。
       - `sig_kernel_only(sig)`：仅对 `SIGKILL`（9）和 `SIGSTOP`（19）返回 `true`。
    - **总结：信号被忽略的条件（同时满足）**
      - 目标进程是容器内1号进程（`SIGNAL_UNKILLABLE` 标志存在）。
      - 信号使用默认处理（未注册自定义Handler）。
        - 而特殊信号无法被注册
      - `!(force && sig_kernel_only(sig))` 为 `true`。

## 二、CPU cgroup

### 1. cgroup 介绍
#### **1.1 什么是 cgroup？**
- **cgroup**（Control Group）是 Linux 内核提供的一种机制，用于**限制、记录和隔离进程组**（如容器）**对宿主机资源**（CPU、内存、磁盘 I/O 等）**的使用**。
  - **CPU cgroup**：cgroup 的一个子系统，用来限制进程对 cpu 资源的使用。它一般是挂载到 `/sys/fs/cgroup/cpu` 目录下。在该目录下，每个控制组都是一个子目录，各控制组之间呈树状的层级关系。（具体使用可看博客）

#### **1.2 为何使用 cgroup？**
- **资源隔离**：防止容器无限制消耗资源（如 CPU、内存），确保宿主机上多个容器共存时的稳定性。
- **优先级控制**：为不同容器分配资源使用权重，实现资源公平调度。

### 2. CPU 状态信息详解

<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/image-20250328201448680.png" alt="image-20250328201448680" style="zoom: 25%;"/>

#### **2.1 CPU 时间分配统计**
- **us（User）**：用户态进程占用 CPU 时间的百分比（不包括低优先级进程）。
- **sys（System）**：内核态进程占用 CPU 时间的百分比。
- **ni（Nice）**：`nice` 值为 1-19 （数值越小优先级越高）的低优先级进程在用户态占用的 CPU 时间百分比。
- **id（Idle）**：系统空闲（S） CPU 时间的百分比。
- **wa（Iowait）**：系统**等待** I/O 完成（不可中断睡眠 D）的 CPU 时间占比（不计入进程 CPU 时间，并没有在使用 CPU）。
- **hi（Hardware IRQ）**：处理硬件中断的时间占比（不计入进程 CPU 时间）。
- **si（SoftIRQ）**：处理软件中断的时间占比（不计入进程 CPU 时间）。
- **st（Steal）**：宿主机上其他虚拟机抢走的 CPU 时间（如云环境中的虚拟机竞争）。

与 CPU 的使用有关的只有 **us、ni**（前两者是用户态进程）和 **sy**（内核态进程）

#### **2.2 关键强调**
1. **中断处理与进程无关**：

   - `hi`（硬件中断）和 `si`（软件中断）的 CPU 时间不计入任何进程，因为中断处理独立于用户态和内核态进程。

     - > **中断的协作流程**：
       >
       > 假设你（CPU）正在专心写作业（运行一个进程），这时有人给你送快递（硬件设备有数据要处理，比如网卡收到了网络数据包）。整个流程分两步：
       >
       > - **1. 硬中断：快递员打电话催你开门（紧急但快速）**
       >   - **发生了什么**：
       >     快递员（硬件设备，比如网卡）突然打电话给你（触发硬中断），大喊：“快递到了！快开门拿！不然我走了！”
       >     （硬件需要CPU立即处理数据，否则可能丢失）
       >   - **你的反应**：
       >     你立刻暂停写作业（中断当前进程），跑过去开门，把快递包裹快速接进来（把数据从硬件设备复制到内存缓冲区）。
       >     这个过程必须**极快完成**（硬中断处理时间极短），否则快递员会不耐烦（数据丢失）。
       >   - **结果**：
       >     你拿到了快递，但作业只写了一半（进程被短暂打断）。
       > - **2. 软中断：拆包裹、整理快递（耗时但可以缓缓做）**
       >   - **发生了什么**：
       >     快递员走后，你看着手里的包裹（数据包），心想：“现在拆包裹太费时间了，作业还没写完，晚点再弄吧！”
       >     （内核决定把耗时的操作交给软中断延迟处理）
       >   - **你的操作**：
       >     1️⃣ 你把包裹先放在桌上（标记为待处理的软中断任务）。
       >     2️⃣ 回去继续写作业（恢复被中断的进程）。
       >     3️⃣ 等作业写到一个自然段落（CPU空闲时），你再开始拆包裹、检查物品、分类放好（软中断处理数据：解析数据包，传递给应用程序）。
       >   - **结果**：
       >     快递处理完了，但几乎没有影响你写作业的进度（进程不会长时间被阻塞）。
       > - 两者配合，既保证了紧急任务不丢失，又避免了长时间打断手头的工作。

2. **不可中断睡眠（D 状态）**：

   - 进程因等待 I/O（如磁盘操作）进入不可中断状态，**不可被终止**。

3. **可中断睡眠（S 状态）**：
   - 进程等待外部事件（如用户输入、网络响应）进入可中断状态，**可被终止**。

### 3. CPU cgroup 中和 CFS 相关参数

具体配置参照博客

CFS（完全公平调度器）：一种算法

- **`cpu.cfs_period_us`**：
  - 定义 CPU 资源分配的周期长度（单位：微秒），默认 `100ms`（100,000μs）。
- **`cpu.cfs_quota_us`**：
  - 指定控制组内进程在一个周期内最多可使用的 CPU 时间（单位：微秒）。
  - **计算示例**：
    - 若 `period=100ms`，`quota=50ms`，则进程最多使用 `50/100=0.5` 个 CPU 核心。

- **`cpu.shares` **：
  - 在**同一层级**的控制组中，**当宿主机 CPU 资源不足时**（资源充足时，该参数不起作用），按 `shares` 值的比例分配 CPU 时间。

  - **示例**：
    - 控制组 A 的 `shares=3072`，B 的 `shares=1024，则 A 和 B 的 CPU 时间占比为 `3：1。
    - 若操作系统共有4颗CPU，而A仅需两颗，B需4颗，按照上例分配，最终会是A两颗，B也两颗（1：1）。

### 4. Kubernetes 中的资源管理
#### **资源申请参数**
- **`request`**（下限）：
  - 对应 cgroup 的 `cpu.shares`，表示容器初始申请的 CPU 资源量。
  - 实际使用可超过 `request`。
- **`limit`**（上限）：
  - 对应 cgroup 的 `cpu.cfs_quota_us` 和 `cpu.cfs_period_us`，限制容器 CPU 使用的上限。
  - **注意**：能否达到 `limit` 取决于宿主机是否有足够资源。

## 三、监控容器 CPU 真实使用率

当容器云平台中一些节点出现负载过高、应用程序异常时，一个主要的解决方案就是找出对 CPU 占用过高的进程。

使用 `top` 命令无法获取容器内准确的 CPU 使用率，因为它还会包含容器外宿主机内进程对 CPU 的使用。

### **1. CPU 使用率**
#### **（1）定义**

- 进程对 CPU 的使用率 =（用户态时间 `us` + 内核态时间 `sy`） / CPU 总时间片。
- **多核场景**：
  - 100% 表示占用 1 个 CPU 核心，200% 表示占用 2 个核心。
  - 若宿主机有 4 核，最大利用率可达 400%。

#### **（2）VS 平均负载（Load Average）**

- **平均负载**：某段时间内平均活跃进程数（包括可运行和不可中断状态）。
- **与 CPU 核心数的关系**：
  - 若宿主机有 4 核，平均负载可能超过 4，但 CPU 使用率上限仍为 400%。

### **2. 获取进程对 CPU 真实使用率**
```plaintext
CPU 使用率 = [(utime₂ - utime₁) + (stime₂ - stime₁)] / (总时间差 * HZ) * 100%
# 总时间差 (Δt)：两次采样的实际时间间隔（单位：秒）
# 分子总和 = 进程在 Δt 时间内消耗的总 CPU ticks
# 分母：在 Δt 秒内，一个 CPU 核心最多能处理 Δt × HZ 个 ticks（相当于单核 CPU 的理论最大工作时间）
```
- `utime`：用户态时间；`stime`：内核态时间（单位：CPU 时钟 **ticks**）。

  ```sh
  # 查看单个进程的utime($14)、stime($15)
  cat /proc/<pid>/stat | awk '{print $14 $15}'

  # /proc/stat 里存放的是整个系统所有进程的累积时间
  ```

- `HZ`：系统时钟频率（单位：ticks/秒），通常为 100，1 tick = 10ms。  （1个tick代表一次中断的周期，这个周期需要耗费的时间由中断频率HZ决定，总时间差*HZ => 总时间内有多少 tick）

  ```sh
  # 查看HZ
  getconf CLK_TCK
  ```

- **示例**：

  在 10 秒内，进程占用 `us + sy` 的总 ticks 为 5000，则利用率 = `5000 / (10*100) * 100% = 500%`（需结合 CPU 核心数判断合理性）。

### 3. 获取容器对 CPU 的真实占用率

容器本质是进程，统计方式与普通进程相同

**操作步骤**

- **进入容器的 cpu CGroup 目录**：

  ```sh
  cd /sys/fs/cgroup/cpu/system.slice/docker-daa5429122d89deb208f802f6e9e4ed690efa6f5d42067cce927e6fbbd95.scope/
  ```

- **读取 CPU ticks**：

  ```sh
  cat cpuacct.stat  # 输出示例：user 66842（用户态 ticks）、system 255（内核态 ticks）
  ```

- **动态计算**：
  间隔 3 秒再次读取 `cpuacct.stat`，根据公式计算实时 CPU 使用率。

**补充说明**

- **CGroup 路径链接**：
  `/sys/fs/cgroup/cpu` 是 `/sys/fs/cgroup/cpuacct/` 的符号链接，两者数据一致。
- **替代方案**：
  使用 LXC 提供的 `lxcfs` 虚拟化 `/proc` 文件系统，使容器内 `top` 等命令显示真实的 CPU 使用率（参考 [lxcfs GitHub](https://github.com/lxc/lxcfs)）。

### 4. **Docker 容器监控工具**

#### 4.1 **原生命令 `docker stats`**

- **功能**：实时查看容器 CPU、内存、网络等资源使用情况。

  ```sh
  docker stats <容器名或ID>
  # 输出示例：
  # CONTAINER ID  NAME       CPU%   MEM USAGE/LIMIT  MEM%  NET I/O    BLOCK I/O  PIDS
  # 487ecf4f      C3         398.82% 199MiB/3.84GiB  .08%  586B/B     eB/eB      5
  ```

- **优点**：简单快捷，无需额外部署。

- **缺点**：

  - 仅支持单宿主机，无法汇总多节点数据。
  - 数据实时但无存储，缺乏历史记录和预警功能。
  - **指标不准确**：
    `PIDS` 统计会计入线程数（例如：单进程含 4 线程时显示 `PIDS=5`，1主进程+4线程）。

#### **4.2 使用 cAdvisor**

##### 4.2.1 **cAdvisor 简介**

Google 开源工具，用于实时监控容器资源使用情况（CPU、内存、网络等），提供 Web 界面和 API。

使用 Go 语言开发，利用 Linux 的 cgroups 获取容器的资源使用信息，在 k8s 中集成在 Kubelet 里作为默认启动项。

**优势**：

- 采集**宿主机所有容器**资源数据，提供 Web 界面和 HTTP 接口。

- 支持与 Prometheus、InfluxDB、Grafana 集成，实现数据持久化（cAdvisor 默认不存储历史数据）与可视化。典型组合：

  - **cAdvisor + Prometheus + Grafana**：长期存储、灵活查询、可视化仪表盘。

  - **cAdvisor + InfluxDB + Grafana**：时间序列数据库存储，适合实时监控。

##### 4.2.2 **部署 cAdvisor 容器**

```bash
docker pull google/cadvisor:latest

docker run \
  --volume=/:/rootfs:ro \          # 挂载宿主机根目录（只读，读取全局资源数据）
  --volume=/var/run:/var/run:rw \  # 挂载进程运行时数据（可写，如容器进程 PID）
  --volume=/sys:/sys:ro \         # 挂载系统信息（只读，系统硬件和内核指标，如 CPU 核心数）
  --volume=/data/docker/:/var/lib/docker:ro \  # 挂载 Docker 数据目录（只读）
  --volume=/dev/disk/:/dev/disk:ro \           # 挂载磁盘信息（只读）
  --publish=8080:8080 \           # 映射 Web 端口
  --detach=true \                 # 后台运行
  --name=cadvisor \               # 容器名称
  google/cadvisor:latest          # 使用官方镜像
```

**访问监控界面**

- 浏览器访问 `http://宿主机IP:8080`，查看容器列表及实时 CPU 使用率。
- cAdvisor 自动计算 CPU 利用率，无需手动处理 HZ 和 ticks。

### 5. 补充

**/sys/fs/cgroup/cpu/group1 VS /sys/fs/cgroup/cpu/system.slice/docker-<容器id>scope/**

| 特性   | /sys/fs/cgroup/cpu/group1                             | /sys/fs/cgroup/cpu/system.slice/docker-...scope/             |
| ------ | ----------------------------------------------------- | ------------------------------------------------------------ |
| 类型   | 自定义cgroup                                          | 系统生成的cgroup                                             |
| 创建者 | 用户或系统管理员                                      | systemd和Docker自动生成                                      |

## 四、D 状态对容器的影响

### 1. 问题现象

- 容器内应用运行缓慢，但资源限制（cgroup）未启用。
- 宿主机资源充足：
  - **CPU使用率低**：未达到瓶颈。
  - **内存充足**：无交换（swap）压力。
  - **平均负载高**：显著高于CPU核心数。

### 2. 问题根源

- **平均负载的计算机制**：
  高负载可能由大量**不可中断睡眠进程（D状态）** 引起，即使CPU和内存空闲。
- **D状态进程的影响**：
  - 进程因等待I/O（如磁盘、网络）进入D状态，**持续占用平均负载指标**。
  - 此类进程不消耗CPU，但导致负载升高，表现为系统“假繁忙”。

### 3. 解决方案

1. **排查I/O瓶颈**：
   - 使用`iostat`检查磁盘I/O利用率、响应时间。
   - 分析`dmesg`日志，确认是否存在硬件或驱动故障。
2. **优化I/O路径**：
   - 增加CPU/内存配额无法缓解I/O等待问题，需优化I/O路径（如更换高速磁盘、优化读写逻辑）。
3. **监控工具建议**：
   - `pidstat -d`：定位高I/O进程。
   - `iotop`：实时查看进程磁盘I/O使用情况。

> **总结**：
>
> - **平均负载高 ≠ CPU繁忙**：可能是I/O密集型任务导致D状态进程堆积。
> - **容器性能问题需多维度分析**：结合CPU使用率、负载、I/O状态综合判断。
