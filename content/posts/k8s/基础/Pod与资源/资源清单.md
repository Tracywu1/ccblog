### 1. **核心概念关系**

- **资源类型**：
  类似于数据库中的表结构，定义了资源的抽象模型（如 `Pod`、`Deployment` 等）。
  
- **资源清单**：
  相当于表中的一条记录，通常以 `YAML` 或 `JSON` 格式描述具体的资源实例。
  仅创建资源清单文件并不等同于创建资源，需通过控制器背后的具体代码实现资源实例化。
  
- **控制器（Controller Manager）**：
  负责监控集群状态，根据资源清单中定义的期望状态（`spec`）创建或调整实际资源。
  
  > 用资源清单创建出来的控制器资源本质是”资源“，其背后有相应的代码逻辑来负责实现控制器的功能
### 2. **资源清单的使用**

#### 2.1 生成与查看清单模板

- **快速生成模板**：
  通过 `kubectl create` 命令模拟创建过程，生成 YAML 模板：

  ```sh
  kubectl create deployment web --image=nginx:1.20 --dry-run=client -o yaml > web.yaml
  ```

  - `--dry-run=client`：仅输出资源定义，不实际创建资源。
  - `-o yaml`：指定输出格式为 YAML。

- **查看资源定义说明/查看不同层级下有什么字段**：
  使用 `kubectl explain` 查看资源字段的文档：

  ```sh
  kubectl explain pods           # 查看 Pod 资源的一级字段
  kubectl explain pods.spec      # 查看 Pod 的 spec 字段定义
  kubectl explain pods.metadata  # 查看 Pod 的元数据字段
  ```

  - **层级查询**：可通过 `.` 逐级深入，如 `pods.spec.containers`。
  - **必填字段**：标记为 `required` 的字段必须显式定义。
#### 2.2 资源清单格式详解

资源清单由 **5 个一级字段** 构成：

- **`apiVersion`**

  - 定义资源所属的 API 组及版本，格式：`<group>/<version>`（核心组可省略组名，如 `v1`）。

  - 查看支持的 API 版本：

    ```sh
    kubectl api-versions
    ```

- **`kind`**

  - 资源类型，如 `Pod`、`Deployment`、`StatefulSet`、`DaemonSet` 等。

- **`metadata`**

  - 资源元数据，包含以下关键字段：
    - **`name`**：资源名称（同一类别下唯一）。
    - **`namespace`**：所属命名空间（默认 `default`）。
    - **`labels`**：键值对标签，用于资源筛选（如 `kubectl get pods -l app=web`）。
    - **`annotations`**：资源注解，非标识性元数据，用于扩展信息（如版本说明、配置参数）。

- **`spec`**

  - **最核心字段**：定义资源的期望状态（Desired State），控制器据此创建或调整资源。
  - 字段结构因资源类型而异，需结合 `kubectl explain` 查询具体定义。

- **`status`**

  - 记录资源的当前状态（Current State），由 k8s 自动维护，用户无需配置。
#### 2.3 yaml文件格式说明

##### 2.3.1 YAML 基础语法

**基本规则**

- 大小写敏感
- 使用空格缩进表示层级关系（**禁止使用 Tab**）
- 缩进空格数不重要，但同级元素必须左对齐（建议缩进 2 个空格）
- `#` 表示注释，从该字符到行尾的内容被忽略
- `---` 用于分隔同一文件中的多个资源定义（可选）
- `-` 表示列表项的开始，用于定义数组（列表）中的元素。

##### 2.3.2 Pod 资源配置清单示例

以下是一个裸 Pod 的完整配置示例（通常更推荐通过 Deployment 管理 Pod）：

```yaml
apiVersion: v1              # 必选，API 版本（核心组可省略组名）
kind: Pod                   # 必选，资源类型为 Pod
metadata:                   # 必选，元数据定义
  name: nginx1              # 必选，Pod 名称（同命名空间内唯一）
  namespace: web-testing    # 可选，默认 default
  labels:                   # 可选，标签列表（键值对）
    app: nginx
  annotations:              # 可选，注解列表（扩展元数据）
    app: nginx
spec:                       # 必选，定义 Pod 的期望状态
  containers:               # 必选，容器列表
  - name: nginx             # 必选，容器名称（RFC1035 规范）
    image: nginx:v1         # 必选，容器镜像地址
    imagePullPolicy: Always # 可选，镜像拉取策略（Always[总是下载镜像]/IfNotPresent[默认，先看本地是否有该镜像，如果没有就下载]/Never[只使用本地镜像]）
    workingDir: /usr/share/nginx/html  # 可选，容器工作目录
    volumeMounts:          	# 可选，存储卷挂载配置
    - name: webroot        	# 存储卷名称（需与 volumes 中定义的名称匹配）
      mountPath: /usr/share/nginx/html # 挂载到容器内的目录
      readOnly: true	   	# 只读
    ports:                  # 可选，容器暴露端口列表（不指出，容器也不会阻止端口暴露）
    - name: http            # 端口名称
      containerPort: 80     # 端口号
      protocol: TCP         # 协议类型（默认 TCP）
    env:                    # 可选，环境变量配置
    - name: TZ				# 时区
      value: Asia/Shanghai
    - name: LANG			# 语言
      value: en_US.utf8
    resources:              # 可选，资源限制与请求
      limits:               # 最大资源限制
        cpu: 1000m          # 1 核（1000m = 1 CPU，m代表千分之一核）
        memory: 1024Mi     # Mi:二进制，M:十进制
      requests:             # 资源请求（调度依据）
        cpu: 100m           # 0.1 核
        memory: 512MiB
    readinessProbe:         # 可选，就绪探针
      httpGet:              # 检测方式（HTTP GET）
        path: /             # 检测路径
        port: 80            # 检测端口
      timeoutSeconds: 2     # 超时时间
      initialDelaySeconds: 60  # 首次检测延迟时间
    livenessProbe:          # 可选，存活探针，下述罗列了2种检测方式（还有HTTP GET方式），实际应用其中一种即可
      exec:                 # 检测方式（执行命令）
        command:
        - cat
        - /health
      tcpSocket:            # 检测方式（TCP 端口检查）
        port: 80
      initialDelaySeconds: 30 # 初始化时间
      periodSeconds: 5      # 检测间隔
      successThreshold: 2   # 成功阈值（连续成功次数）
      failureThreshold: 1   # 失败阈值（连续失败次数）
    securityContext: 		# 可选，限制容器不可信行为
      privileged: false     # 表示容器不会以特权模式运行
  restartPolicy: Always     # 可选，重启策略（Always/OnFailure/Never）
  nodeSelector:             # 可选，节点选择器（标签筛选）
    region: subnet7
  imagePullSecrets:         # 可选，拉取镜像的 Secret（通常用于拉取私有仓库镜像）
  - name: default-dockercfg-86258
  hostNetwork: false        # 可选，是否使用主机网络（默认 false）
  							# 若为 true, Pod 中的容器将使用主机的网络命名空间（IP、端口的部分），不适用k8s网络模式（极个别情况设为 true）
  volumes:                  # 可选，共享存储卷列表
  - name: webroot           # 存储卷名称
    emptyDir: {}            # 临时存储卷（Pod 生命周期内有效）
  - name: hosts
    hostPath:               # 主机路径存储卷
      path: /etc/hosts
      type: File            # 文件类型（File/Directory）
  - name: secret-volume
    secret:                	# Secret 存储卷
      secretName: default-token-tf2jp
      defaultMode: 420     	# 文件权限（八进制）
  - name: configmap-volume
    configMap:             	# ConfigMap 存储卷
      name: nginx-conf
      defaultMode: 420
```

### 3. 补充（了解）

#### **3.1 拉取私有镜像的步骤**

##### **（1）创建 Docker 配置文件**
- **配置文件路径**：`~/.docker/config.json`  
  
  - 若已通过 `docker login` 登录私有仓库，该文件会自动生成。
  - **手动创建示例**（以阿里云镜像仓库为例）：
    ```json
    {
      "auths": {
        "registry.cn-shanghai.aliyuncs.com": {
          "auth": "ZwdvbmxpbmhhawZ1bmc6bGhmQDA5NDU3Mw=="
        }
      }
    }
    ```
  - **说明**：`auth` 字段为 `username:password` 的 Base64 编码值。
##### **（2）创建 Kubernetes Secret**
- **命令**：
  ```bash
  kubectl create secret generic my-registry-secret \
    --from-file=.dockerconfigjson=<配置文件路径> \
    --type=kubernetes.io/dockerconfigjson
  ```
- **参数解析**：
  - `--from-file`: 指定 Docker 配置文件路径。
  - `--type`: 必须为 `kubernetes.io/dockerconfigjson`。
- **验证 Secret**：
  
  ```bash
  kubectl get secret my-registry-secret -o yaml
  ```
##### **（3）在 Pod 中引用 Secret**
- **Pod 配置示例**：
  ```yaml
  apiVersion: v1
  kind: Pod
  metadata:
    name: my-pod
  spec:
    containers:
    - name: my-container
      image: registry.cn-shanghai.aliyuncs.com/your-namespace/your-image:tag
    imagePullSecrets:  # 引用 Secret
    - name: my-registry-secret
  ```
#### **3.2 资源单位详解**

> ##### **（1） CPU 单位**
> - **单位**：`m`（千分之一核）  
>   - `1000m = 1 核`  
>   - 示例：`500m` 表示 0.5 核。
> - **物理 CPU 计算**：
>   - **总物理核数** = 物理 CPU 数量 × 每 CPU 核数  
>   - **总逻辑核数** = 物理 CPU 数量 × 每 CPU 核数 × 超线程数  
>
> ---
>
> ##### **（2）内存单位**
> - **单位**：  
>   - `Mi`：二进制单位，1 MiB = 1024 × 1024 Bytes  
>   - `MB`：十进制单位，1 MB = 1000 × 1000 Bytes  
> - **示例**：  
>   - `512Mi` = 512 MiB  
>   - `1Gi` = 1024 MiB  
>
> ---
>
> ##### **（3）节点可分配内存**
> - **查看节点资源**：
>   
>   ```bash
>   kubectl describe node <节点名>
>   ```
>   
> - **计算公式**：  
>
>   **Allocatable Memory = 物理内存 - kube-reserved - system-reserved - eviction-threshold**  
>
>   - `kube-reserved`：Kubernetes 组件预留内存（如 kubelet、容器运行时）  
>   - `system-reserved`：系统守护进程预留内存（如 sshd、journald）  
>   - `eviction-threshold`：节点驱逐 Pod 的内存阈值  
>
#### **3.3 常见问题与注意事项**

##### **1. Secret 配置问题**
- **错误提示**：`ErrImagePull` 或 `ImagePullBackOff`  
  - **排查步骤**：
    1. 检查 Secret 名称与 Pod 中引用的名称是否一致。
    2. 确保 `imagePullSecrets` 字段缩进正确（位于 `spec` 下）。
    3. 验证私有镜像地址是否正确（含命名空间和标签）。

##### **2. 私有仓库证书问题**
- **自签名证书**：需在节点 Docker 配置中信任仓库证书。
  
  ```bash
  # 将证书复制到 Docker 证书目录
  sudo cp ca.crt /etc/docker/certs.d/registry.example.com/
  ```

##### **3. 更新 Secret**
- **更新配置文件后**：需重新创建 Secret 并重启 Pod。
  ```bash
  kubectl delete secret my-registry-secret
  kubectl create secret ...  # 重新创建
  kubectl delete pod my-pod  # 触发 Pod 重建
  ```
