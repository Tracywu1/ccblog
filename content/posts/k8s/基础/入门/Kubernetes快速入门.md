## 一、k8s是什么？
- **全称**：Kubernetes（k8s是缩写，K与s之间有8个字母）。  
- **定义**：一个跨宿主机管理容器的系统，用于自动化容器化应用的部署、扩展和管理。  
## 二、为何使用k8s？
### 1. 解决裸跑容器的问题
- **Docker本质**：容器引擎，基于命名空间、cgroup、联合文件系统**创建**容器。（docker造容器，k8s管理容器）
- **管理缺陷**：  
  1. 缺乏容器存活监控与自动重启机制。  
  2. 集群化模式下容器调度（如何分配容器到不同的物理主机）能力不足。  
  3. 跨宿主机网络通信支持有限。  

- **k8s的解决方案**

  - **自动化运维能力**：
    - **故障监控与自愈**：实时监控集群状态，一旦检测到故障，自动进行恢复，确保服务的高可用性。
    - **服务编排**：通过声明式配置，自动部署、管理和维护容器化应用，简化复杂的应用部署流程。
    - **服务滚动升级和在线扩容**：支持无缝的滚动升级和在线扩容，不影响现有服务的运行，保证业务的连续性。
    - **配置管理**：集中管理应用的配置信息，实现配置的版本控制、动态更新和回滚。

  - **强大的调度功能**：
    - **智能调度算法**：根据资源使用情况、应用需求等因素，智能分配容器到最合适的主机，实现负载均衡和资源优化。
    - **全局视图**：提供集群级别的资源和使用情况的全局视图（Prometheus、Grafana），便于管理和决策。

### 2. 为云原生而生
#### 2.1 **云原生 = Cloud + Native**
- **Cloud（云计算）**  
  - **本质**：通过虚拟化技术将资源统一管理，形成**资源池**，实现资源的弹性分配和循环利用。  
  - **典型平台**：  
    - **虚拟机平台**（如阿里云）。  
    - **容器云平台**（如Kubernetes）。  

- **Native（原生）**  
  - **内涵**：应用在设计之初即适配云环境，或通过改造充分利用云平台特性，以提升软件质量。  
  - **关键特性**：  
    - **故障自愈**：自动检测并恢复异常。  
    - **弹性伸缩**：动态调整资源以应对负载变化。  
    - **高效资源利用**：按需分配，减少浪费。  
- **终极目标**  
  - 软件从设计、开发到部署，完全围绕云平台特性构建，最大化利用云的优势。
#### 2.2 云原生的核心技术体系
- **微服务架构**

  - **作用**：将单体应用拆分为独立服务，提升灵活性和可维护性。  

  - **与k8s的关系**：  
    
    k8s通过**Pod**和**Service**管理微服务，实现服务发现、负载均衡和跨节点通信。


- **DevOps**

  - **矛盾背景**：  
    - **开发（Dev）**：追求快速迭代和功能更新。  
    - **运维（Ops）**：追求系统稳定性和减少变更风险。  

  - **解决方案**：  
    - **自动化工具链**：通过CI/CD流水线弥合开发与运维的鸿沟。  
    - **k8s的作用**：提供统一的部署和管理平台，支持灰度发布、滚动更新等运维操作。


- **CI/CD（持续集成/持续交付）**

  - **典型流程**：  
    
    ```mermaid
    graph LR
      A[GitLab代码仓库] --> B[Jenkins流水线]
      B --> C[构建镜像]
      C --> D[Harbor镜像仓库]
      D --> E[k8s生产环境]
    ```

  - **关键环节**：  
    1. **代码提交**：触发Jenkins流水线（拉代码、测试、构建、推送镜像到 Harbor、发布[ master 分支需要人工确认]）。  
    2. **镜像构建**：通过Dockerfile生成标准化容器镜像。  
    3. **镜像存储**：Harbor提供安全的镜像仓库管理。  
    4. **部署到k8s**：k8s根据镜像快速拉起容器，完成服务发布。  


- **容器化**

  - **核心价值**：  
    - **环境一致性**：开发、测试、生产环境完全一致。  
    - **资源隔离**：通过cgroups和命名空间限制资源使用。  

  - **k8s的容器管理能力**：  
    - 自动化调度容器到最优节点。  
    - 监控容器健康状态并自动重启。  
    - 提供跨主机的容器网络（CNI插件）和存储（PV/PVC）。  
#### 2.3 k8s如何支撑云原生？
- **解决容器管理痛点**

  | 问题           | k8s能力                                                    |
  | -------------- | ---------------------------------------------------------- |
  | 容器跨主机通信 | CNI网络插件（如Calico、Flannel）                           |
  | 容器调度       | Scheduler预选+优选策略，支持节点亲和性、污点容忍等         |
  | 故障自愈       | 控制器（如Deployment）自动重启Pod或迁移到健康节点          |
  | 弹性伸缩       | HPA（Horizontal Pod Autoscaler）根据CPU/内存指标自动扩缩容 |

- **赋能云原生核心特性**

  - **自动化运维**：  
    - **声明式API**：用户定义期望状态，k8s自动调谐（如`kubectl apply -f`）。  
    - **Operator模式**：扩展k8s API，管理有状态应用（如数据库）。  

  - **资源高效利用**：  
    - **资源配额（Resource Quota）**：限制Namespace的资源总量。  
    - **优先级抢占（PriorityClass）**：确保关键服务优先获得资源。  

  - **服务治理**：  
    - **Service Mesh集成**（如Istio）：实现细粒度流量控制、熔断和监控。  
#### 2.4 云原生与k8s的共生关系
- **k8s是云原生的基石**

  - **标准化运行时**：容器化应用在k8s上无需关心底层基础设施差异。  

  - **生态中心**：CNCF（云原生计算基金会）以k8s为核心，推动Prometheus、Envoy等工具集成。


- **云原生驱动k8s演进**
  - **新需求推动功能迭代**：  
    - **Serverless**：k8s的Knative项目支持无服务器架构。  
    - **边缘计算**：kube-edge扩展k8s到边缘设备。  
#### 2.5 总结
**Kubernetes是云原生落地的核心引擎**，其通过容器编排、自动化运维和弹性架构，完美支撑了云原生的四大支柱（微服务、DevOps、CI/CD、容器化）。从资源池化到应用自愈，从开发到交付，k8s实现了云原生理念的全生命周期闭环，成为现代云计算基础设施的事实标准。
## 三、何时使用k8s？
### 1. **适用场景**
- 微服务架构或应用数量多、运维成本高。  
- 需快速部署测试环境验证新功能。  
- 需降低硬件成本、提高资源利用率。  
- 需自动化管理（监控、自愈、滚动更新）。  

### 2. **不适用场景**
- 轻量级单体应用，无高并发需求。  
- 团队技术栈不适应容器化变革。

### 1. 架构：主从模式

#### 1.1 **Master 节点（控制平面）**

- **核心职责**：
  负责集群的全局决策（如调度）、资源管理和状态维护，是集群的“大脑”。

- **高可用建议**：
  生产环境建议部署 **至少3个Master节点**，通过冗余保证集群可靠性。

- **关键组件**：

  | 组件                   | 功能描述                                                     |
  | :--------------------- | :----------------------------------------------------------- |
  | **API Server**         | 集群唯一入口，处理所有REST请求，读写etcd数据库。             |
  | **Scheduler**          | 决策Pod调度到哪个节点（分预选+优选两阶段）。<br />**预选**：过滤不满足条件的节点（如资源不足）。 <br />**优选**：打分选择最优节点（如资源利用率最低）。 |
  | **Controller Manager** | 运行控制器（如Deployment、ReplicaSet），确保资源状态与声明一致。 |
  | **etcd**               | 分布式键值数据库，存储集群所有配置和状态数据（使用RAFT协议保证一致性）。 |

- **etcd关键特性**：

  - 超过半数节点故障时集群不可用（如3节点集群允许1节点故障）。
  - 性能直接影响集群规模（单集群建议不超过5000节点）。

  > #### **API Server**
  >
  > - **核心角色**：
  >   集群的“网关”，所有组件仅通过API Server操作etcd。
  > - **关键特性**：
  >   - **认证与鉴权**：支持Token、证书、RBAC等机制。
  >   - **准入控制**：通过Webhook修改或验证请求（如资源配额检查）。
  >
  > ------
  >
  > #### **Controller Manager**
  >
  > - **核心机制**：
  >   通过**调谐循环（Reconcile Loop）** 确保资源状态与声明一致。
  >
  >   - controller-mange r会通过api server 访问 etcd，来获取每种控制器的状态
  >   
  >   - 一旦控制器的状态不符合预期，就会根据该控制器的特性，来进行调谐
  >   
  > - **典型控制器**：
  >
  >   | 控制器          | 管理资源       | 功能场景                                       |
  >   | :-------------- | :------------- | :--------------------------------------------- |
  >   | **Deployment**  | Pod副本集      | 实现滚动更新、回滚。                           |
  >   | **ReplicaSet**  | Pod副本数      | 确保指定数量的 **Pod 副本** 始终在集群中运行。 |
  >   | **StatefulSet** | 有状态应用     | 提供稳定网络标识和持久存储。                   |
  >   | **DaemonSet**   | 节点级守护进程 | 如日志收集Agent（每节点运行一个Pod）。         |
  >   | **CronJob**     | 定时任务       | 替代Linux Cron的分布式任务调度。               |
#### 1.2 **Worker 节点（工作节点）**

- **核心职责**：
  运行容器化应用，接收Master指令并上报状态。

- **关键组件**：

  | 组件           | 功能描述                                                     |
  | :------------- | :----------------------------------------------------------- |
  | **kubelet**    | 1. 上报节点资源/Pod状态至API Server，并存入 etcd。 <br />2. 调用容器引擎创建Pod。 |
  | **kube-proxy** | 维护Service的负载均衡规则（基于IPVS/iptables），实现Pod间通信。 |
  | **容器引擎**   | 实际运行容器的组件（如containerd、Docker）。                 |

  > #### **kubelet**
  >
  > - **核心功能**：
  >
  >   1. 节点状态上报（CPU/内存/Pod状态）。
  >
  >   2. 根据API Server指令（来自 master 的请求）创建Pod：
  >
  >      - **容器引擎对接**：
  >
  >        <img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/v2-506c2575e6bd73d014afbf29d3850ef9_1440w.jpg" alt="img" style="zoom:50%;" />
  >
  >        | Kubernetes版本      | 对接方式                                                     |
  >        | :------------------ | :----------------------------------------------------------- |
  >        | **<1.24**（已废弃） | kubelet → Docker Shim（在 kubelet 进程中） → dockerd → containerd。 |
  >        | **≥1.24**（更高效） | kubelet（在 containerd 进程中） → CRI插件（containerd内置）→ containerd。 |
  >
  >        对接 containerd 的优点：
  >
  >        1. 调用路径更短，效率更高
  >        2. k8s的kubelet版本更新迭代更便捷
  >
  >   3. **垃圾回收（GC）**：自动清理无用镜像和已退出容器。
  >
  > ------
  >
  > #### **kube-proxy**
  >
  > **1. 核心作用**
  >
  > - **核心职责**：
  >   是负责实现 **Service 资源** 的组件，专注于为 Pod 提供 **四层（传输层，基于TCP/UDP协议的端口和IP地址）的智能负载均衡**。
  >   - **不负责 Pod 网络**：Pod 的网络连通性（如跨节点通信、IP 分配等）由 **CNI 网络插件**（如 Calico、Flannel、Cilium）实现。
  >
  >   - **负载均衡机制**：通过维护 Service 的访问规则，将请求流量分发到后端 Pod，并确保流量的均衡性。
  >
  > - **关键行为**：
  >   - 监听 API Server 的 Service 和 Endpoints 对象变化。
  >   - 动态更新负载均衡规则（如 IPVS 表或 iptables 规则），确保流量转发到健康的 Pod。
  >
  > ------
  >
  > **2. Service 的核心特点**
  >
  > - **动态关联 Pod（标签选择机制）**
  >
  >   - **标签选择器（Label Selector）**：
  >     Service 通过标签（Labels）动态关联后端 Pod。只有匹配 `spec.selector` 的 Pod 才会被纳入 Service 的流量分发范围。
  >
  >     - **示例**：
  >
  >       ```yaml
  >       apiVersion: v1
  >       kind: Service
  >       metadata:
  >         name: nginx-svc
  >       spec:
  >         selector:
  >           app: nginx  # 关联所有含app=nginx标签的Pod
  >         ports:
  >           - protocol: TCP
  >             port: 80
  >             targetPort: 9376
  >       ```
  >
  >   - **动态维护机制**：
  >
  >     - **Endpoints 对象**：Service 的实际代理目标由 Endpoints 对象维护，该对象实时存储匹配标签的 Pod 的 IP:Port 列表。
  >     - **自动更新**：当 Pod 发生扩缩容、重启或故障时，Endpoints 自动更新，kube-proxy 同步调整负载均衡规则，剔除不可用 Pod 或添加新 Pod。
  >
  > - ##### **流量负载均衡（基于 IPVS）**
  >
  >   - **底层技术演进**：
  >     - **早期模式**：Kube-Proxy 曾使用 iptables 或 userspace 模式，但存在性能瓶颈（如规则线性匹配效率低）。
  >     - **当前主流**：**IPVS（IP Virtual Server）**，专为大规模负载均衡设计，支持多种调度算法（如轮询、加权轮询、最少连接等），性能更优。
  >   - **IPVS核心优势**：
  >     - **高效性**：基于内核哈希表实现 O(1) 时间复杂度，适合高并发场景。
  >     - **灵活性**：支持多种负载均衡算法（`rr`、`wrr`、`lc`、`sh` 等）。
  >       - **哈希表是「加速器」**：解决**已建立连接**的快速转发问题（性能核心）。
  >       - **调度算法是「决策者」**：解决**新连接**的服务器选择问题（逻辑核心）。
  >     - **透明性**：对应用无侵入，流量直接转发到 Pod，无需额外代理层。
  >   - **四层负载均衡**：
  >     Service 工作在传输层（TCP/UDP），基于 IP 和端口进行流量分发，不解析应用层协议（如 HTTP）。
  >
  > ------
  >
  > **3. 工作机制全景图**
  >
  > ```
  >    Client
  >      │
  >      ▼
  > [ Service VIP (ClusterIP) ]
  >      │
  >      ▼
  > [ Kube-Proxy (IPVS) ]  → 动态监听 Endpoints 变化
  >      │
  >      ├───▶ Pod 1 (IP:Port)  [健康状态通过 kubelet/Endpoint Controller 更新]
  >      ├───▶ Pod 2 (IP:Port)
  >      └───▶ Pod 3 (IP:Port)
  > ```
  >
  > ------
  >
  > **4. 关键补充说明**
  >
  > - **Service 类型**：
  >   虽然未在原始内容中提及，Service 支持多种类型（如 `ClusterIP`、`NodePort`、`LoadBalancer`），但无论哪种类型，流量代理均由 kube-proxy 实现。
  > - **健康检查**：
  >   Service 不直接检测 Pod 健康状态，依赖底层组件（如 kubelet 和 Endpoint Controller）通过 Pod 的 `readinessProbe`/`livenessProbe` 更新 Endpoints。
  >
  > ------
  >
  > **5. 总结对比**
  >
  > | 组件/技术      | 职责                         | 关键特点                       |
  > | :------------- | :--------------------------- | :----------------------------- |
  > | **kube-proxy** | Service 负载均衡规则维护     | 基于 IPVS，动态更新，四层转发  |
  > | **Service**    | 暴露 Pod 服务，抽象访问入口  | 标签选择器、ClusterIP/VIP 机制 |
  > | **Endpoints**  | 维护 Service 关联的 Pod 列表 | 实时更新，与 Pod 生命周期同步  |

### 2. 核心概念

#### （1）组件和资源

- **组件**：
  - 定义：实际存在的软件。
  - 特点：具有实际运行代码，能够执行特定功能。
- **资源**：
  - 定义：由组件创建出的逻辑单位。
  - 特点：用于管理和配置组件的运行。
  - 分类及举例：
    - **Namespace**：用于隔离不同租户或项目的资源，避免命名冲突。
    - **Pod**：Kubernetes中的最小部署单元，包含一个或多个容器，共享网络和存储资源。
    - **Deployment**：用于管理Pod的创建、更新和删除，确保指定数量的Pod副本在运行。
    - **ConfigMap**：用于存储配置文件或命令行参数等配置数据，供Pod使用。
    - **StatefulSet**：用于管理有状态的应用，确保Pod的稳定网络标识和持久存储。

#### （2）Namespace（命名空间）
- **技术本质**：
  资源隔离技术，将集群虚拟化为多个逻辑分区。
- **作用场景**：
  - **多租户隔离**：不同团队/项目共用一套k8s集群时，使用独立Namespace，避免资源冲突。
  - **统一管理**：
    - **Resource Quota**：限制Namespace的总CPU/内存用量。
    - **LimitRange**：限制单个Pod/容器的资源上限。
- **与容器命名空间的区别**：
  Kubernetes Namespace隔离的是资源对象（如Pod、Service），而非底层Linux命名空间（PID、网络等）。

#### （3）Pod
- **定义**：
  Kubernetes最小工作/调度单元，包含一个或多个**共享资源**的相关容器。（一个pod内至少有两个容器，因为启动一个容器就会同时启动一个pause容器为其配置网络相关信息，一个pod对应一个pause）

- **核心特性**：

  | 特性           | 说明                                                |
  | :------------- | :-------------------------------------------------- |
  | **共享网络**   | 所有容器共享同一IP和端口空间（通过Pause容器实现）。 |
  | **共享存储卷** | 容器可访问Pod级别的Volume（如emptyDir、NFS）。      |
  | **原子性调度** | Pod内所有容器必须调度到同一节点（主机）。           |

- **设计意义**：

  - **封装容器差异**：以Pod为统一接口，屏蔽底层容器引擎差异。
  - **支持多容器协作**：如日志收集Sidecar与应用容器共享日志目录。

#### （4）Service
- **功能**：为Pod提供稳定的虚拟IP和DNS名称，实现负载均衡与服务发现。  
- **动态关联**：通过Label Selector自动关联后端Pod。  
- **底层实现**：基于IPVS或iptables规则。  
## 五、List-Watch机制

<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/1550788-20221206154649327-573443108.png" alt="k8s的list-watch机制 - BetterManEddy - 博客园" style="zoom:50%;" />

- **设计目标**：实现 Kubernetes 组件（如 Controller Manager、Scheduler 等）与 API Server 之间的 **实时资源状态同步**。

  - **Controller Manager**：监听 `ReplicaSet`、`Deployment`、`Pod` 等资源。
  - **Scheduler**：监听未调度（`spec.nodeName` 为空）的 `Pod` 资源。
  - **Kubelet**：监听分配给本节点（`spec.nodeName` 匹配自身节点名）的 `Pod` 资源。

- **两种操作模式**：

  | 操作      | 连接类型 | 适用场景                 | 特点                             |
  | :-------- | :------- | :----------------------- | :------------------------------- |
  | **List**  | 短连接   | 低频访问（如初始化同步） | 一次性获取全量资源数据           |
  | **Watch** | 长连接   | 高频监听（如实时更新）   | 持续监听资源变化事件（增/删/改） |

  > **watch 是如何通过 HTTP 长连接接收 api server 发来的资源变更事件?**
  >
  > **问题**：长连接需要服务器在开始发送消息体前先发送Content-Length消息头字段。但是对于动态生成的内容来说，在内容创建完之前，Content-Length是不可知的。对于一些很耗时的动态操作，服务器要等到所有操作完成后，才能发送数据，显然这样做的效率不高。
  >
  > **解决方法**（采用流模式取代缓存模式）:
  >
  > - **HTTP/1.1**：
  >
  >   使用 **分块传输编码（Chunked Transfer Encoding）**，将响应体分割为多个块（chunks），并以一个或多个块发送。
  >
  > - **HTTP/2**：
  >   支持多路复用和流式传输，性能更优，Kubernetes 逐步向 HTTP/2 迁移。

- ##### **工作流程**

  1. **初始化阶段**：
     组件首次启动时，通过 **List 操作** 从 API Server 获取当前所有资源的全量数据（如全部 Pod 列表）。
  2. **持续监听阶段**：
     基于 List 返回的 **Resource Version**（资源版本号），发起 **Watch 长连接**，监听该资源后续的增量变更事件。
     - **Resource Version**：
       每个 K8s 资源对象携带一个唯一版本号，API Server 保证版本号严格递增。
       - List 返回的 Resource Version 表示当前全量数据的快照版本。
       - Watch 使用该版本号作为起点，确保不遗漏后续变更。
  3. **容错机制**:
     - **网络中断恢复**：
       - 若 Watch 连接断开（如网络抖动），组件重新发起 **List 操作** 获取全量数据，并基于最新 `Resource Version` 重建 Watch 连接。
       - 确保事件不丢失（API Server 会返回所有 `Resource Version` 之后的事件）。
     - **定期 Resync**：
       - Controller Manager 定期（默认 10 小时）执行全量 List，防止 Watch 事件遗漏导致状态不一致。

- **优点**：

  | **优点**       | **核心原理**                                   | **简要解释**                                                 |
  | :------------- | :--------------------------------------------- | :----------------------------------------------------------- |
  | **消息可靠性** | 基于 `ResourceVersion` 断点续传 + 全量补偿机制 | 若 Watch 中断，客户端通过 List 获取全量数据（携带最新版本号），确保事件不丢失，数据最终一致。 |
  | **消息实时性** | 长连接事件推送（非轮询）                       | 资源变更后，API Server **立即推送事件**，避免轮询延迟（如传统 HTTP 轮询【通过定时发送HTTP请求来模拟实时数据传输】秒级延迟）。 |
  | **消息顺序性** | 全局单调递增的 `ResourceVersion`               | etcd 保证所有事件按版本号严格有序，客户端按顺序处理，避免状态混乱（如先删后增的时序问题）。 |
  | **高性能**     | 增量传输 + 长连接复用                          | 仅传输变更事件（非全量数据），且watch作为异步消息通知机制，复用一条TCP连接 |

## 六、k8s 核心交互流程

### 1. 🌟🌟🌟**Pod创建流程**

<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/1550788-20221206154649327-573443108.png" alt="k8s的list-watch机制 - BetterManEddy - 博客园" style="zoom:50%;" />

> 0、controller-manager、scheduler、kubelet均监听API-Server，只watch自己关注的资源  
>
> 1、kubectl提交创建pod副本的请求---> API-server
>
> 2、创建pod副本的事件记入ETCD 	3、上报事件给 API-server
>
> 4、controller-manager从API-server那里watch到创建pod副本的事件，会调用具体的控制器来工作，不同的控制器有不同的特性，例如Replicaset控制器，该控制器的核心作用就是控制pod数到指定个数，如果kubectl提交过来的是创建3个pod，而当前pod为2个，则需要新增一个pod
>
> 5、创建Pod的事件提交给API-server	
>
> 6、 API-server将该事件记入ETCD	7、上报事件给 API-server
>
> 8、scheduler从API-server那里watch到创建pod副本的事件，负责选出一个工作节点来创建pod，筛选过程分为两个阶段
>
> - （1） 预选：根据污点、选择器等筛选出符合条件的节点
>
> - （2） 优选：从筛出的节点里进行打分，选择资源最优的
>
> 9、将选出的节点提交给API-server 
>
> 10、记入etcd	11、上报事件给 API-server
>
> 12、指定的工作节点上的的kubelet组件watch到创建POD的事件，开始创建POD 

##### **阶段 0：组件初始化与全量同步**

- **Controller Manager**：
  - 执行 **List 操作** 获取所有 `ReplicaSet` 和 `Pod` 的全量数据。
  - 记录当前 `Resource Version`，建立 **Watch 长连接** 监听后续变更。
- **Scheduler**：
  - 执行 **List 操作** 获取所有未调度的 `Pod` 全量数据。
  - 基于 `Resource Version` 建立 **Watch 连接**，持续监听新创建的 `Pod`。
- **Kubelet**：
  - 执行 **List 操作** 获取已分配到本节点的 `Pod` 全量数据。
  - 建立 **Watch 连接**，监听本节点 `Pod` 的创建/更新事件。
##### **阶段 1：用户提交请求**

```bash
# 用户创建 ReplicaSet（隐含 Pod 模板）
$ kubectl apply -f replicaset.yaml
```

##### **阶段 2：API Server 写入 ReplicaSet**

1. **API Server** 接收请求，将 ReplicaSet 定义写入 **etcd**。
2. **触发事件**：生成 `ReplicaSet Added` 事件，广播给监听 `ReplicaSet` 的组件。
##### **阶段 3：Controller Manager 调谐**

1. **Watch 触发**：
   - Controller Manager 的 ReplicaSet 控制器监听到 `ReplicaSet Added` 事件。
   - 对比 `spec.replicas` 和当前 Pod 数量（初始为 0），触发调谐逻辑。
2. **创建 Pod**：
   - 发起创建 Pod 的请求（携带 ReplicaSet 的标签选择器）。
3. **API Server 处理**：
   - 将 Pod 定义写入 etcd（此时 `spec.nodeName` 为空，标记为未调度）。
   - 生成 `Pod Added` 事件。
##### **阶段 4：Scheduler 监听与调度**

1. **Watch 触发**：
   - Scheduler 监听到 `Pod Added` 事件（未调度状态）。
2. **调度决策**：
   - **预选（Filter）**：排除不满足条件的节点（如资源不足）。
   - **优选（Score）**：对候选节点评分（如负载均衡、亲和性）。
3. **绑定节点**：
   - 更新 Pod 的 `spec.nodeName` 为目标节点（如 `node-1`）。
   - API Server 将更新后的 Pod 写入 etcd，生成 `Pod Updated` 事件。
##### **阶段 5：Kubelet 创建容器**

1. **Watch 触发**：
   - 目标节点（如 `node-1`）的 Kubelet 监听到 `Pod Updated` 事件（`spec.nodeName` 指向自身）。
2. **容器创建流程**：
   - **Step 1：拉取镜像**
     - 检查本地镜像缓存，若缺失则从容器仓库拉取。
   - **Step 2：创建 Pause 容器**
     - 调用容器引擎（如 containerd）创建 Infra 容器，初始化共享网络命名空间。
   - **Step 3：配置网络**
     - 调用 CNI 插件（如 Calico）为 Pod 分配 IP，设置网络规则（路由、iptables）。
   - **Step 4：创建业务容器**
     - 以共享网络模式启动业务容器（如 Nginx），挂载存储卷（如有）。
3. **状态上报**：
   - Kubelet 将 Pod 状态（`Running`）更新至 API Server，写入 etcd。
##### **阶段 6：Controller Manager 终态确认**

1. **Watch 触发**：
   - Controller Manager 监听到 Pod 状态变更为 `Running`。
2. **更新 ReplicaSet 状态**：
   - 将 `ReplicaSet.status.availableReplicas` 更新为 1。
   - API Server 写入 etcd，流程闭环。
