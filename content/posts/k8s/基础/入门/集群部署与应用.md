#### **Kubernetes 版本说明**
- Kubernetes 目前没有官方长期支持版（LTS），需自行选择稳定版本。

- 部署方式对比：
  
  - **kubeadm 工具**：简化部署
    
    - 容器引擎、kubelet：使用yum安装
    
      - 为什么master节点需要安装容器引擎和kubelet？
        - 容器引擎（containerd）：master节点的关键组件需要以静态pod形式部署，这些pod需要通过容器引擎启动容器；
        - kubelet：
        
          - **是一个守护进程**：
        
            - > **启动过程：**
              >
              > 1. **系统初始化 (如 systemd)**：安装 kubelet 后，kubelet 会被自动注册为一个系统服务（例如 `kubelet.service`）。然而，**在 `kubeadm init/join`命令成功完成之前，`kubelet`服务是处于 `inactive (dead)`或 `failed`状态的。** 这是 `kubeadm`的故意设计，因为此时 kubelet 的配置还不完整（缺失 kubeconfig、正确的参数等），如果提前启动可能会出错或行为不正确。
              > 2. 启动的 `kubelet`会：
              >    - 加载它的配置（通常是 `/etc/kubernetes/kubelet.conf`或由 `--config`参数指定的文件，这些文件由 `kubeadm`生成，告知它如何连接到 API Server、使用哪个容器运行时、以及寻找静态 Pod 定义文件的路径（对于静态 Pod 场景））。
              >    - 连接到本地的 API Server（其静态 Pod 清单 `kubeadm`已经写入 `/etc/kubernetes/manifests/`，`kubelet`会立即检测并启动它们）。
              >    - 开始管理节点上的容器：首先启动控制平面组件（API Server, etcd, Controller Manager, Scheduler）的静态 Pod，然后是后续通过 API Server 调度的 Pod（如 CoreDNS, kube-proxy, CNI 插件）。
          - **直接管理静态Pod**：kubelet负责监控节点上的静态Pod配置文件（如`/etc/kubernetes/manifests`），并根据文件内容自动创建/更新控制平面组件（如API Server）。
          - **与集群交互**：kubelet作为节点代理，向API Server上报节点状态（包括静态Pod的运行状态），即使是在master节点上也需要这一功能。
        - **自举（Bootstrap）机制**：kubeadm通过kubelet启动master节点的控制平面组件（静态Pod），而kubelet自身需依赖容器引擎运行这些组件，形成闭环。
        - **统一架构**：Kubernetes要求所有节点（包括master）遵循相同的基础设施模型，避免特殊化配置。
    
    - 关键组件以容器形式运行（静态 Pod），适合快速搭建环境。
    
      - **静态pod**：
    
        > **核心特性**
        >
        > 1. **本地管理**
        >    静态Pod的配置以文件形式（YAML/JSON）存储在节点特定目录（如 `/etc/kubernetes/manifests`），由kubelet直接监控并创建，无需通过Kubernetes API Server。
        > 2. **用途场景**
        >    常用于运行关键系统组件（如kube-apiserver、kube-scheduler自身），尤其是在`kubeadm`部署的集群中，这些控制平面组件默认以静态Pod形式运行。
        > 3. **生命周期**
        >    - 由节点本地的kubelet全权管理，**不受Deployment/DaemonSet等控制器调度**。
        >    - 节点宕机时，静态Pod不会自动迁移到其他节点。
        > 4. **配置更新**
        >    修改静态Pod文件后，kubelet会自动重启Pod生效，无需通过`kubectl`命令操作。
        > 5. **集群可见性**
        >    kubelet会将静态Pod状态上报API Server，其名称会附带节点后缀（如`my-pod-node1`），但删除需在节点本地操作。
    
        | 特性       | 静态Pod                 | 普通Pod                         |
        | :--------- | :---------------------- | :------------------------------ |
        | 创建方式   | kubelet直接读取本地文件 | 通过API Server提交（如kubectl） |
        | 调度与管理 | 不受控制器管理          | 由Deployment/DaemonSet等管理    |
        | 高可用性   | 依赖节点存活            | 可通过控制器跨节点重建          |
        | 配置更新   | 直接修改本地文件        | 通过`kubectl apply`更新         |
    
  - **二进制安装**（不推荐）：手动安装所有组件，灵活但复杂度高。
  
  - **Kind ：快速搭建本地 Kubernetes 集群**
  
    - **全称**：Kubernetes IN Docker
    - **定位**：一款轻量级工具，用于在 **Docker 容器** 中快速创建本地 Kubernetes 集群。
    - **核心功能**：
      - 通过**容器模拟 Kubernetes 节点**（控制平面和工作节点）。
      - 支持多节点集群，适用于开发、测试和 CI/CD 场景。
  
    - **核心优势**
  
      | **优势**               | **说明**                                                     |
      | :--------------------- | :----------------------------------------------------------- |
      | **轻量快速**           | 基于 Docker 容器，秒级启动集群，资源占用低（对比 VM 方案如 Minikube）。 |
      | **多节点支持**         | 可模拟生产环境的多节点拓扑（如 1 主节点 + 2 工作节点）。     |
      | **与 Kubernetes 兼容** | 使用官方 `kubeadm` 部署集群，100% 兼容原生 Kubernetes 行为。 |
      | **易于销毁重建**       | 一键删除集群，避免环境残留问题。                             |
      | **跨平台支持**         | 支持 Linux、macOS、Windows（需 Docker Desktop）。            |
  
    - **架构与工作原理**
  
      - **架构图**
  
        ```
        +-------------------+       +-------------------+
        |   Docker Host     |       |   Docker Host     |
        | +---------------+ |       | +---------------+ |
        | | Kind Cluster  | |       | | Kind Cluster  | |
        | | - Control     | |       | | - Worker      | |
        | | - Worker      | |       | | - Worker      | |
        | +---------------+ |       | +---------------+ |
        +-------------------+       +-------------------+
        ```
  
    - **关键组件**
  
      - **Docker 容器作为节点**：
        - 每个 Kubernetes 节点（控制平面或工作节点）运行在一个独立的 Docker 容器中。
        - 容器内预装 `kubelet`、`kube-apiserver`、`containerd` 等组件。
  
      - **kubeadm 初始化**：
        - 使用 `kubeadm` 工具自动化部署 Kubernetes 集群。
  
      - **本地镜像加载**：
        - 支持通过 `kind load docker-image` 将本地镜像直接加载到集群节点，无需推送至远程仓库。
        
          > - 当你在 K8s YAML 中指定 `image: your-local-image:tag`并创建 Pod 时：
          >   - 调度器将其调度到一个 `kind`节点上（通常是任意一个节点，除非指定）。
          >   - 该节点的 Kubelet 询问本地容器运行时：“有 `your-local-image:tag`这个镜像吗？”
          >   - 容器运行时回答：“有！（因为 `kind load`刚才塞给我了）”
          >   - 容器运行时直接用这个已经存在的本地镜像启动容器。
          > -  整个过程只涉及本地数据传输，速度极快，不需要互联网访问远程仓库，镜像也不会离开你的本地环境。
#### **环境规划**
- **节点规划**：
  - Master: `192.168.88.134`
  - Node01: `192.168.88.135`
  - Node02: `192.168.88.136`
- **硬件要求**：
  - 每台机器内存 ≥ 2GB
  - 配置静态 IP 地址
  - 系统时间同步（NTP）
#### **前置配置（所有节点执行）**
1. **禁用 Swap 和防火墙**：
   ```bash
   swapoff -a && sed -i '/ swap / s/^/#/' /etc/fstab  # 永久禁用 Swap
   systemctl stop firewalld && systemctl disable firewalld  # 关闭防火墙
   setenforce 0 && sed -i 's/^SELINUX=.*/SELINUX=disabled/' /etc/selinux/config  # 关闭 SELinux
   ```

2. **配置 containerd 镜像加速**：
   ```bash
   mkdir -p /etc/containerd/certs.d/docker.io
   cat > /etc/containerd/certs.d/docker.io/hosts.toml << EOF
   server = "https://docker.io"
   [host."https://dockerproxy.com"]
     capabilities = ["pull", "resolve"]
   [host."https://docker.m.daocloud.io"]
     capabilities = ["pull", "resolve"]
   [host."https://docker.agsv.top"]
     capabilities = ["pull", "resolve"]
   [host."https://registry.docker-cn.com"]
     capabilities = ["pull", "resolve"]
   EOF
   systemctl restart containerd
   ```
#### **部署步骤**


0. **环境准备(在所有节点上执行)**

   - **操作系统要求：** 满足 Kubernetes 对操作系统版本、内核参数、交换空间、cgroup 驱动等的特定要求。

   - **容器运行时安装：** 安装并配置 Docker、containerd 或 CRI-O 等容器运行时。

   - **安装 `kubeadm`、`kubelet`、`kubectl`：** 在所有节点（主节点和工作节点）上安装这三个核心二进制包。

     ```
     yum install -y kubelet-1.30* kubeadm-1.30* kubectl-1.30*
     systemctl enable kubelet
     ```

     `kubelet`会被注册为系统服务（如 systemd），但**此时 `kubelet`服务是停止状态**，等待 `kubeadm`的指令。

1. **Master 节点初始化**：

   ```bash
   kubeadm init \
     --config=kubeadm.yaml \
     --ignore-preflight-errors=SystemVerification \
     --ignore-preflight-errors=Swap
   ```
   > 它通常会执行以下一系列操作：
   >
   > - **预检：** 检查系统状态（内核版本、内存、CPU、端口占用、容器运行时状态、`kubelet`服务状态等），确保环境符合要求。
   > - **生成证书：**
   >   - 创建 `/etc/kubernetes/pki`目录。
   >   - 使用内置的 `kubeadm`PKI 工具生成集群所需的所有 TLS 证书和密钥（CA 根证书、API Server 证书、etcd 证书、前端代理证书等）。这些证书用于集群组件间的安全通信。
   >   - 默认情况下，证书有效期是 1 年（生产环境需要考虑轮换）。
   > - **生成 kubeconfig 文件：**
   >   - 创建 `/etc/kubernetes`目录。
   >   - 生成 `admin.conf`、`kubelet.conf`、`controller-manager.conf`、`scheduler.conf`等 kubeconfig 文件。这些文件包含了访问 API Server 所需的客户端证书、密钥和 API Server 地址信息。
   > - **启动 etcd：**
   >   - `kubeadm`将 `etcd`的 Pod 定义文件（YAML）写入 `/etc/kubernetes/manifests/etcd.yaml`。
   >   - 因为 `kubelet`被配置为监视 `/etc/kubernetes/manifests`目录（通过 `kubeadm`设置的 `kubelet`参数 `--pod-manifest-path`），它会**立即检测到这个文件**。
   >   - `kubelet`**启动本机容器运行时**（如 containerd），根据 `etcd.yaml`的描述启动一个 `etcd`**容器**。这是集群的第一个关键组件。
   > - **启动控制平面组件：**
   >   - 类似地，`kubeadm`将 `kube-apiserver`、`kube-controller-manager`、`kube-scheduler`的 Pod 定义文件写入 `/etc/kubernetes/manifests/`。
   >   - `kubelet`检测到这些文件，依次启动这些组件的**容器**。
   >   - 这些组件（API Server, Controller Manager, Scheduler）**以静态 Pod 的形式运行在主节点上**，由该节点的 `kubelet`直接管理。
   > - **配置 RBAC：** 创建必要的 ClusterRoleBinding（如 `cluster-admin`绑定到 `system:masters`组）。
   > - **应用核心插件：**
   >   - **DNS (CoreDNS)：** `kubeadm`应用一个 Deployment 来部署 CoreDNS。这个 Deployment 是通过 API Server 提交的。
   >   - **Proxy (kube-proxy)：** `kubeadm`应用一个 DaemonSet 来部署 kube-proxy。同样通过 API Server 提交。
   > - **生成加入令牌：** 创建一个包含 CA 证书哈希和令牌的 `kubeadm join`命令。这个命令用于后续工作节点加入集群。
   > - **输出重要信息：** 打印出 `kubectl`的配置说明（如何将 `admin.conf`复制到用户家目录下的 `.kube/config`）和 `kubeadm join`命令。
   > - **启动 `kubelet`服务：** 在完成所有初始化步骤后，`kubeadm`会**启动 `kubelet`系统服务**（`systemctl start kubelet`）。此时，`kubelet`已经连接到本地的 API Server（通过 `/etc/kubernetes/kubelet.conf`），并开始正常工作，管理静态 Pod（控制平面组件）和接收来自 API Server 的指令（管理 CoreDNS、kube-proxy 等动态 Pod）。
   >
   > **关键点：`kubeadm init`本身不直接启动容器。它准备配置（证书、kubeconfig、静态 Pod 清单），然后依靠 `kubelet`来启动控制平面组件（etcd, API Server 等）的容器。`kubelet`是容器世界的“启动引擎”。**

   - 初始化成功后**记录 `kubeadm join` 命令**（用于 Node 节点加入）。

2. **配置 kubectl（Master 节点）**：

   ```bash
   mkdir -p $HOME/.kube
   cp -i /etc/kubernetes/admin.conf $HOME/.kube/config
   chown $(id -u):$(id -g) $HOME/.kube/config
   ```

   > - 按照 `kubeadm init`的输出提示，将 `/etc/kubernetes/admin.conf`复制到 `~/.kube/config`。
   > - 设置正确的文件权限（`chmod 600 ~/.kube/config`）。
   > - 现在可以使用 `kubectl`命令管理集群了（如 `kubectl get nodes`，此时主节点状态应为 `NotReady`，因为网络插件尚未安装）。

3. **安装 Pod 网络插件（CNI，这里选用 Flannel）（Master 节点）**：

   ```bash
   kubectl apply -f https://raw.githubusercontent.com/flannel-io/flannel/master/Documentation/kube-flannel.yml
   ```
   > - Kubernetes 集群**必须**安装一个 CNI 插件才能让 Pod 相互通信。
   > - 选择一个 CNI 插件（如 Calico, Flannel, Weave Net, Cilium）。
   > - 使用 `kubectl apply -f <url-to-plugin-manifest>`命令部署该插件的 DaemonSet 和/或其他资源。
   > - 插件部署完成后，节点状态会变为 `Ready`。

   若镜像拉取失败，替换为国内镜像源：
   ```yaml
   # 修改 kube-flannel.yml 中的镜像地址为：
   ```

4. **Node 节点加入集群（Worker 节点）**：

   ```bash
   kubeadm join 192.168.71.121:6443 --token abcdef.0123456789abcdef \
           --discovery-token-ca-cert-hash sha256:f8cf22d2382b09786f2467f1e830215f4eb4c501d1f21b3f9d231e82ebb96033
   ```

   > - `kubeadm join`在工作节点上的工作：
   >   - **预检：** 类似 `init`的预检。
   >   - **连接 API Server：** 使用令牌和 CA 哈希向主节点的 API Server 发起请求，进行身份验证。
   >   - **获取配置：** 从 API Server 获取该节点加入集群所需的必要信息（集群信息、kubelet 的 kubeconfig 文件内容）。
   >   - **写入 kubelet 配置：** 将获取到的 kubeconfig 写入 `/etc/kubernetes/kubelet.conf`。
   >   - **启动 `kubelet`服务：** 启动 `kubelet`系统服务（`systemctl start kubelet`）。
   > - **节点注册：** `kubelet`启动后，使用 `/etc/kubernetes/kubelet.conf`连接到 API Server，向 API Server 注册自己（报告节点信息）。
   > - **调度 Pod：** 控制平面（API Server, Scheduler）检测到新节点注册后，可以将 Pod 调度到该节点上运行。`kubelet`负责在节点上拉取镜像并运行这些 Pod。
   > - **网络插件生效：** CNI 插件的 DaemonSet 会在新节点上自动部署一个 Pod（通常是 `calico-node`, `flanneld`等），该 Pod 负责配置该节点的 Pod 网络。
#### **故障排查与重置环境**
- **若部署失败**：
  
  1. **Master 节点重置**：
     ```bash
     kubeadm reset -f
     ```
  2. **所有节点清理残留文件**：
     ```bash
     rm -rf ~/.kube/ /etc/kubernetes/ /etc/cni /opt/cni /var/lib/etcd /var/etcd
     rm -rf /run/flannel /run/xtables.lock
     systemctl stop kubelet
     yum remove kube* -y
     for i in $(df | grep kubelet | awk '{print $NF}'); do umount -l $i; done
     rm -rf /var/lib/kubelet /etc/systemd/system/kubelet.service*
     iptables -F
     reboot
     ```
#### **验证部署**
- **检查节点状态**：
  ```bash
  kubectl get nodes  # 所有节点应显示 Ready
  ```
- **检查 Pod 状态**：
  ```bash
  kubectl get pods -n kube-system  # 确保 CoreDNS、Flannel 等组件运行正常
  ```
#### **注意事项**
- 确保所有节点时间同步（使用 `chronyd` 或 `ntpd`）。
- 若使用代理，需配置 Docker、kubelet 等服务的代理环境变量。
- 生产环境建议关闭 Swap 以提高稳定性。
