### 1. 核心概念
#### 1.1 **优先级类(PriorityClass)**
- 包含优先级数值的K8s**资源对象**，数值越大优先级越高
- 通过**`priorityClassName`**字段与Pod关联
- 特性：
  - 非命名空间资源，集群范围内 `PriorityClass` 的资源，可以被所有命名空间中的 Pod 引用
  - 数值范围：32位整数，超过一亿的数字被预留给系统组件，未配置默认为0
  - 在 `PriorityClass` 资源中设置 `globalDefault: true` 时，集群中任何没有明确指定 `priorityClassName` 的 Pod 都将自动使用这个全局默认的 `PriorityClass`。
  - 不可更新（创建后不可修改value值）

#### 1.2 **强制抢占机制**
- 高优先级Pod可抢占节点资源
- 被抢占的Pod进入终止状态（优雅终止期默认30秒）
- 抢占过程由调度器自动完成
### 2. 应用场景
#### 2.1 资源优化
- 提高大规模集群资源利用率
- 支持资源超配策略（错峰使用）
  - **资源超配策略**：允许将多个 Pod 的资源请求总量配置得超过该节点的实际物理资源总量（有多个pod并未设置资源请求/请求量<限制量）

- 解决重要Pod因资源不足pending的问题

#### 2.2 业务分级
- 保障关键业务优先运行
- 实现服务分级调度策略
- 应对突发流量时的资源保障
### 3. 核心机制对比

| 机制     | 触发条件             | 执行组件       | 决策依据                                  | 影响范围          |
| -------- | -------------------- | -------------- | ----------------------------------------- | ----------------- |
| **调度** | Pod新建/更新         | kube-scheduler | 节点资源+调度策略                         | 待调度Pod         |
| **抢占** | 高优先级Pod无法调度  | kube-scheduler | PriorityClass值                           | 运行中低优先级Pod |
| **驱逐** | 节点资源压力达到阈值 | kubelet        | 资源用量（触发条件）+ QoS等级（驱逐依据） | 本节点Pod         |

#### 3.1 抢占逻辑
- **不参考QoS**：仅依据PriorityClass数值
- 选择策略：
  1. 优先驱逐最低优先级Pod
  2. 移除最低优先级Pod不足以让调度程序调度抢占式Pod/受PDB保护时考虑次低优先级
  3. 相同优先级时按资源用量排序

#### 3.2 驱逐逻辑
- **QoS主导**：BestEffort > Burstable > Guaranteed
- 保护机制：
  - 资源用量未超requests的Pod受保护
  - 高优先级Pod若超用requests可能被驱逐
### 4. 配置实践

#### 4.1 创建优先级类
```yaml
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: high-priority
value: 1000000                # 优先级数值（必需字段）
globalDefault: false          # 是否设为集群默认
description: "重要业务专用优先级"
```

#### 4.2 资源占满示例（StatefulSet）
```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: resource-consumer
spec:
  template:
    spec:
      containers:
      - name: busybox
        resources:
          requests:
            cpu: 1           # 申请1核CPU
            memory: 1Gi      # 申请1GB内存
          limits:
            cpu: 1
            memory: 1Gi
```

> 在 K8s 中，判断一个节点的资源是否被 Pod 占满，主要依据的是 Pod 的**资源请求**（Requests），而不是真正在 Pod 中运行的进程实际占用的资源量。
>
> 而**驱逐**决策主要基于**节点的实际资源使用情况**。

#### 4.3 高优先级Pod配置
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: critical-pod
spec:
  containers:
  - name: nginx
    resources:
      requests:
        cpu: 1               # 同样申请1核CPU
        memory: 2Gi          # 但申请更多内存
  priorityClassName: high-priority  # 关联高优先级类
```
### 5. 关键特性说明

- **调度优先级**：

   抢占优先级高于其他调度策略（如反亲和性）

   - **示例**：
     - 假设我们有以下场景：
       1. 在某个拓扑域中，有两个节点：`nodeA` 和 `nodeB`。
       2. `nodeA` 上已经运行着一个名为 `pod1` 的 Pod，并且 `pod1` 设置了反亲和规则，要求相同类型的 Pod 尽量不要调度到同一个拓扑域内的节点上。
       3. 现在有一个新的 Pod（我们称其为 `pod2`），它的优先级高于 `pod1`，并且希望被调度到同一个拓扑域的 `nodeB` 上。
     - **结果**：最终，`pod2` 被调度到 `nodeB` 上运行，而 `pod1` 被抢占并终止。这满足了 `pod2` 的高优先级需求，同时也尽量遵守了反亲和性规则。

- **抢占过程特征**：

   - 异步执行，不保证立即成功
   - 可能触发多次抢占尝试
   - 需要满足PDB约束条件

- **特殊场景处理**：
   - 系统Pod（如kube-proxy）默认优先级较高（system-cluster-critical）
   - DaemonSet创建的Pod默认不可被抢占
