## 一、介绍
#### **1. 调度的定义**  
- **调度**：指在创建新Pod时，确定将其分配到哪个节点运行的过程。  
  - **执行者**：由`kube-scheduler`组件负责调度决策，`kubelet`负责在节点上实际创建Pod。  
#### **2. 何时会触发新Pod的创建？**  
- **用户主动请求**：  
   - 直接提交创建Pod的请求（如`kubectl apply -f pod.yaml`）。  
- **节点异常或资源不足**：  
   - 节点故障（如宕机）或资源不足时，Pod被驱逐（即被**删除**）并重新调度到其他节点。  
- **控制器调谐**：  
   - 控制器（如Deployment、StatefulSet）检测到副本数与预期不符时，触发新Pod的创建。  
#### **3. 静态Pod与非静态Pod的调度行为**  
| **类型**      | **调度行为**                                                 |
| ------------- | ------------------------------------------------------------ |
| **静态Pod**   | 不参与调度流程，直接由`kubelet`在指定节点上创建（通过节点配置文件定义）。 |
| **非静态Pod** | 必须经过调度流程，由`kube-scheduler`分配节点。               |
## 二、Pod调度流程

### **1. Pod调度的完整流程**  

**调度流程分为三个阶段：预选（Filter）、优选（Score）、节点选定（Bind）。**

#### **1.1 预选阶段（Filter）**  
- **目标**：筛选出符合Pod运行基本条件的节点。  
- **筛选规则**：  
  - **资源需求**：节点剩余资源（CPU、内存）是否满足Pod的申请值。  
  - **标签匹配**：NodeSelector、NodeAffinity（硬策略）。  
  - **污点与容忍**：检查节点污点是否被Pod容忍。  
  - **端口冲突**：节点是否已占用Pod声明的端口。  
  - **卷挂载检查**：节点是否支持Pod声明的存储卷类型。  
#### **1.2 优选阶段（Score）**  
- **目标**：对预选出的节点进行优先级排序，选择最优节点。  
- **评分策略**：  
  - **资源均衡**：优先选择资源利用率较低的节点（如`LeastRequested`策略）。  
  - **亲和性优化**：根据PodAffinity/NodeAffinity（软策略）权重打分。  
  - **拓扑分布**：避免同一拓扑域（如主机、机房）中密集部署Pod（反亲和性）。  
  - **自定义策略**：用户可扩展调度器插件实现自定义评分逻辑。  
#### **1.3 节点选定（Bind）**  
- **规则**：选择优先级最高的节点，若多个节点得分相同，则随机选择一个。  
- **结果**：调度器将Pod与节点绑定，通知对应节点的`kubelet`创建Pod。  
#### 1.4 总结

- 预选阶段确保节点满足硬性条件。  
- 优选阶段通过评分策略选择最优节点。  
- 节点绑定后不可逆，除非Pod被驱逐或删除。
### **2. 调度流程示意图**  
```plaintext
触发Pod创建
        ↓
调度器接收请求
        ↓
预选（Filter） → 筛选出符合基本条件的节点
        ↓
优选（Score） → 对节点打分并排序
        ↓
节点选定（Bind） → 绑定到最高分节点
        ↓
kubelet创建Pod
```
## 三、Pod调度规则
### **1. NodeSelector**

> 简单但功能有限，逐步被nodeAffinity替代

#### **1.1 介绍**  

- **功能**：将Pod调度到指定标签的物理节点，适用于有特殊硬件需求的场景（如GPU、SSD）。  
- **应用场景**：  
  - 将AI应用调度到带GPU的节点。  
  - 将MySQL调度到带SSD的节点。  

#### **1.2 实践**  

1. **节点打标签**：  
   ```bash
   kubectl label nodes <node-name> disk=ssd  # 添加标签
   kubectl label nodes <node-name> disk-     # 删除标签
   ```
2. **Pod配置NodeSelector**：  
   ```yaml
   apiVersion: v1
   kind: Pod
   metadata:
     name: test-pod
   spec:
     containers:
       - name: test-pod
         image: centos
     nodeSelector:
       disk: ssd  # 匹配节点标签
   ```

#### 1.3 **注意事项**

- 若多个节点有相同标签，调度器按算法选择。  
- 无匹配标签时，Pod状态为Pending。  
- 节点有预定义标签（如`kubernetes.io/hostname`）可直接使用。
### **2. 亲和性与反亲和性**
#### **2.1 介绍**  

- **亲和性**：Pod需调度到同一位置（如依赖服务）。  
- **反亲和性**：Pod需分散调度（如冗余容灾）。  
- **策略类型**：  
  - **硬策略（`requiredDuringSchedulingIgnoredDuringExecution`）**：必须满足，否则Pending。  
  - **软策略（`preferredDuringSchedulingIgnoredDuringExecution`）**：尽量满足，可设置权重。（实在无法满足的时候，可以调度到一个不匹配规则的节点上）
  - **`IgnoredDuringExecution`**：
    - 当 Pod 基于节点亲和性规则被调度到某个节点上后，如果节点的标签发生了改变，导致 Pod 不再符合该节点的亲和性要求，调度器不会将 Pod 从该节点上移除。
    - 因为 `IgnoredDuringExecution` 规则仅对**新建的** Pod 对象有效，而不是在 Pod 的整个生命周期内都有效。
#### **2.2 节点维度（nodeAffinity）**

> 筛选的是节点  

- **语法支持的操作符**：`In`、`NotIn`、`Gt`、`Lt`、`Exists`、`DoesNotExist`。  
- **示例配置**：  
  ```yaml
  # .spec.template.affinity
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:  # 硬策略
        nodeSelectorTerms:
          - matchExpressions:
              - key: node_type
                operator: In
                values: [gpu]
      preferredDuringSchedulingIgnoredDuringExecution:  # 软策略（权重）
        - weight: 1
          preference:
            matchExpressions:
              - key: com
                operator: In
  ```

- **注意事项**：  

  - 若同时定义`nodeSelector`和`nodeAffinity`，两者需同时满足。  

  - `nodeSelectorTerms`间为逻辑“或”，`matchExpressions`间为逻辑“与”。  
#### **2.3 Pod维度（podAffinity/podAntiAffinity）**  

> 筛选的是节点  

- **拓扑域（Topology Key）**：  
  - **定义**：通过节点标签划分逻辑区域（如地区zone、机房、机架、主机名【单独一台机器】）。  
  - **作用**：控制Pod在拓扑域内的分布（亲和或反亲和），实现Pod跨集群、跨机房、跨地区的调度。
  - **示例**：  
    
    ```yaml
    topologyKey: kubernetes.io/hostname  # 按主机名划分拓扑域（默认）
    topologyKey: zone                    # 按区域划分拓扑域：创建新pod的时候，会往通过标签选择器选中的pod所在节点所归属的拓扑域的所有节点上创建，不一定就是往该节点上调（以亲和性为例）
    		 # 比如该节点的标签为zone: beijing，则往带该标签的节点上调
    ```
  
- **Pod亲和性**：  
  
  ```yaml
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution: # 硬策略：如果有一个带标签app: nginx的pod在某个节点上，那么其他pod也都在这个节点（归属的拓扑域：主机名，即它本身）创建
        - labelSelector:
            matchExpressions:
              - key: app
                operator: In
                values: [nginx]
          topologyKey: kubernetes.io/hostname # 指定拓扑域
          #namespaces: 
            #- kube-system # 指定命名空间的话，则会去该命名空间下查找带有标签app=nginx的pod，与其亲和，不设置则默认从本pod当前的命名空间下找
      preferredDuringSchedulingIgnoredDuringExecution: # 软策略：尽量满足权重较大的规则，然后一直往满足权重较大的规则的pod所在节点（归属的拓扑域）创建pod直至因资源不足等原因无法创建时，再到满足其他权重的规则的pod所在节点创建pod
        - weight: 80
          podAffinityTerm:
            labelSelector:
              matchExpressions:
                - key: app
                  operator: In
                  values: [mysql]
            topologyKey: kubernetes.io/hostname
        - weight: 20
          podAffinityTerm:
            labelSelector:
              matchExpressions:
                - key: app
                  operator: In
                  values: [db]
            topologyKey: kubernetes.io/hostname
  ```
  
- **Pod反亲和性**：  
  
  ```yaml
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution: # 不会在带有标签app: nginx的pod所在的节点（归属的拓扑域）创建pod（远离）
        - labelSelector:
            matchExpressions:
              - key: app
                operatogginx]
          topologyKey: zone
      # 省略软策略示例，同亲和性
  ```
## **三、污点与容忍**

### 1. 污点（Taint）

- **定义**：节点标记，阻止**阻止未配置容忍的Pod调度到该节点**。

- **格式**：`<key>=<value>:<effect>`
- **应用场景**  

  - **Master节点隔离**：默认带污点`node-role.kubernetes.io/control-plane:NoSchedule`。  
  - **独占节点**：通过污点限制非指定Pod调度。 
- **污点策略**：

  | **策略类型**           | **作用**                                                     | **应用场景**                                      |
  | :--------------------- | :----------------------------------------------------------- | :------------------------------------------------ |
  | **`NoSchedule`**       | **强制阻止新Pod调度**到该节点，**已运行的Pod不受影响**。     | 节点维护、独占硬件资源（如GPU节点）。             |
  | **`PreferNoSchedule`** | **尽量不调度新Pod**到该节点，若无其他可用节点仍可能调度。    | 软性隔离，如节点负载较高但允许应急调度。          |
  | **`NoExecute`**        | **立即驱逐未容忍该污点的Pod**（包括已运行的Pod），并阻止新 Pod 调度。 | 节点故障或需强制驱逐（如节点下线、kubelet异常）。 |
- **污点操作**  

  - **添加污点**：  

    ```bash
    kubectl taint nodes <node-name> key=value:NoSchedule # =value可省略，默认空值
    ```

  - **删除污点**：  

    ```bash
    kubectl taint nodes <node-name> key:NoSchedule-
    kubectl taint nodes <node-name> key-
    ```

  - **查看污点**：

    ```bash
    kubectl describe nodes <node-name> | grep -i tiants # 只显示了一行
    ```


### 2. 容忍（Toleration）

- **定义**：定义在Pod上的配置，允许Pod**忽略节点的污点**并调度到该节点。 若节点带了多个污点，且 Pod 只容忍了其中一个，Pod 也还是会被阻止调度。

- **容忍配置**  

  - **关键配置项**：

    | **字段**                | **说明**                                                     |
    | :---------------------- | :----------------------------------------------------------- |
    | **`operator`**          | **匹配规则**： <br />- `Equal`（默认）：需严格匹配`key`、`value`和`effect`。 <br />- `Exists`：仅需匹配`key`和`effect`（`value`可省略）。 |
    | **`effect`**            | **匹配污点的策略类型**（可省略，表示匹配所有策略）。         |
    | **`tolerationSeconds`** | **仅对`NoExecute`生效**：定义Pod被驱逐前的等待时间（默认永久容忍，即没有为其赋值）。 |

  - ##### **特殊值规则**

    | **场景**               | **说明**                                                     |
    | :--------------------- | :----------------------------------------------------------- |
    | **空`key` + `Exists`** | 容忍所有污点（无论`key`是什么）。                            |
    | **空`effect`**         | 容忍所有策略类型（`NoSchedule`、`PreferNoSchedule`、`NoExecute`）。 |

  - **示例**：  

    ```yaml
    tolerations:
    # 容忍key为gpu、value为true的NoSchedule污点
    - key: "gpu"
      operator: "Equal"
      value: "true" # operator为Equal时需指定value
      effect: "NoSchedule"
    
    # 容忍所有key为ssd的污点（无论value和策略类型）
    - key: "ssd"
      operator: "Exists"
    
    # 容忍所有污点（危险！）
    - operator: "Exists"
    
    # 容忍NoExecute污点
    - key: "test"
      operator: "Exists"  # 或 "Equal"
      value: "k8s-node-02"  
      effect: "NoExecute"
      tolerationSeconds: 30  # 仅对NoExecute生效，30秒后驱逐
    ```
### 3. `tolerationSeconds`详解
#### **3.1 `NoExecute`污点的行为**
当节点被打上`NoExecute`污点时，对已运行Pod的影响分为三种情况：  
1. **无容忍配置的Pod**：立即被驱逐。  
2. **有容忍但未设置`tolerationSeconds`的Pod**：永久保留在节点上。  
3. **有容忍且设置`tolerationSeconds`的Pod**：在指定时间后被驱逐并重新调度。  

**核心规则**：  
- `tolerationSeconds`仅对`NoExecute`污点生效。  
- 若Pod的容忍中未指定`tolerationSeconds`，则默认永久容忍。  
#### **3.2 `tolerationSeconds`的作用**
1. **定义驱逐等待时间**：  
   - 当节点被标记为`NoExecute`时，Pod在`tolerationSeconds`指定的时间（秒）后被驱逐。  
   - **典型场景**：节点临时故障时，允许Pod等待节点恢复，避免频繁迁移。  

2. **避免死循环的关键**：  
   
   - **人为添加`NoExecute`污点**：若节点未被真正隔离，Pod驱逐后可能重新调度到原节点，导致循环驱逐（需结合节点状态处理）。  
   
     - **原因**：
   
       - ##### **节点仍处于健康状态**：
   
         - **手动添加`NoExecute`污点**仅标记节点的“排斥性”，但**不会改变节点的健康状态**。
         - 若节点`kubelet`正常运行，K8s仍认为节点是`Ready`状态，调度器将其视为候选节点。
   
       - #####  **Pod配置了容忍**：
   
         - **容忍配置允许Pod忽略污点**：
   
           ```yaml
           tolerations:
             - key: "test"
               operator: "Exists"
               effect: "NoExecute"
               tolerationSeconds: 30  # 30秒后驱逐
           ```
   
           - **容忍生效时**：Pod被驱逐后可能重新调度到原节点（若节点仍健康且资源充足）。
           - **容忍失效时**：Pod在`tolerationSeconds`后被驱逐，调度到其他节点。
   
     - **如何隔离节点**：如关闭 kubelet，k8s自动添加污点（3.3.1），节点状态变为`NotReady`或`Unreachable`，调度器排除该节点。即使 Pod 有容忍，也不会调度到不可用节点。
   
   - **节点实际故障**：节点不可用（如 kubelet 停止），Pod重新调度到健康节点，终止循环。  
#### **3.3 节点故障的自动处理**
1. **K8s自动添加的污点**：  
   
   - **`node.kubernetes.io/not-ready`**：节点未就绪（如kubelet停止）。  
   - **`node.kubernetes.io/unreachable`**：节点无法访问（如网络故障）。  
   - **默认容忍时间**：300秒（5分钟）。  
   
2. **修改默认容忍时间**：  
   ```yaml
   # 修改kube-apiserver启动参数（/etc/kubernetes/manifests/kube-apiserver.yaml）
   spec:
     containers:
       - command:
           - kube-apiserver
           - --default-not-ready-toleration-seconds=30    # 调整未就绪容忍时间
           - --default-unreachable-toleration-seconds=30  # 调整不可达容忍时间
   ```
#### **3.4 关键总结**
| **场景**         | **行为**                                                     | **配置建议**                                            |
| ---------------- | ------------------------------------------------------------ | ------------------------------------------------------- |
| **临时节点维护** | 手动添加`NoExecute`污点，设置短时间`tolerationSeconds`（如30秒）。 | 结合监控系统，确保节点恢复后移除污点。                  |
| **节点永久故障** | Kubernetes自动添加污点，Pod在`tolerationSeconds`后迁移。     | 保持默认300秒，避免频繁迁移；严重故障时可手动调低时间。 |
| **控制Pod分布**  | 使用`NoExecute`驱逐策略，强制Pod迁移到健康节点。             | 为关键Pod配置容忍，非关键Pod允许快速驱逐。              |
#### **3.5 注意事项**
1. **避免人为污点循环**：  
   - 手动添加`NoExecute`污点时，需确保节点真正不可用（如关闭kubelet），否则Pod可能循环调度。  

2. **慎用全局容忍**：  

   ```yaml
   tolerations:
     - operator: "Exists"  # 容忍所有污点（包括NoExecute）
   ```
   - 此配置可能导致Pod被调度到不期望的节点。  

3. **优先级覆盖**：  
   - 若Pod同时定义多个容忍，调度器按最长`tolerationSeconds`生效。  

4. **节点故障恢复时的Pod终止行为**：

   - 当节点完全挂掉（如断电、`kubelet`进程终止），运行在该节点上的Pod**不会立即被驱逐**，而是进入`Terminating`状态，因为是由 `kubelet` 负责删除 Pod 的，节点恢复后 Pod 才会真正被删除。
### **4. 综合练习（模拟DaemonSet）**  

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ds-demo
spec:
  replicas: 3
  selector:
    matchLabels:
      app: ds-demo
  template:
    metadata:
      labels:
        app: ds-demo
    spec:
      tolerations:
        - key: node-role.kubernetes.io/control-plane
          operator: Exists
          effect: NoSchedule
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            - labelSelector:
                matchExpressions:
                  - key: app
                    operator: In
                    values: [ds-demo]
              topologyKey: kubernetes.io/hostname
      containers:
        - name: nginx
          image: nginx:1.18
```
