## 一、k8s API

### 1. API-Server

`apiserver` 作为 API 的提供者，承担**三大核心功能**：

- **对组件的贡献**

  - **封装底层操作**：封装对 etcd 数据库的增删改查操作。
  - **提供通信机制**：通过 `List-Watch` 机制实现组件间高效通信。

- **对客户端的贡献**

  - **RESTful API 设计**：

    - 提供符合 REST 风格的 API 接口。

      - > **REST（Representational State Transfer，表现层状态转换）**
        >
        > - 一种软件架构风格，用于设计网络应用程序。主要基于HTTP协议，强调客户端-服务器之间的无状态通信，以及资源的标识和操作。
        > - **关键概念**：
        >   - **资源（Resource）**：
        >     - REST架构中的所有内容都被视为资源，例如用户、订单、产品等。
        >     - 每个资源都有一个唯一的URI（Uniform Resource Identifier，统一资源标识符）来标识它。
        >   - **表现层（Representation）**：
        >     - 资源可以有多种表现方式，如JSON、XML、HTML等。
        >     - 客户端和服务器通过这些表现层来交换资源的状态。
        >   - **状态转换（State Transfer）**：
        >     - 客户端通过发送请求来改变资源的状态。
        >     - 服务器通过响应来返回资源的新状态。
        >   - **无状态（Stateless）**：
        >     - 每个请求都必须包含所有必要的信息，服务器不需要保存关于客户端的任何状态。
        >     - 这有助于提高系统的可伸缩性和简化服务器设计。
        >   - **HTTP方法**：
        >     - REST使用HTTP协议的方法（如GET、POST、PUT、DELETE等）来执行不同的操作。
        >     - 例如，GET用于获取资源，POST用于创建资源，PUT用于更新资源，DELETE用于删除资源。
        >   - **统一接口（Uniform Interface）**：
        >     - REST架构强调统一的接口设计，使得不同的客户端和服务器能够以一致的方式交互。
        >     - 这包括资源的标识、资源的操作、自描述消息和超媒体作为应用状态的引擎（HATEOAS）。

    - 示例 URL：
      `https://<IP>:端口/api/apps/v1/namespaces/default/pods`

- **对安全的贡献**

  - **客户端访问安全链**：
    1. **凭证检查**：验证用户身份合法性（如 Token、证书）。
    2. **授权检验**：检查用户对请求 API 的操作权限（如 RBAC）。
    3. **准入控制器**：
       - **Mutating Webhook**：修改请求内容（如注入 Sidecar）。
       - **Validating Webhook**：校验请求合法性（如资源配额检查）。

### 2. API 设计面临的挑战

- **核心问题**：如何实现新老 API 版本的**兼容与演进**？
- **解决方案**：
  - **多版本共存**：同一资源可通过不同 API 路径访问不同版本。
    - 示例：
      - 实验版：`v1beta1/v1/deployment`
      - 稳定版：`apps/v1/deployment`
  - **灵活切换**：用户可选择稳定版（如 `apps/v1`）或实验版（如 `v1beta1`）。
  - **演进路径**：
    `alpha 版 → Beta 版 → 稳定版（如 v1）`

- **分组与多版本**：
  - **顶级路径**：
    - `/api`：核心组（如 Pod、Node）。
      - 核心组没有细分组名
      - 存放 k8s 核心资源
    - `/apis`：非核心组（如 Deployment、Job）。
    - `/metrics`：监控指标。
    - `/healthz`：健康检查。
  - **细分组名**：全局唯一（如 `apps`、`batch`）。
  - **版本控制**：同一资源可存在多个版本（如 `v1`、`v2alpha1`）。
  - **具体的资源**：如 `jobs`、`namespaces`。

- **路径结构（核心组 VS 非核心组）**

  - **核心组**：
    `/api/<版本>/namespaces/<命名空间>/<资源类型>/<资源名>`
    示例： `/api/v1/namespaces/default/pods/nginx`

  - **非核心组**：
    `/apis/<细分组>/<版本>/namespaces/<命名空间>/<资源类型>/<资源名>`
    示例： `/apis/apps/v1/namespaces/default/deployments/nginx`

  - **完整路径**：

    `https://<apiserver的ip:端口> + 以上`

- **查看**

  - 查看集群中的 API 组织形式：

    ```bash
    kubectl get --raw /apis/batch/ |python -m json.tool
    ```

### 4. API 版本管理

#### 版本级别与演进

| 版本级别 | 特点                           | 稳定性 | 使用场景           |
| :------- | :----------------------------- | :----- | :----------------- |
| Alpha    | 尝鲜特性，可能包含未测试功能   | 低     | 内部测试           |
| Beta     | 功能基本稳定，API 可能小幅调整 | 中     | 小范围生产环境试用 |
| Stable   | 功能稳定，长期支持（如 `v1`）  | 高     | 正式生产环境       |

#### 资源生命周期

1. **Alpha 阶段**：新功能首次引入，默认禁用。
2. **Beta 阶段**：功能优化，默认启用但需显式配置。
3. **Stable 阶段**：功能成熟，成为默认支持版本。
## 二、认证、授权与准入控制
### 1. 储备知识
#### 权限控制的三大要素

权限控制：限制用户对某个api下的某种类型的资源的权限

| 要素类型     | 描述                               | 示例                                 |
| ------------ | ---------------------------------- | ------------------------------------ |
| **API 组**   | 资源所属的 API 分组（`apiGroups`） | `apps`, `batch`, `core` (核心组为空) |
| **资源类型** | 具体的资源类型（`resources`）      | `pods`, `deployments`, `secrets`     |
| **操作权限** | 对资源的操作动作（`verbs`）        | `create`, `delete`, `get`, `exec`    |

##### 常见操作权限 (`verbs`)

- **写操作**：`create`, `delete`, `update`, `patch`
- **读操作**：`get`, `list`, `watch`
- **特殊操作**：`exec`（进入容器执行命令）
### 2. 用户实体 (`Subject`)
#### 账号分类
| 账号类型           | 用途                       | 凭证形式                                                     | 归属命名空间       |
| ------------------ | -------------------------- | ------------------------------------------------------------ | ------------------ |
| **用户账号**       | 供人类用户使用（如管理员） | 证书（X.509）、Token                                         | 无固定命名空间     |
| **ServiceAccount** | 供 Pod 内应用访问 K8s      | 自动挂载的 Token（`/var/run/secrets/kubernetes.io/serviceaccount`，该目录内含CA证书、命名空间和token） | 绑定到指定命名空间 |

- 每种账号都只是代号而已，必须关联上凭证才能用，k8s确认用户身份用的是凭证
### 3. RBAC（Role-Base Access Control，基于角色的访问控制）机制

**工作机制**：管理员为用户实体（用户、组、SA）分配一个或多个角色，每个角色代表了一组权限。用户实体通过角色获得相应的权限，从而实现对系统资源的访问和控制。

#### 3.1 角色类型
| 角色类型        | 可见范围     | 权限范围                 | 适用场景                   |
| --------------- | ------------ | ------------------------ | -------------------------- |
| **Role**        | 单个命名空间 | 仅限当前命名空间内资源   | 命名空间内的细粒度控制     |
| **ClusterRole** | 集群全局可见 | 可跨命名空间或集群级资源 | 集群级资源或跨命名空间权限 |

#### 3.2 角色绑定方式
| 绑定方式               | 绑定对象            | 权限生效范围             | 典型场景                                                     |
| ---------------------- | ------------------- | ------------------------ | ------------------------------------------------------------ |
| **RoleBinding**        | Role 或 ClusterRole | 限制为单个命名空间       | 通过ClusterRole定义通用权限，避免重复定义，并通过RoleBinding将权限分配到特定命名空间中分别管理 |
| **ClusterRoleBinding** | ClusterRole         | 全局生效（所有命名空间） | 授予集群管理员权限（如 `cluster-admin`）                     |

  - **上下文（Context）**：用于管理 `kubectl`（K8s 命令行工具）的操作环境。将**集群（Cluster）、用户（User）和命名空间（Namespace）**三者绑定在一起，形成一个可快速切换的操作环境配置。

    - **为什么需要上下文？**

      - **场景示例**：

        假设你需要管理以下环境：

        - **开发环境**：集群 `dev-cluster`，用户 `dev-user`，命名空间 `dev-ns`。
        - **生产环境**：集群 `prod-cluster`，用户 `prod-admin`，命名空间 `prod-ns`。

        如果每次操作都手动指定集群、用户和命名空间，命令会非常冗长：

        ```bash
        kubectl get pods --cluster=dev-cluster --user=dev-user --namespace=dev-ns
        kubectl get pods --cluster=prod-cluster --user=prod-admin --namespace=prod-ns
        ```

        通过定义上下文，可以简化为：

        ```bash
        kubectl config use-context dev-context   # 切换到开发环境
        kubectl get pods                         # 自动操作 dev-cluster/dev-ns
        
        kubectl config use-context prod-context  # 切换到生产环境
        kubectl get pods                         # 自动操作 prod-cluster/prod-ns
        ```

    - **核心作用**

      | 作用                      | 说明                                                         |
      | :------------------------ | :----------------------------------------------------------- |
      | **简化命令输入**          | 自动填充集群、用户和命名空间参数，避免重复输入。             |
      | **隔离操作环境**          | 防止误操作（如将开发环境的命令误执行在生产环境）。           |
      | **支持多集群/多角色管理** | 快速切换不同的集群、用户或命名空间（如开发、测试、生产环境）。 |
      | **权限控制**              | 结合 RBAC，确保用户只能在指定命名空间内操作资源。            |

    - **上下文的配置文件**

      所有上下文信息存储在 `~/.kube/config` 文件中。文件结构示例：

      ```yaml
      apiVersion: v1
      clusters:
      - name: kubernetes
        cluster: { server: https://192.168.1.100:6443, certificate-authority-data: ... }
      users:
        user: { client-certificate-data: ..., client-key-data: ... }
      contexts:
        context:
          cluster: kubernetes
          namespace: kube-system  # 默认命名空间
      ```

    - **上下文检索的优先级规则**

      `kubectl` 在决定使用哪个上下文时，按以下顺序从高到低检查配置来源：

      | 优先级 | 配置来源                          | 说明                                                         |
      | :----- | :-------------------------------- | :----------------------------------------------------------- |
      | **1**  | **命令行显式指定**                | 通过 `--context` 参数直接指定上下文。                        |
      | **2**  | **当前上下文（current-context）** | 在 `kubeconfig` 文件中标记为 `current-context` 的上下文。    |
      | **3**  | **环境变量 `KUBECONFIG`**         | 若通过 `KUBECONFIG` 环境变量指定多个配置文件，合并后按加载顺序覆盖。 |
      | **4**  | **默认配置文件**                  | 默认加载 `~/.kube/config`，若未显式指定其他配置。            |

    - **常用操作**：

      **（1）查看当前上下文**

      ```bash
      kubectl config current-context
      ```

      ```bash

      # CURRENT   NAME           CLUSTER        AUTHINFO   NAMESPACE
      ```

      ```bash
      kubectl config view
      ```

      ```bash
      # 删除单个上下文
      # 1. 命令行
      kubectl config delete-context <context-name>
      # 2. 手动编辑 kubeconfig 文件

      vim ~/.kube/config

      unset KUBECONFIG  # 取消环境变量设置

      # 删除用户
      kubectl config unset users.<user-name>
      # 删除集群
      kubectl config unset clusters.<cluster-name>
      
      # 验证
      kubectl config get-contexts
      kubectl config get-users
      kubectl config get-clusters
      ```

###### **（2）创建 Role**

- **目标**：定义一个 Role `role1`，授予其在 `kube-system` 命名空间内对 `pods`、`deployments`、`replicasets` 的完整操作权限。

- **步骤详解**

  - **生成 Role 模板**  

     使用 `kubectl create role` 生成 YAML 模板：

     ```bash
     kubectl create role role1 \
       --verb=get,list,watch,create,update,patch,delete \
       --resource=pods,deployments,replicasets \
       --namespace=kube-system \
       --dry-run=client -o yaml > role-demo.yaml
     ```

  - **编辑 Role YAML**  

     最终 `role-demo.yaml` 内容：

     ```yaml
     apiVersion: rbac.authorization.k8s.io/v1
     kind: Role
     metadata:
       name: role1
       namespace: kube-system
     rules:
     - apiGroups: ["", "apps"]  # "" 表示核心组(pods)，apps 对应 Deployment/ReplicaSet
       resources: ["pods", "deployments", "replicasets"]
       verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
     ```

  - **应用 Role**  

     创建 Role：

     ```bash
     kubectl apply -f role-demo.yaml
     ```

     role1只能在 `kube-system` 被查看到

     ```bash
     kubectl create rolebinding rolebinding1 \
       --role=role1 \
       --namespace=kube-system \

     # 若相对一组用户进行权限分配，可将这一组用户归并到一个组内，对group进行访问权限的分配
     ```

     ```yaml
     - kind: User

       kind: Role
       name: role1
       apiGroup: rbac.authorization.k8s.io
     ```

     创建 RoleBinding：

     ```bash

  # 返回 403 Forbidden
  # kubernetes.default.svc：K8s 集群中的一个 DNS 名称，用于访问默认命名空间中的 Kubernetes API 服务
  ```
  
- **访问 `kube-system` 命名空间的 Pod（有权限）**  
  ```bash
  curl --header "Authorization: Bearer $TOKEN" --insecure \
    https://kubernetes.default.svc/api/v1/namespaces/kube-system/pods
  # 返回 Pod 列表
  ```

- **访问 `kube-system` 的 Deployments（有权限）**  
  ```bash
  curl --header "Authorization: Bearer $TOKEN" --insecure \
    https://kubernetes.default.svc/apis/apps/v1/namespaces/kube-system/deployments
  # 返回 Deployments 列表
  ```
### 4. API Server 验证流程

#### 4.1 认证（Authentication）
- **目的**：验证用户身份合法性。
- **支持的凭证类型**：
  - X.509 证书
  - Bearer Token
  - ServiceAccount Token
  - 静态密码文件（仅测试环境）

#### 4.2 授权（Authorization）
- **目的**：验证用户是否有权执行操作。
- **RBAC 校验逻辑**：
  1. 检查请求的 `apiGroup`、`resource`、`verb`。
  2. 匹配用户关联的 Role/ClusterRole 权限规则。

#### 4.3 准入控制（Admission Controllers）
- **目的**：在请求持久化前拦截并修改/验证请求。
- **两类 Webhook**：
  
  | Webhook 类型   | 作用           | 典型用例                            |
  | -------------- | -------------- | ----------------------------------- |
  | **Mutating**   | 修改请求内容   | 注入 Sidecar 容器、设置默认资源配额 |
  | **Validating** | 验证请求合法性 | 检查资源配额、Pod 安全策略（PSP）   |
### 5. 核心总结
- **认证**：确认用户身份（你是谁）。
- **授权**：控制用户能做什么（你能做什么）。
- **准入控制**：确保操作合规（是否符合规则）。
- **RBAC 核心**：通过角色（Role/ClusterRole）和绑定（RoleBinding/ClusterRoleBinding）实现细粒度权限管理。

## 三、Security Context 安全上下文

### 1. Linux Capability
#### **1.1 诞生背景**
- **传统权限模型的问题**  
   - Linux 内核 2.2 之前，进程权限仅有 **root** 和 **非 root** 两种划分：
     - root 用户拥有全部特权。
     - 非 root 用户无法执行任何特权操作。
   - **SUID（Set User ID）** 机制的缺陷：
     - 例如：普通用户需通过 `/bin/passwd` 修改 `/etc/shadow` 文件，需为其设置 SUID，使其以 root 身份运行。
     - 问题：SUID 赋予进程完整的 root 权限，导致权限过度开放，存在安全隐患。

- **Capability 的诞生目的**  
   - 将 root 用户的特权细化为多个独立的能力（Capabilities）。
   - 进程可按需获取最小化特权，而非完整的 root 权限。
#### **1.2 Linux Capability 核心概念**
- **定义**  
   - 对 root 特权的细化拆分，每个 Capability 代表一项特定的特权操作。
   - 进程可被单独授予某些 Capability，而非全部 root 权限。

- **Capabilities 列表**  
   - 官方文档：[Capabilities 手册](https://man7.org/linux/man-pages/man7/capabilities.7.html)
   - 常见示例：
     - `CAP_NET_RAW`：允许使用原始套接字（如 ping 命令依赖此权限）。
     - `CAP_NET_ADMIN`：允许执行网络管理操作（如修改接口、路由表）。
     - `CAP_SYS_ADMIN`：近似 root 权限，可执行系统管理操作。
#### **1.3 Capability 设置方法**
##### **（1） 宿主机上的操作**
- **查看 Capability**  
  
  ```bash
  getcap /bin/ping   # 查看 /bin/ping 的 Capabilities
  # 输出示例：/bin/ping = cap_net_admin,cap_net_raw+p
  ```
  
- **添加/删除 Capability**  
  
  ```bash
  # 移除 Capability（禁止普通用户执行 ping）
  setcap cap_net_admin,cap_net_raw-p /bin/ping
  
  # 恢复 Capability（允许普通用户执行 ping）
  setcap cap_net_admin,cap_net_raw+p /bin/ping
  ```
  - **`+p` 与 `-p`**：表示修改 Permitted 集合（进程实际可用的权限集合）。

##### **（2） 容器内的操作**
- **容器权限的本质**  
  
  - 容器是宿主机上的进程，共享宿主机内核。
  - 容器内 root 用户仅拥有宿主机的部分特权，非完整 root 权限。
  
- **权限配置选项**  
  - **`--privileged`**：赋予容器完整的宿主机 root 权限（包括所有 Capabilities 和设备访问权限）。  
    
    ```bash
    docker run --privileged -it ubuntu /bin/bash
    ```
  - **`--cap-add` 与 `--cap-drop`**：细粒度控制 Capabilities。  
    
    ```bash
    # 添加单个 Capability（如允许网络管理）
    docker run -it --rm --cap-add NET_ADMIN busybox /bin/sh
    
    # 移除单个 Capability（如禁用 RAW 套接字）
    docker run -it --cap-add=ALL --cap-drop=NET_RAW ubuntu
    
    # 示例：在容器内创建虚拟网络接口
    ip link add temp_dummy0 type dummy  # 需 CAP_NET_ADMIN
    ```
#### **1.4 注意事项**
- **权限最小化原则**  
   - 优先使用 `--cap-add` 而非 `--privileged`，避免过度授权。
   - 通过 `--cap-drop=ALL` 移除所有默认 Capabilities，再按需添加。

- **Capability 集合类型**  
   - **Permitted**：进程可使用的 Capabilities。
   - **Effective**：当前实际生效的 Capabilities。
   - **Inheritable**：子进程可继承的 Capabilities。

- **安全风险**  
   - 错误配置 Capabilities（如 `CAP_SYS_ADMIN`）可能导致容器逃逸或系统漏洞。
### 2. 介绍

- **定义**  

   Security Context（安全上下文）是 K8s 中用于定义 **Pod 或容器安全设置**的配置，通过限制进程权限提升容器运行安全性。

- **作用背景**  
   - 容器本质是宿主机上的进程，需控制其权限以防止越权操作。  
   - 示例场景：容器需修改宿主机内核参数（如 `sysctl` 命令操作 `net`、`vm` 等），需通过 Security Context 赋予部分特权。

- **核心控制权限**  

   | 权限类型                                     | 说明                                                         |
   | -------------------------------------------- | ------------------------------------------------------------ |
   | **用户与组 ID**                              | 设置容器进程的 `runAsUser`（用户 ID）和 `runAsGroup`（组 ID）。 |
   | **特权模式**                                 | 启用 `privileged` 后，容器内 root 用户拥有宿主机 root 的绝大部分权限。 |
   | **访问控制**                                 | 包含以下子项：                                               |
   | - Linux Capabilities                         | 授予特定特权（如 `CAP_NET_ADMIN`）而非完整 root 权限。       |
   | - AppArmor                                   | 通过配置文件限制程序的权限。                                 |
   | - SELinux                                    | 为对象分配安全标签实现强制访问控制。                         |
   | - Seccomp                                    | 过滤容器进程的系统调用（如限制危险系统调用）。               |
   | **AllowPrivilegeEscalation**（允许特权扩大） | 布尔值，控制进程是否可以比其父进程获取更多特权。若以下任一为 `true`，此字段自动为 `true`：<br> - 容器以 `privileged` 模式运行<br> - 容器拥有 `CAP_SYS_ADMIN` 权限。 |

- **配置层级**  
   - **容器级**：仅作用于指定容器。  
   - **Pod 级**：作用于 Pod 内所有容器及 Volume。  
   - **Pod Security Policies (PSP，已废弃)**：原用于集群全局策略，建议使用替代方案（如 OPA Gatekeeper）。  
### **3. Pod 级 Security Context 配置**
#### **3.1 示例配置**
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: security-context-pod-demo
spec:
  securityContext:   # Pod 级配置
    runAsUser: 1000  # 所有容器以 UID 1000 运行
    runAsGroup: 3000 # 所有容器以 GID 3000 运行（默认 GID 为 0，即 root 组）
    fsGroup: 2000    # Volume 文件所属组 ID 为 2000
  volumes:
    - name: my-volume
      emptyDir: {}
  containers:
    - name: sec-ctx-demo
      image: busybox
      command: ['sh', '-c', 'sleep 60m']
      volumeMounts:
        - name: my-volume
          mountPath: /pod/demo  # 该路径的 GID 及子文件 GID 为 2000
      securityContext:  # 容器级配置（覆盖 Pod 级）
        allowPrivilegeEscalation: false  # 禁止特权提升
```

#### **3.2 验证结果**
- **进程权限**  
  
  ```bash
  $ kubectl exec -ti security-context-pod-demo sh
  / $ ps aux
  PID   USER     TIME  COMMAND
  1     1000     0:00  sh -c sleep 60m
  16    1000     0:00  sh
  ```
  所有进程以 UID 1000 和 GID 3000 运行。
  
- **文件系统权限**  
  ```bash
  $ ls -dl /
  drwxr-xr-x 1 root root 4096 May 22 12:37 /tmp  # 文件所有者有读、写、执行权限，同组用户和其他用户有读和执行权限，但没有写权限，所有在该目录下touch文件会失败
  
  $ ls -dl /tmp
  drwxrwxrwt 1 root root 4096 May 22 12:37 /tmp
  $ touch /tmp/1.txt  # 成功创建文件
  $ ls -l /tmp/1.txt
  -rw-r--r-- 1 1000 3000 May 22 12:37 /tmp/1.txt  # 属主为 1000:3000
  
  / $ touch /pod/demo/2.txt 
  / $ ls -l /pod/demo/2.txt
  -rw-r--r-- 1 1000 2000 0 May 22 10:33 /pod/demo/2.txt
  # 在挂载目录下创建的文件组要继承自目录
  ```
  - `fsGroup` 确保挂载卷的组权限为 2000。  
  - `/tmp` 目录因父目录权限不足无法操作。
  
- **用户身份**  
  ```bash
  / $ id
  uid=1000 gid=3000 groups=2000,3000
  ```
### **4. 容器级 Security Context 配置**
#### **4.1 示例配置**
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: security-context-container-demo
spec:
  securityContext:  # Pod 级配置（被容器级覆盖）
    runAsUser: 1000
  containers:
    - name: demo1
      image: busybox
      command: ['sh', '-c', 'sleep 60m']
      securityContext:  # 容器级配置
        runAsUser: 2000  # 覆盖 Pod 级 runAsUser
        allowPrivilegeEscalation: false
    - name: demo2
      image: busybox
      command: ['sh', '-c', 'sleep 60m']
```

#### **4.2 验证结果**
- **容器 1（demo1）**  
  
  ```bash
  $ kubectl exec -ti security-context-container-demo -c demo1 sh
  / $ id
  uid=2000 gid=3000  # runAsUser 被覆盖为 2000
  / $ touch /tmp/1.txt  # 文件属主为 2000:root
  ```
  
- **容器 2（demo2）**  
  ```bash
  $ kubectl exec -ti security-context-container-demo -c demo2 sh
  / $ id
  uid=1000 gid=3000  # 沿用 Pod 级 runAsUser
  / $ touch /tmp/1.txt  # 文件属主为 1000:root
  ```
### **5. 特权模式配置**
#### **5.1 启用方法**
- **仅在容器级设置**  
  ```yaml
  apiVersion: v1
  kind: Pod
  metadata:
    name: demo3
  spec:
    containers:
      - name: demo3
        image: busybox
        command: ['sh', '-c', 'sleep 60m']
        securityContext:
          privileged: true  # 启用特权模式
  ```

#### **5.2 验证结果**
- **未启用特权模式**  
  ```bash
  / # echo 0 > /proc/sys/net/ipv4/ip_forward
  sh: can't create /proc/sys/net/ipv4/ip_forward: Read-only file system
  ```

- **启用特权模式后**  
  ```bash
  / # echo 0 > /proc/sys/net/ipv4/ip_forward  # 修改成功（仅容器内生效）
  ```
  - **注意**：容器内修改的 `/proc/sys` 参数仅作用于容器自身，不影响宿主机。
### **6. 配置 Linux Capabilities**
#### **6.1 示例配置**
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: cpb-demo
spec:
  containers:
    - name: cpb
      image: busybox
      args: ["sleep", "3600"]
      securityContext:
        capabilities:
          add: ["NET_ADMIN"]  # 添加网络管理权限
          drop: ["KILL"]      # 移除默认的 KILL 权限
```

#### **6.2 验证结果**
```bash
$ kubectl exec -it cpb-demo /bin/sh
/ # ip link add temp-dummy0 type dummy  # 成功（依赖 CAP_NET_ADMIN）
```
#### **六、安全最佳实践**
- **最小权限原则**  
   - 优先使用 `capabilities.add` 而非 `privileged`，避免过度授权。
   - 通过 `capabilities.drop` 移除不必要的默认权限。

- **避免特权模式**  
   - `privileged: true` 赋予容器完整 root 权限，增加安全风险。

- **卷权限管理**  
   - 使用 `fsGroup` 控制 Volume 的组权限，避免敏感文件泄露。

- **特权升级限制**  
   - 设置 `allowPrivilegeEscalation: false` 防止进程获取额外权限。
