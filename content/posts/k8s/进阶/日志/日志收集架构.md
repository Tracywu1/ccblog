## 一、日志系统介绍

### 1. 什么是日志系统
- **定义**：  
  
  负责将软件及系统的运行日志统一采集到集中平台的系统
  
- **核心价值**：
  - 提供统一的日志观测与分析平台
  - 通过集中化日志快速定位问题根因

### 2. 为何引入日志系统
1. **报警系统的局限性**
   - 报警仅能反映异常现象
   - 根因分析必须依赖具体运行日志

2. **运维效率问题**
   - 服务数量膨胀时，逐个节点查看日志效率低下
   - 需要统一的日志检索/查看入口

3. **K8s环境特殊需求**
   - Pod日志存在生命周期风险：
     
     未关联持久化存储的Pod被删除时，日志数据会随容器销毁永久丢失（无论日志是输出到哪里）

### 3. 储备知识：容器日志处理方式

**场景一：日志输出到标准输出**

1. **日志存储特性**
   - 宿主机存储路径：  
     
     `/var/log/pods/<namespace>_<pod-name>_<pod-uid>/`
     
   - 软链接路径：  
     
     `/var/log/containers/<pod-name>_<namespace>_<container-name>-<container-id>.log`
   
2. **日志查看方式**
   ```shell
   kubectl logs -f <pod-name>
   ```

**场景二：日志写入容器内文件**

1. **日志存储特性**
   - 日志文件路径不统一（取决于应用配置，但 `emptyDir` 映射到宿主机的路径是有规律的）
   - 无标准宿主机映射路径

2. **日志管理痛点**
   - 无法通过 `kubectl` 直接查看，需进入容器查看
   - 需要自定义日志采集策略（非标准路径 + 宿主机路径动态变化 → 无法自动发现 → 需人工逐个配置 → 运维负担重）
   - 必须配置持久化存储防止数据丢失
## 二、K8s 环境日志收集方案

### 方案1（最常见）：节点级日志代理 (Node-Level Agent)

<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/1120683-20191012142558461-1776204980.png" alt="img" style="zoom: 67%;" />

[^logrotate]: k8s环境自带日志轮转功能

#### 实现原理
- **前提条件**  
  
  Pod 内应用需将日志输出到标准输出(stdout/stderr)  
  
- **核心机制**  
  - 每个节点部署DaemonSet管理的日志代理 logging-agent-pod（如Fluentd/Filebeat）  
  - 在 logging-agent-pod 上，通过hostPath挂载宿主机日志目录：  
    `/var/log/containers` → 容器内的日志采集路径  
  - logging-agent-pod 持续**读取**日志文件并转发至日志后端（如ELK/EFK）

#### 部署方式
```yaml
# 示例DaemonSet片段
volumes:
  - name: varlog
    hostPath:
      path: /var/log/containers
volumeMounts:
  - name: varlog
    mountPath: /var/log/containers	# 容器内的挂载路径
```

#### 优缺点对比
| 优势                                       | 局限性                                                       |
| ------------------------------------------ | ------------------------------------------------------------ |
| ✅ 简单易部署<br>✅ 无侵入性<br>✅ 资源消耗低 | ❌ 仅支持标准输出日志<br>❌ 无法处理容器内文件日志<br>❌ 宿主机存储依赖 |

[^无侵入性]: 只需要在每个节点上运行一个代理程序，并不需要对节点上运行的应用程序进行更改，对应用程序没有任何侵入性。
### 方案2：Sidecar 容器处理日志

<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/1120683-20191012142620391-1592777068.png" alt="img" style="zoom: 67%;" />

#### 基础版：日志重定向
**实现原理**  
1. 在Pod内添加Sidecar容器  
2. Sidecar容器与业务容器共享存储卷（emptyDir），并读取业务容器日志文件  
3. 将日志内容输出到标准输出

**部署架构**  
- 日志采集Agent：DaemonSet + hostPath（同方案1）  
- 业务Pod改造：注入Sidecar容器（也可以使用webhook完成这一操作）
- 日志轮转：可以手动添加这一功能

**示例配置**  

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: counter
spec:
  containers:
    - name: count
    image: busybox
    args:
      - /bin/sh
      - -c
      - >
        i=0;
        while true;
        do
          echo "$i: $(date)" >> /var/log/1.log;
          echo "$(date) INFO $i" >> /var/log/2.log;
          i=$((i+1));
          sleep 1;
        done
    # 业务容器配置
    volumeMounts:
      - name: varlog
        mountPath: /var/log
    
  - name: count-log-1
    image: busybox
    # 读取业务容器日志文件，将日志内容输出到标准输出
    args: [/bin/sh, -c, 'tail -n+1 -f /var/log/1.log']
    # Sidecar容器与业务容器共享存储卷
    volumeMounts:
      - name: varlog
        mountPath: /var/log
        
  - name: count-log-2
    image: busybox
    args: [/bin/sh, -c, 'tail -n+1 -f /var/log/2.log']
    volumeMounts:
      - name: varlog
        mountPath: /var/log

volumes:
  - name: varlog
    emptyDir: {}
```

#### 优缺点分析
| 优势                 | 缺陷                                              |
| -------------------- | ------------------------------------------------- |
| ✅ 支持容器内文件日志 | ❌ 双倍存储消耗<br>❌ 需要Pod改造<br>❌ 日志延迟增加 |
#### 升级版：集成日志Agent

<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/1120683-20191012142723404-1955968291.png" alt="img" style="zoom: 67%;" />

**实现原理**  

- 在Pod内直接运行日志采集Agent作为Sidecar  
- 跳过标准输出，直接将日志发送至后端

**配置示例**  
```yaml
# Pod配置片段
containers:
  - name: app
    image: myapp
    volumeMounts:
      - name: logs
        mountPath: /var/log

  - name: fluentd-sidecar
    image: fluent/fluentd:v1.14
    volumeMounts:
      - name: logs
        mountPath: /var/log
      - name: fluentd-config
        mountPath: /etc/fluentd

volumes:
  - name: logs
    emptyDir: {}
  - name: fluentd-config # 后端的路径
    configMap:
      name: fluentd-config
```

**配套ConfigMap**  

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: fluentd-config
data:
  # 定义收集容器内的/var/log/1.log 和 /var/log/2.log 的日志数据，然后通过 google_cloud 插件将数据推送到 Stackdriver 日志后端去。
  fluentd.conf: |
    <source>
	 type tail
	 format none
	 path /var/log/1.log
	 pos_file /var/log/1.log.pos
	 tag count.format1
	</source>

	<source>
	 type tail
	 format none
	 path /var/log/2.log
	 pos_file /var/log/2.log.pos
	 tag count.format2
	</source>

	<match **>
	 type google_cloud
	</match>
```

也可以使用其他的日志采集工具来替换 fluentd，比如 logstash、fluent-bit 等等。

#### 升级版特性
| 优势                                 | 注意事项                                                     |
| ------------------------------------ | ------------------------------------------------------------ |
| ✅ 避免双倍存储<br>✅ 灵活定制采集逻辑 | ❌ 无法使用kubectl logs<br>❌ 配置复杂度增加<br>❌ 需维护不同Agent配置 |
### 方案3：应用直接推送日志

<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/f5a940d15ba1832d8b1a586dfbf8fc5638446735.png" alt="img" style="zoom:67%;" />

#### 实现原理
- 在应用程序代码中集成日志SDK（实现起来最不切实际）  
- 直接通过API将日志发送到日志服务（如Logstash/Kafka）

#### 典型架构
```
[Application] → (SDK) → [Log Backend]
```

#### 优缺点对比
| 优势                                         | 挑战                                                         |
| -------------------------------------------- | ------------------------------------------------------------ |
| ✅ 架构最精简<br>✅ 零存储开销<br>✅ 实时性最佳 | ❌ 需代码改造（侵入性最高）<br>❌ 语言/框架依赖性<br>❌ 维护成本高 |
### **选型建议**  
1. 优先使用方案1处理标准输出日志 
2. 对容器内文件日志采用方案2基础版 （作为其向云原生转型的过渡方案，方案1是最简单的也是最推荐的，但需要服务云原生特性）
3. 仅在新开发项目且团队能力足够时考虑方案3
