### 一、节点分类与高可用方案
#### 1. **节点类型**  

- **Master Node**：运行控制平面组件（`kube-apiserver`、`kube-scheduler`、`kube-controller-manager`、`etcd`）。
- **Worker Node**：运行业务容器，通过 VIP 访问控制平面。

#### 2. **高可用方案**  

- **架构**：负载均衡（如 HAProxy/Nginx） + Keepalived 代理多个 Master，对外提供 VIP。
- **目标**：Worker Node 通过 VIP 访问控制平面，实现 Master 节点故障时的无缝切换。
### 二、两种 etcd 部署方案对比
#### 1. **Stacked Control Plane Nodes（内部 etcd 服务）**

<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/image-20250507143413471.png" alt="image-20250507143413471" style="zoom: 33%;" />

- **架构特点**：

  - etcd 服务部署在 Master 节点上，每个 Master 节点运行一个 etcd 成员。
  - 本地的 `kube-apiserver` 仅与本地 `etcd` 通信。
  - 控制平面组件（`kube-scheduler`、`kube-controller-manager`）仅与本地 `kube-apiserver` 通信。

- **访问配置**：

  ```yaml
  - --etcd-servers=https://127.0.0.1:2379  # 本地访问
  ```

- **优点**：

  - **节省资源**：无需额外主机部署 etcd。
  - **部署简单**：通过 `kubeadm` 统一管理。

- **缺点**：
  - **耦合风险**：Master 节点故障会导致该节点的 etcd 和控制平面组件同时不可用，需多节点（奇数）冗余。

#### 2. **External etcd Cluster（外部 etcd 服务）**

<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/image-20250507143440005.png" alt="image-20250507143440005" style="zoom:33%;" />

- **架构特点**：
  - etcd 集群独立于 Master 节点，运行在专用主机上。
  - 所有 Master 节点的 `kube-apiserver` 均与外部 etcd 集群通信。

- **访问配置**：

  ```yaml
  - --etcd-servers=https://<etcd1-IP>:2379,https://<etcd2-IP>:2379,https://<etcd3-IP>:2379
  ```

- **优点**：
  - **故障隔离**：etcd 与 Master 节点分离，降低单点故障影响。

- **缺点**：
  - **资源消耗**：需至少 3 台独立主机部署 etcd，成本较高。
  - **部署复杂度**：需独立维护 etcd 集群。
### 三、部署命令示例

具体部署步骤参考pdf

##### 1. **加入 Master 节点**
```bash
kubeadm join api-server:8443 --token abcdef.0123456789abcdef \
  --discovery-token-ca-cert-hash sha256:e53ab548e5582cf86d3646cbfed824b28631906a0639265b123f285d66659e5d \
  --control-plane \
  --certificate-key 76efa192afc6bfb83aafddb4d2959aee2794e711567c0f4cb767a27f3df73eff
```
- **关键参数**：
  - `--control-plane`：标识加入的节点为 Master。
  - `--certificate-key`：用于加密控制平面证书的密钥。

##### 2. **加入 Worker 节点**
```bash
kubeadm join api-server:8443 --token abcdef.0123456789abcdef \
  --discovery-token-ca-cert-hash sha256:e53ab548e5582cf86d3646cbfed824b28631906a0639265b123f285d66659e5d
```
### 四、etcd 数据操作
#### 1. **etcdctl 工具下载**
- 官方地址：[https://github.com/etcd-io/etcd/releases](https://github.com/etcd-io/etcd/releases)

#### 2. **查询 etcd 集群成员列表**
- **外部 etcd 集群**：
  ```bash
  ETCDCTL_API=3 etcdctl \
    --endpoints=https://192.168.71.103:2379,https://192.168.71.101:2379,https://192.168.71.102:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key \
    member list
  ```
- **内部 etcd 服务**（单节点）：
  ```bash
  ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key \
    member list
  ```

#### 3. **查询 etcd 数据**
- **查看所有键**：
  ```bash
  ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key \
    get / --prefix --keys-only
  ```
- **查看具体资源**（如 Deployment）：
  
  ```bash
  ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key \
    get /registry/deployments/default/mynginx
  # 查看某key的值（文件的内容）
  ```

#### 4. **数据存储格式**
- etcd 中存储的数据为 **Protocol Buffer 序列化格式**，非原始 JSON，以提高存储效率和减少空间占用。
