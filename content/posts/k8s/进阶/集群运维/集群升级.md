#### **一、升级背景**
- Kubernetes **无官方 LTS（长期支持）版本**，需谨慎规划升级路径，避免因版本差异导致集群不稳定。
#### **二、核心注意事项**

##### **1. 备份关键数据**
- **备份 etcd 数据**（*必须操作*）：
  
  ```bash
  # 创建 etcd 快照（在任意 Master 节点执行）
  ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key \
    snapshot save etcdbackupfile.db
  ```
  
- **恢复 etcd 数据**（故障时使用）：
  ```bash
  # 停止 etcd 服务并清理数据目录
  mv /var/lib/etcd /var/lib/etcd_bak
  mkdir /var/lib/etcd
  
  # 从快照恢复数据
  ETCDCTL_API=3 etcdctl \
    --endpoints=https://127.0.0.1:2379 \
    --cacert=/etc/kubernetes/pki/etcd/ca.crt \
    --cert=/etc/kubernetes/pki/etcd/server.crt \
    --key=/etc/kubernetes/pki/etcd/server.key \
    snapshot restore etcdbackupfile.db --data-dir=/var/lib/etcd
  
  # 重启 etcd 服务
  systemctl restart kubelet
  ```

##### **2. 版本升级限制**
- **禁止跨两个大版本升级**（如 `1.19.x` → `1.21.x`），需逐版本升级：
  - **安全路径示例**：
    ```plaintext
    1.19.4 → 1.20.4 → 1.21.4
    ```
  - **高风险操作**：
    ```plaintext
    1.19.4 → 1.21.4  # 跳过中间版本，可能导致 API 不兼容或未修复的 Bug
    ```

##### **3. 测试环境演练与回滚方案**
- **必选步骤**：在测试环境模拟升级流程，验证业务兼容性。
- **回滚脚本**：预先编写回滚操作脚本，确保生产环境故障时快速恢复。
#### **三、升级策略**

##### **1. 蓝绿部署（Blue-Green）**
- **适用场景**：资源充足，需最小化业务中断。
- **操作步骤**：
  1. 部署新版本 Kubernetes 集群（独立环境）。
  2. 迁移业务应用到新集群。
  3. 切换流量至新集群，旧集群作为备份。

- **优点**：零业务中断，快速回滚。
- **缺点**：资源消耗翻倍，需同步配置和数据。

##### **2. 滚动升级（Rolling Update）**
- **适用场景**：高可用集群，允许短暂节点不可用。
- **操作步骤**：
  1. 新增一台节点，部署新版本 Master/Worker 组件。
  2. 将新节点加入集群，验证功能正常。
  3. 逐步下线旧节点，重复直到所有节点升级完成。

- **优点**：资源占用低，逐步验证。
- **缺点**：升级期间部分节点版本不一致，需兼容性保障。

##### **3. 原地升级（In-Place）**🌟🌟🌟
- **适用场景**：高可用集群，无额外资源。
- **操作步骤**：
  1. **升级 Master 节点**（逐台操作）：
     ```bash
     # 使用包管理工具升级组件（如 apt/yum）
     apt-get update && apt-get install kubeadm=1.31.0-00 kubelet=1.31.0-00 kubectl=1.31.0-00
     
     # 应用升级
     kubeadm upgrade apply v1.31.0
     
     # 重启服务
     systemctl restart kubelet
     ```
  2. **升级 Worker 节点**（批量操作）：
     ```bash
     # 驱逐节点上的 Pod
     kubectl drain <node-name> --ignore-daemonsets
     
     # 升级组件并重启
     apt-get update && apt-get install kubeadm=1.31.0-00 kubelet=1.31.0-00
     kubeadm upgrade node
     systemctl restart kubelet
     
     # 恢复节点调度
     kubectl uncordon <node-name>
     ```

  **严格遵循步骤的先后顺序**：高版本的master通常可以管理更低版本的worker，但版本差异过大也会有问题
  
- **优点**：无需额外资源，直接更新。
- **缺点**：升级期间集群短暂不可用，依赖高可用架构。回滚难度较方案二大。
