## 一、控制器概述
### 1. 什么是控制器？
- **核心作用**：自动化管理与维护资源状态的工具
- **分类**：
  - **内置控制器**：ReplicaSet、Deployment、DaemonSet、StatefulSet、Job、CronJob（各具特定管理逻辑）
  - **自定义控制器**：用户根据需求自行开发

### 2. 为何需要控制器？
- **资源创建本质**：仅向 etcd 数据库写入数据
- **核心需求**：
  - Watch 资源变化（监听数据库表）
  - 创建资源并持续维护状态
  - 确保实际状态（status）与期望状态（spec）一致

### 3. 控制器原理
- **Watch 机制**：持续监听特定资源类型（如Deployment）
- **调谐循环（Reconciliation Loop）**：
  ```go
  for {
    desired := getDesiredState()  // 从 spec 获取期望状态
    current := getCurrentState()   // 从 status 获取当前状态
    if current != desired {
      reconcile(current, desired) // 调整至一致状态
    }
  }
  ```
## 二、内置控制器详解
### 1. ReplicaSet
- **核心功能**：维护 Pod 副本数量

- **特点**：
  - 通过 `spec.replicas` 控制副本数
  
  - 通常由 Deployment 管理，不直接使用
  
    - **必要性**：
  
      - ReplicaSet仅能感知副本数量的变化，但**无法感知Pod模板的更新**（如镜像版本变化）
  
      ```yaml
      apiVersion: apps/v1
      kind: Deployment
      metadata:
        name: nginx-deployment
      spec:
        replicas: 3  
        selector:
          matchLabels:
            app: nginx
        template:
          metadata:
            labels:
              app: nginx
          spec:
            containers:
            - name: nginx
              image: nginx:1.18  # 更新镜像版本时会触发新建ReplicaSet
              ports:
              - containerPort: 80
      ```

- **OwnerReferences字段与级联删除的关系**

  - **`ownerReferences` 字段**：
    - 存在于被管理资源（如 Pod）的 metadata 中
    - 记录其**属主资源**（如 ReplicaSet、Deployment）的 UID
    - 用于建立资源间的**所有权关系**

  - **级联删除行为**
    - **删除属主资源**时：
      - 默认触发**级联删除**（Cascading Delete）
      - 自动删除其管理的所有子资源（如 ReplicaSet 删除 → 其 Pod 被删除）

  - **删除子资源**时：
    - 若子资源被单独删除（如手动删除 Pod）
    - 控制器（如 ReplicaSet）会**立即重建**该资源（以维持 `replicas` 数量）

  **结论**：**直接操作 Pod 无意义**，控制器会持续调谐状态（除非删除其属主资源）

  > **示例验证**
  >
  > **场景**：ReplicaSet（`myapp-rs`）管理 3 个 Pod
  >
  > ```yaml
  > # 查看 Pod 的属主信息
  > kubectl get pod <pod-name> -o yaml
  > ```
  >
  > 输出片段：
  >
  > ```yaml
  > metadata:
  >   ownerReferences:
  >   - apiVersion: apps/v1
  >     kind: ReplicaSet
  >     name: myapp-rs
  >     uid: 5a3f5f8e-d6a1-11e9-8f9b-42010a8000a3
  > ```
  >
  > **操作验证**：
  >
  > ```sh
  > # 删除 ReplicaSet（触发级联删除）
  > kubectl delete replicaset myapp-rs
  > 
  > # 观察结果：所有关联 Pod 被自动删除
  > kubectl get pods
  > 
  > # 若仅删除 Pod（会被 ReplicaSet 重建）
  > kubectl delete pod <pod-name>
  > ```

### 2. Deployment
- **层级关系**：Deployment -> ReplicaSet -> Pods

  > - **命名规则**
>
  >   | 资源类型       | 命名格式                      | 说明                                                         |
  >   | :------------- | :---------------------------- | :----------------------------------------------------------- |
  >   | **Deployment** | `<用户定义的名称>`            | 直接使用资源清单中的 `metadata.name`                         |
  >   | **ReplicaSet** | `<Deployment名称>-<哈希值>`   | 哈希值标识 **模板版本**（相同模板生成的ReplicaSet哈希值相同） |
  >   | **Pod**        | `<ReplicaSet名称>-<随机哈希>` | 随机哈希保证 **Pod唯一性**                                   |
>
  >   ```
  >   # 示例：
  >   # Deployment
  >   nginx-deployment
  >   
  >   # 关联的ReplicaSet
  >   nginx-deployment-5d59d67564
  >   
  >   # 关联的Pod
  >   nginx-deployment-5d59d67564-7kqvh
  >   nginx-deployment-5d59d67564-9tx2l
  >   ```
>
  > - Pods会被打上 `pod-template-hash` 标签
>
  >   ```
  >   # 示例：
  >   labels:
  >     pod-template-hash: 5d59d67564  # 与所属ReplicaSet哈希值一致
  >   ```
>
  >   - **作用**：
  >     - 标识 Pod 所属的模板版本
  >     - 防止不同版本 ReplicaSet 管理的 Pod 发生冲突
  >     - 支持滚动更新时新旧版本共存

- **核心功能**：

  [具体操作]("C:\Users\Lenovo\桌面\暑期实习\技术相关\k8s\基础\04 k8s常用操作一把梭.md")

  - **滚动更新**（RollingUpdate）

    - 一种渐进式的更新策略，用于在不中断服务的情况下，用新的Pod实例逐步替换旧的Pod实例 。

    - **触发条件：**

      - 当资源（以Deployment为例）中的Pod模板镜像被修改时，触发滚动更新。

    - **更新过程：**

      <img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/2108785-20210103211612542-2030719852.png" alt="img" style="zoom: 25%;" align='left'/>

      1. **继续服务：** 原有的ReplicaSet（RS）管理的Pod继续服务已有的连接。
      2. **逐步替换：** 逐渐关闭旧的Pod，并在新的RS中启动新的Pod来接收新的服务请求。
      3. **点点向前：** 更新过程是逐步进行的，像“滚动”一样，一点一点地向前推进。
      4. **完成更新：** 最终，所有旧的Pod都被新的Pod替换，完成整个滚动更新过程。

    - **本质**：

      - 通过 **动态调整新旧 ReplicaSet 的 `replicas` 值**，实现新旧版本 Pod 的渐进式替换

      - 整个过程由 **Deployment 控制器** 协调完成，本质是多个 ReplicaSet 控制器协同工作的结果

      - > ##### **核心调谐步骤**
        >
        > 1. **创建新 ReplicaSet**
        >    - 根据新 Pod 模板生成新 ReplicaSet（名称带新哈希值）
        >    - **初始 `replicas: 0`**（此时旧 ReplicaSet 仍维持原副本数）
        > 2. **渐进式调整副本数**
        >    - **新 ReplicaSet**：`replicas` **逐步增加**（按 `maxSurge` 策略）
        >    - **旧 ReplicaSet**：`replicas` **逐步减少**（按 `maxUnavailable` 策略）
        > 3. **最终状态收敛**
        >    - 新 ReplicaSet 的 `replicas` 达到目标值
        >    - 旧 ReplicaSet 的 `replicas` 降为 0（但保留历史记录以便回滚）

    - **更新策略**：

      ```yaml
      spec:
        minReadySeconds: 5     # 新创建的 Pod 必须在就绪状态（Ready）保持至少多少秒后，才会被视为可用并开始接收流量
        						 # 作用：确保新 Pod 真正稳定运行后再进行下一步的更新操作，从而避免因新 Pod 不稳定而导致的潜在问题
        strategy:
          type: RollingUpdate  # 或 Recreate（重建式更新）
        					     # Recreate:先同一时间删除所有旧Pod，然后再创建新Pod
          rollingUpdate:
            maxSurge: 5        # 默认1，更新期间允许超出spec.replicas额外新建的最大 Pod 数（新建速度）
            				     # 值可用时一个绝对值或百分比
          				     # 如果是30%，意味着新老pod总数不能超过期望pod数的130%
            maxUnavailable: 2  # 默认1，更新期间最大可终止 Pod 数（删除速度）
          				   # 如果是30%，意味着启动滚动更新后旧pod会立即缩容至期望pod数的70%
          
      # 更新结束后会根据期望状态继续调谐
      ```

      > **注意**：maxSurge 和 maxUnavailable 的属性值**不可同时为0**，否则 Pods 的副本数量无法进行合理变动以完成更新操作。
      >
      > - `maxSurge=0`：新的Pod只能替换旧的 Pod，而不会增加总的 Pod 数量。
      >
      > - `maxUnavailable=0`：在升级过程中不允许杀死任何旧的 Pods，必须等待 maxSurge 指定的新Pods 全部启动并准备就绪后，才会开始删除旧 Pods，影响更新速度。

      <img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/cbafd541b9d32c99fd3df461cff6ab4e38446735.png" alt="img" style="zoom: 30%;" align='left'/>

      **问题**：新老版本共存的时候，无法区分Bug出在哪

  - **版本回滚**（Rollback）

    - **版本回滚实战**

      ```bash
      # 查看版本历史
      kubectl rollout history deployment nginx-ba
      # 输出示例：
      # REVISION  CHANGE-CAUSE
      # 1         <none>        # 初始版本（nginx:1.14）
      # 2         <none>        # 更新后的版本（nginx:1.15）
      # 可指定具体版本
      kubectl rollout history deployment nginx-ba --revision=2
      
      # 回滚到上一版本
      kubectl rollout undo deployment nginx-ba
      # 回滚到指定版本
      kubectl rollout undo deployment nginx-ba --to-revision=1
      
      # 观察回滚过程
      kubectl rollout status deployment nginx-ba
      # 输出示例：
      # Waiting for deployment "nginx-ba" rollout to finish: 1 old replicas pending termination...
      # deployment "nginx-ba" successfully rolled out
      
      # 验证回滚结果
      # 查看Pod的Nginx版本
      kubectl exec -it <reverted-pod-name> -- nginx -v
      # 预期输出：nginx/1.14.2
      ```


- **水平扩缩**（非核心功能）：

  ```sh
  # 方式一：scale
  kubectl scale deployment nginx-deploy --replicas=4
  
  # 方式二：修改yaml，然后apply -f
  
  # 方式三：kubectl edit修改
  kubectl edit deploy nginx-deploy
  
  # 方式四：打补丁
  kubectl patch deployment nginx-deployment -p '{"spec":{"replicas":4}}'
  ```

- **补充**：

  创建Deployment时，可以使用 `--record` 参数来记录创建或更新的命令。它将执行的命令保存为 Deployment 的注解，从而在后续查看修订历史（revision history）时，能够显示每个修订的`CHANGE_CAUSE`，即变更原因。

  ```sh
  kubectl apply -f test.yaml --record
  ```

### 3. DaemonSet

- **核心特点**：默认在每个节点运行一个 Pod 副本

  - **如何保证**：
    - **获取Node列表**：DaemonSet 控制器首先从 Etcd 中获取到集群中所有 Node的列表。
    - **遍历Node**：控制器遍历每一个 Node，准备进行Pod的部署或清理操作。
    - **检查调度配置**：对于每个 Node，控制器根据 DaemonSet 资源清单中的调度相关配置（如节点选择器、亲和性等）检查该Node是否符合运行Pod的条件。
    - **检查Pod存在性**（在符合运行Pod条件的节点上）：
      - **无Pod**：如果在该Node上没有找到对应的Pod，控制器会在该Node上创建一个Pod。
      - **多个Pod**：如果在该Node上找到了多个对应的Pod，控制器会删除多余的Pod，确保只有一个。
      - **一个Pod**：如果在该Node上正好有一个对应的Pod，则符合预期，不做任何操作。

- **层级关系**：DaemonSet -> Pods

  - DaemonSet 有自己的副本管理机制

- **应用场景**：

  - 网络插件（如 Flannel）
  - 负载均衡（如 kube-proxy）
  - 日志收集（如 Filebeat）
  - 节点监控（如 Prometheus Node Exporter）

- **调度控制**：可通过 `nodeSelector` 或 `nodeAffinity` 限制在哪些节点上运行一个Pod副本

- **滚动更新**：

  - **更新策略类型**

    DaemonSet 支持两种更新策略，通过 `.spec.updateStrategy.type` 字段定义：

    | 策略类型            | 触发条件                                                     | 特点                                                         | 适用场景                             |
    | :------------------ | :----------------------------------------------------------- | :----------------------------------------------------------- | :----------------------------------- |
    | **`OnDelete`**      | 更新 DaemonSet 模板后，**需手动删除旧 Pod** 才会创建新 Pod   | - 完全由人工控制更新节奏                                     | 需严格验证新版本稳定性后再部署的场景 |
    | **`RollingUpdate`** | 更新 DaemonSet 模板后，**自动终止旧 Pod 并按策略创建新 Pod**（默认策略） | - 自动滚动更新 <br />- 支持配置并发控制（`maxUnavailable`） <br />- 保证服务连续性 | 常规自动化部署场                     |

  - ##### **`RollingUpdate` 策略流程**

    - 更新 DaemonSet 的 Pod 模板
    - 控制器按以下顺序操作：
      - 根据 `maxUnavailable` 值选择一批节点
        - `maxUnavailable`:  默认1，更新期间最大可终止的**节点**数（删除速度）
      - **终止旧 Pod** → **创建新 Pod** → **等待新 Pod 进入 Ready 状态**
      - 循环上述过程，直到所有节点完成更新

  - **配置示例**：

    ```yaml
    apiVersion: apps/v1
    kind: DaemonSet
    metadata:
      name: nginx-ds
    spec:
      updateStrategy:
        type: RollingUpdate
        rollingUpdate:
          maxUnavailable: 2  # 允许同时更新2个节点
      template:
        metadata:
          labels:
            app: nginx
        spec:
          containers:
          - name: nginx
            image: nginx:1.20  # 更新镜像版本触发滚动更新
    ```

### 4. StatefulSet

#### 4.1 **核心概念**

- **StatefulSet**：K8s 中专门用于管理 **有状态应用** 的控制器资源
  - 唯一 一个管理有状态服务的内置控制器
  - 有状态应用的每个 Pod 都是独一无二的
  - 为 Pod 提供：
    - 稳定的网络标识（固定 DNS 域名）
    - 持久化存储（独立 PVC/PV）
    - 有序部署/扩缩容（按序号顺序操作）

- **有状态 vs 无状态：**

  | 特性         | 有状态应用                                 | 无状态应用                          |
  | :----------- | :----------------------------------------- | :---------------------------------- |
  | **数据依赖** | 依赖持久化数据（如数据库、分布式存储）     | 不依赖持久化数据（如 Web 前端服务） |
  | **调度限制** | Pod 必须绑定到特定节点（需访问持久化数据） | Pod 可自由调度到任意节点            |
  | **典型场景** | MySQL 集群、ZooKeeper、Elasticsearch       | Nginx、Stateless Microservices      |
#### 4.2 **核心特性**：

| 特性               | 实现方式                              | 作用                    |
| :----------------- | :------------------------------------ | :---------------------- |
| **固定 Pod 名称**  | 名称格式：`<statefulset-name>-<序号>` | 有序操作                |
| **固定 FQNS 域名** | Headless Service + Pod 名称组合       | 提供稳定的网络访问入口  |
| **持久化存储**     | `volumeClaimTemplates` 创建独立 PVC   | 数据生命周期与 Pod 解耦 |

##### 4.2.1 **有序操作原则**

- **启动顺序**：按序号递增顺序（如 web-0 → web-1 → web-2），只有当上一个 Pod 处于 Ready 状态时，下一个 Pod 才会开始启动。

- **终止顺序**：按序号递减顺序（如 web-2 → web-1 → web-0）

- **滚动更新**：逆序删除 + 顺序创建新 Pod

##### 4.2.2 **关键组件**：

- **Headless Service**（无头服务）：
  
  ```yaml
  # nginx-svc.yaml（无头服务示例）
  apiVersion: v1
  kind: Service
  metadata:
    name: nginx
  spec:
    clusterIP: None  # 标识为无头服务
    ports:
    - port: 80
    selector:
      app: nginx
  ```
  
  **特性**：
  
  - 无 ClusterIP（`clusterIP: None`）
  
  - 直接解析 Pod IP
  
  - **FQDN 格式**：`<pod-name>.<svc-name>.<namespace>.svc.cluster.local`
  
    - 示例：`web-0.nginx.default.svc.cluster.local`
    - FQDN（Fully Qualified Domain Name，完全限定域名）：
      - 一个完整的域名，包含了主机名和域名，能够唯一地标识互联网上的一台设备
      - 典型示例：www.example.com，其中“www”是主机名，“example”是域名，“.com”是顶级域名。
    - `coredns` 负责解析
  
  - **VS 普通Service**：
  
    | 服务类型         | 域名格式                                              | DNS 解析结果                              | 流量转发方式       | 适用场景                 |
    | :--------------- | ----------------------------------------------------- | :---------------------------------------- | :----------------- | :----------------------- |
    | **普通 Service** | `<svc-name>.<namespace>.svc.cluster.local`            | 解析到 Cluster IP                         | 负载均衡到随机 Pod | 无状态服务               |
    | **Headless**     | `<pod-name>.<svc-name>.<namespace>.svc.cluster.local` | 直接解析到 Pod IP（返回所有 Pod IP 列表） | 客户端直连指定 Pod | 有状态服务（需精准访问） |

- **VolumeClaimTemplates**：

  ```yaml
  volumeClaimTemplates:
  - metadata:
      name: www
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 1Gi
  ```

  - **特性**：

    - 自动为**每个** Pod 创建**独立**的 PVC（Persistent Volume Claim，持久卷声明）

    - PVC 名称格式：`<卷声明模板名称>-<Pod 名称>`（如 `www-web-0`）

    - 删除 Pod 时 **保留 PVC**（需手动清理）

  - **注意**：

    若 PV 使用节点本地存储（如`local`类型），Pod 重启后仍在原节点。因为 PV 与节点强绑定，Pod 必须调度到拥有对应 PV 的节点才能挂载存储。

    - **完整调度决策流程**：

      - 当 StatefulSet Pod 发生重启（如手动删除或故障）时：

        1. **触发重建**
           StatefulSet 控制器检测到 Pod 缺失，启动重建流程

        2. **名称复用**
           新 Pod 必须使用原名称（如 `web-0`）

        3. **存储约束检查**

           - 检查该 Pod 关联的 PVC（如 `data-web-0`）

           - 若 PVC 绑定到 **本地 PV**，调度器强制选择原节点

        4. **亲和性规则匹配**

           - 检查 Pod 的 `nodeName`（原调度节点记录）

           - 检查 `nodeAffinity` 或 `podAffinity` 规则

        5. **最终调度决策**

        **综合上述约束**，若原节点满足条件，则优先调度到原节点

    - PV示例：

      ```yaml
      apiVersion: v1
      kind: PersistentVolume
      metadata:
        name: hostpath-pv-demo
      spec:
        capacity:
          storage: 5Gi
        accessModes:
          - ReadWriteOnce
        storageClassName: hostpath-storage
        hostPath:
          path: /mnt/data  # pv被pod绑定时会自动在pod被调度到的节点上创建该目录
      ```

#### 4.3 **完整配置示例**

```yaml
# StatefulSet 示例
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: web
spec:
  serviceName: "nginx"  # 关联的 Headless Service 名称
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.20
        ports:
        - containerPort: 80
        volumeMounts:
        - name: www
          mountPath: /usr/share/nginx/html
  volumeClaimTemplates: # 可看作 PVC 模板
  - metadata:
      name: www
    spec:
      accessModes: ["ReadWriteOnce"]
      storageClassName: "ssd"
      resources:
        requests:
          storage: 1Gi
```

#### 4.4 **操作验证**

```sh
# 查看固定 DNS 记录
kubectl run -it --rm dns-test --image=busybox:1.28 --restart=Never -- nslookup web-0.nginx

# 检查持久化存储
kubectl get pvc  # 预期看到 www-web-0, www-web-1 等 PVC

# 观察有序扩缩容
kubectl scale sts web --replicas=5  # 按顺序创建 web-3, web-4
kubectl scale sts web --replicas=2  # 按逆序删除 web-4, web-3, web-2
```

#### 4.5 补充

- **`.spec.podManagementPolicy`字段**：指定 Pod 启动和删除行为的策略，可选值有：
  - **`OrderedReady（默认值）`**：所有 Pod 有序启动，有序删除
  - **`Parallel`**：所有 Pod 并行启动，并行删除
- **`.spec.updateStrategy.type`字段**：指定 Pod 更新策略，主要选项：
  - **`OnDelete（默认值）`**：必须手动删除 Pod，StatefulSet 才会创建新的 Pod 来替换被删除的 Pod
  - **`RollingUpdate`**：自动进行滚动更新
- **`.spec.volumeClaimTemplates.storageClassName`字段**：指定存储类
  - 如果指定的 StorageClass 支持动态存储分配，K8s 会自动为每个 PVC 分配相应的 PV。否则，k8s 的调度会受限于 PV 的数量。
- **`.spec.persistentVolumeReclaimPolicy`字段**：指定 PVC 释放后 PV 的处理方式：
  - `Retain`：保留数据（手动清理）
  - `Delete`：自动删除（需存储系统支持）
  -  `Recycle`（已废弃）：基础擦除
### 5. Job

#### 5.1 核心特定

- **用途**：启动 Pod 执行**一次性任务**，任务完成后 Pod 状态为 `Completed`。
- **Completed 状态**：
  - 表示 Pod **正常执行完毕**，容器退出状态码为 `0`。
  - 若任务非正常退出（如崩溃或非零状态码），可能触发 `CrashLoopBackOff`，表示容器反复崩溃且重启失败。
#### 5.2 VS **普通 Pod 模板**

- **容器要求**：
  - 必须是**可执行完毕的任务**，而非常驻前台进程（如 `while true` 循环需谨慎使用）。
- **重启策略**：
  - 仅支持 `Never` 或 `OnFailure`，不支持 `Always`（因 Job 为一次性任务）。

#### 5.3 **关键参数解析**

##### 1. **超时控制 `activeDeadlineSeconds`**

- 设置 Job 的最长运行时间（单位：秒），超时后强制终止 Job。

- **示例**：

  ```yaml
  # job1.yaml
  apiVersion: batch/v1
  kind: Job
  metadata:
    name: job-demo
  spec:
    activeDeadlineSeconds: 15
    template:
      spec:
        restartPolicy: Never
        containers:
        - name: counter
          image: busybox
          command:
          - "bin/sh"
          - "-c"
          - "i=0;while true; do echo $i; sleep 1;let i++;done"
  ```

  **行为分析**：

  - 容器执行无限循环，无法自行退出。
  - 因设置 `activeDeadlineSeconds: 15`，Job 在 15 秒后超时终止，状态为失败。

##### 2. **重启策略与 `backoffLimit`**

- **`restartPolicy: OnFailure`**：

  - Pod **自身重启**（不新建 Pod），直到成功或达到 `backoffLimit` 次数。
  - 重启间隔呈**指数退避**（间隔时间逐渐增加）。

- **`restartPolicy: Never`**：

  - Pod 失败后不会重启，但 Job 控制器会**创建新 Pod**，直到成功或达到 `backoffLimit` 次数。

- **`backoffLimit`**：

  - 默认值为 `6`，表示允许的失败重试次数（包括 Pod 重启或新建 Pod）。
  - 超过此限制后，Job 标记为失败。

- **示例**：

  ```yaml
  # job2.yaml
  apiVersion: batch/v1
  kind: Job
  metadata:
    name: job-demo
  spec:
    template:
      spec:
        restartPolicy: Never
        containers:
        - name: counter
          image: busybox
          command:
          - "bin/sh"
          - "-c"
          - "for i in 15 14 ... 2 1; do echo $i; sleep 1; if [ $i -eq 2 ];then exit -1;fi;done"
  ```

  **行为分析**：

  - 容器在 `i=2` 时以非零状态码退出（`exit -1`）。
  - 因 `restartPolicy: Never`，Job 控制器创建新 Pod 重试，最多允许 `backoffLimit` 次（默认 6 次）。
  - 若所有重试均失败，Job 最终状态为失败。

##### 3. **并行控制与批处理任务**

- **`parallelism`**：同时运行的 Pod 数量。
- **`completions`**：要求任务成功完成的总次数。
- **示例**：`parallelism: 2` 且 `completions: 4` 表示同时运行 2 个 Pod，共需成功完成 4 次。

##### 4. **自动清理 `ttlSecondsAfterFinished`**

- Job 完成后自动删除的延迟时间（单位：秒）。

- **示例**：

  ```YAML
  # job3+4.yaml
  apiVersion: batch/v1
  kind: Job
  metadata:
    name: job-para-test
  spec:
    ttlSecondsAfterFinished: 60
    parallelism: 2
    completions: 4
    template:
      spec:
        containers:
        - name: test-job
          image: busybox
          command: ["/bin/sh","-c","echo test;sleep 10"]
        restartPolicy: Never
  ```

  **行为分析**：

  - 每次并行运行 2 个 Pod，共需成功完成 4 次（`completions: 4`）。
  - 每个 Pod 执行 `echo test` 后退出（状态码 0），Job 最终状态为 `Completed`。
  - 完成后 60 秒自动清理 **Job 及其 Pod**。

#### 5.4 状态总结

| 状态                   | 含义                                                         |
| :--------------------- | :----------------------------------------------------------- |
| `Completed`            | 任务正常完成，所有 Pod 退出码为 0。                          |
| `CrashLoopBackOff`     | `CrashLoop`：容器反复崩溃，k8s 重启失败。<br />`back-off`（退避策略）：如果容器连续快速地崩溃，kubelet 会逐渐增加重启之间的等待时间。<br />出现场景：`restartPolicy: OnFailure` |
| `BackoffLimitExceeded` | 超过 `backoffLimit` 限制，Job 终止并标记为失败。             |
### 6. CronJob

#### 6.1 **核心特点**

- **作用**：在 Job 基础上添加**时间调度功能**，用**于定期**执行一次性任务。
- **资源层级**：`CronJob` → 创建 `Job` → 启动 `Pod`。
- **适用场景**：周期性管理 K8s 资源（如每日备份、定时批处理等）。
#### 6.2 **Cron 表达式格式**

- **语法**：`分钟 小时 日 月 周`，使用 Unix Cron 格式。
- **示例**：
  - `*/5 * * * *`：每 5 分钟执行一次。
  - `0 3 * * 1`：每周一凌晨 3 点执行。
#### 6.3 **关键参数解析**

##### 6.3.1 **并发策略 `concurrencyPolicy`**

- **定义**：控制新 Job 与正在运行的 Job 发生**重叠**时的处理方式。

- **可选值**：

  | 策略      | 行为                                                         | 适用场景                       |
  | :-------- | :----------------------------------------------------------- | :----------------------------- |
  | `Allow`   | **默认值**。允许并发执行，新 Job 直接启动，不等待旧 Job 完成。 | 任务可重叠，不依赖前一次结果。 |
  | `Forbid`  | **禁止并发**。若旧 Job 未完成，新 Job 被忽略。               | 需确保同一时间仅一个实例运行。 |
  | `Replace` | **替换旧 Job**。终止未完成的旧 Job，立即启动新 Job。         | 需要优先执行最新调度的任务。   |

- **示例**：

  ```yaml
  # Allow 策略示例
  apiVersion: batch/v1
  kind: CronJob
  metadata:
    name: my-cronjob
  spec: # CronJob模板
    schedule: "*/1 * * * *"  # 每分钟执行一次
    concurrencyPolicy: Allow  # 允许并发
    jobTemplate: # job模板
      spec:
        template: # pod模板
          spec:
            containers:
            - name: my-container
              image: busybox
              command: ["sh","-c","for i in `seq 1 120`;do echo $i;sleep 1;done"]  # 任务耗时 120 秒（超过调度间隔）
            restartPolicy: OnFailure
  ```

##### 6.3.2 **历史记录限制**

- **参数**：

  - `successfulJobsHistoryLimit`：保留成功的 Job 记录数量（默认无限制）。
  - `failedJobsHistoryLimit`：保留失败的 Job 记录数量（默认无限制）。

- **作用**：避免集群中积累大量历史 Job，节省资源。

- **示例**：

  ```yaml
  apiVersion: batch/v1
  kind: CronJob
  metadata:
    name: my-cronjob
  spec:
    schedule: "*/1 * * * *"
    successfulJobsHistoryLimit: 3  # 保留最近 3 个成功 Job
    failedJobsHistoryLimit: 1      # 保留最近 1 个失败 Job
    jobTemplate:
      spec:
        template:
          spec:
            containers:
            - name: my-container
              image: busybox
              command: ["sh","-c","for i in `seq 1 120`;do echo $i;sleep 1;done"]
            restartPolicy: OnFailure
  ```
### 7. HPA（Horizontal Pod Autoscaler）

#### 7.1 介绍

- **定义**：
  - HPA（水平 Pod 自动扩缩器）是一种根据**指标变化**自动调整 Pod 副本数量的控制器。
  - 通过监控 CPU、内存等资源利用率或自定义/外部指标，动态扩缩应用实例。
- **为何使用HPA**
  - **传统方式**：人工监控资源使用 → 手动扩缩 → 效率低，响应慢。
  - **HPA 优势**：自动化监控与扩缩 → 实时响应负载变化，提升资源利用率。
- **HPA基本原理**
  - **数据来源**：
    - **常规指标**（CPU、内存）：由 `metrics-server` 提供。
    - **自定义指标**（如请求延迟）：通过 Prometheus 等工具采集。
    - **外部指标**（如消息队列长度）：来自集群外部服务（如云服务）。
  - **计算逻辑**：HPA 控制器根据指标值与目标阈值计算所需副本数。
  - **版本差异**：
    - **v1 版本**：依赖 Heapster（已废弃）。
    - **v2 版本**：依赖 Metrics Server，支持多指标和自定义指标。
#### 7.2 HPA的工作原理

![img](https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/ad2b3d7b30956ee223ded852cfa589be38446735.jpg)

**核心工作流程**：

##### **1. Metrics Server 采集指标数据**

- **Metrics Server 角色**：
  - 持续采集所有 Pod 的实时资源指标（如 CPU、内存使用率）。
  - 提供 `metrics.k8s.io` **API**，供 HPA 控制器查询指标数据。

**2. HPA 控制器周期性获取指标**

- **控制器监控周期**：
  - HPA 控制器通过 `kube-controller-manager` 的 `--horizontal-pod-autoscaler-sync-period` 参数设置监控间隔，默认 15 秒。
  - 每个周期内，HPA **通过 Metrics Server 的 API** 拉取目标 Pod 的最新指标数据。
- **数据过滤**：
  - 仅采集与 HPA 规则关联的 Pod 的指标（如特定 Deployment 下的 Pod）。

##### **3. 扩缩容决策与执行（Scaling Decision & Action）**

- **算法计算**：

  - HPA 根据用户定义的规则（如目标 CPU 使用率 50%）计算期望的 Pod 副本数。

  - **计算公式**：
    $$
    \text{期望副本数} = \left\lceil \frac{\text{当前指标值}}{\text{目标指标值}} \times \text{当前副本数} \right\rceil
    $$

    - 当前指标值是所有 Pod 副本的平均值
    - 注意计算值**向上取整**
    - **示例**：requests副本数4，当前副本数 3，得到实际使用率 75% ，又目标 CPU 使用率 50%→ 期望副本数 ⌈3×(75/50)⌉=5⌈3×(75/50)⌉=5

  - 若同时监控多指标（如 CPU 和内存），取各指标计算结果的最大值。

- **扩缩容触发**：

  - 当计算出的期望副本数与当前副本数不同时，HPA 向 Pod 的控制器（Deployment/ReplicaSet/StatefulSet）发起 `Scale` 操作。

  - **边界限制**：确保副本数在用户定义的 **`minReplicas`** 和 **`maxReplicas`** 范围内。

    - 即使计算出的期望副本数大于 `minReplicas` 或小于 `maxReplicas` ，HPA 也会将其限制在定义的范围内。

  - **防抖机制**：

    - **目的**：避免因指标瞬时波动导致的频繁扩缩容，确保系统稳定性。

    - **原理**：

      > - ##### **冷却时间（Cool-Down Periods）**
      >
      >   - HPA 在每次扩缩容操作后进入**冷却期**，在此期间**暂停处理新事件**，防止短时间内连续调整副本数。
      >   - **参数配置**：
      >     - **扩容冷却时间**（默认 **3 分钟**）：
      >       通过 `--horizontal-pod-autoscaler-upscale-stabilization` 参数设置。
      >       例如：扩容后 3 分钟内即使指标再次超限，也不会触发新扩容。
      >     - **缩容冷却时间**（默认 **5 分钟**）：
      >       通过 `--horizontal-pod-autoscaler-downscale-stabilization` 参数设置。
      >       例如：缩容后 5 分钟内即使指标低于阈值，也不会触发新缩容。
      >   - **作用**：
      >     - 防止因负载短暂波动导致的“抖动”（如流量突发后快速回落）。
      >     - 避免频繁扩缩对应用和集群的冲击（如 Pod 频繁创建/销毁导致服务中断）。
      >
      > - ##### **指标趋势容忍度（Tolerance）**
      >
      >   - HPA 要求指标变化必须超过一定**容忍度阈值**才触发扩缩容，默认容忍度为 **10%**。
      >
      >   - **计算逻辑**：
      >     $$
      >     \text{触发条件} = \left| \frac{\text{当前指标值} - \text{目标阈值}}{\text{目标阈值}} \right| \geq \text{容忍度}
      >     $$
      >
      >     - **示例**：目标 CPU 利用率为 50%，当前为 54%（差异 8% < 10%），不触发扩容。
      >
      >   - **调整容忍度**：
      >     通过 `--horizontal-pod-autoscaler-tolerance` 参数修改（如设置为 0.05 表示 5%）。
      >
      > - **工作流程**：
      >   1. **指标采样**：HPA 每隔 15 秒获取一次指标数据。
      >   2. **趋势判断**：
      >      - 若连续多次（如 3 次）检测到指标持续超过阈值**且超过容忍度**，才判定需要扩缩容。
      >   3. **冷却期检查**：
      >      - 检查是否处于上次扩缩容的冷却期内，若在冷却期内则忽略本次事件。
      >   4. **执行扩缩容**：
      >      - 满足条件后触发调整，并重置冷却计时器。
#### 7.2 **Metrics Server**

##### 7.2.1 原理

<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/6f7d13a0d5744678db8abf87cdde11f538446735.png" alt="img" style="zoom:33%;" align='left'/>

- ##### 数据采集机制

  - **数据来源**：
    Metrics Server 通过 **kubelet 的 Summary API** 获取节点和 Pod 的实时资源使用数据。

    - **kubelet**：每个节点上的代理组件，负责管理 Pod 生命周期并暴露监控指标。
    - **Summary API 路径**：`https://<node-ip>:10250/stats/summary`（需认证访问）。

    - **采集流程**：
      1. Metrics Server **周期性**（默认 15 秒）向集群中所有节点的 kubelet 发起 HTTP 请求。
      2. kubelet 返回当前节点的资源使用数据（包括 Pod 的 CPU/内存用量）。
      3. Metrics Server 将数据**聚合**并存储于内存中（不持久化历史数据）。

  - **关键特性**：
    - **高效性**：直接与 kubelet 通信，绕过 API Server，减少延迟。
    - **轻量化**：仅处理实时指标，无长期存储，资源占用低。

- ##### 数据暴露方式

  - **Metrics API**：
    Metrics Server 通过 **API Aggregator** 将指标数据暴露为标准的 API 资源。
    - **API 路径**：`/apis/metrics.k8s.io/v1beta1`
    - **资源类型**：
      - `nodes`：节点级指标（如 `kubectl top nodes`）。
      - `pods`：Pod 级指标（如 `kubectl top pods`）。
##### 7.2.2 API Aggregator（了解）

K8s 的**扩展机制**，允许将自定义 API 服务（如 Metrics Server）**无缝集成到 K8s 核心 API 中**。无需修改 K8s 核心代码即可扩展 API，实现**统一入口**和**认证/鉴权流程**。

- **工作原理**：

  - **注册自定义 API**：

    - 自定义 API 服务（如 Metrics Server）通过 **APIService** 资源注册到 Kubernetes。

    - 示例 APIService 定义：

      ```yaml
      apiVersion: apiregistration.k8s.io/v1
      kind: APIService
      metadata:
        name: v1beta1.metrics.k8s.io
      spec:
        service:
          name: metrics-server
          namespace: kube-system
        group: metrics.k8s.io
        version: v1beta1
        insecureSkipTLSVerify: true  # 跳过 TLS 验证（可选）
      ```

  - **请求路由流程**：

    ```mermaid
    graph LR
      User[kubectl top nodes] --> APIServer
      APIServer -->|路由请求| Aggregator
      Aggregator -->|转发请求| MetricsServer
      MetricsServer -->|采集数据| Kubelet
      Kubelet -->|cAdvisor 指标| MetricsServer
      MetricsServer -->|返回数据| Aggregator
      Aggregator -->|返回结果| User
    ```

    - 用户发起请求（如 `kubectl top nodes`）→ 请求发送到 Kubernetes API Server。
    - API Server 检查请求路径：
      - 若路径匹配核心 API（如 `/api/v1`），由内置逻辑处理。
      - 若路径匹配注册的自定义 API（如 `/apis/metrics.k8s.io`），转发给对应的 **Aggregator** 服务（如 Metrics Server）。

  - **代理与响应**：
    - API Server 将请求代理到 Metrics Server 的 Service（`kube-system/metrics-server`）。
    - Metrics Server 处理请求并返回指标数据 → API Server 将结果返回给客户端。

  > **补充**：
  >
  > - **Metrics Server 与 API Aggregator 的部署关系**
  >
  >   - Metrics Server 以 Deployment 形式运行在 `kube-system` 命名空间。
  >
  >   - 通过 APIService 资源注册到 API Aggregator，使 `metrics.k8s.io` API 可用。

- ##### **关键配置**

  - **启用 Aggregator 路由**（Metrics Server部署前提）❗❗❗：
    修改 API Server 配置（`/etc/kubernetes/manifests/kube-apiserver.yaml`）：

    ```yaml
    # 在.spec.containers.command下
    - --enable-aggregator-routing=true
    ```

  - **证书要求**：
    - 自定义 API 服务（如 Metrics Server）需提供有效的 TLS 证书。
    - 若使用自签名证书，需在 APIService 中设置 `insecureSkipTLSVerify: true`（生产环境不推荐）。

##### 7.2.3 **部署步骤**

1. **修改 API Server 配置**  

   在 `/etc/kubernetes/manifests/kube-apiserver.yaml` 中添加参数：  

   ```yaml
   - --enable-aggregator-routing=true
   ```
   重启 API Server 生效。

2. **下载并修改 Metrics Server 配置**  
   ```bash
   wget https://github.com/kubernetes-sigs/metrics-server/releases/download/v0.7.1/components.yaml
   ```
   修改 `components.yaml`：  
   - 添加 `--kubelet-insecure-tls` 跳过证书验证（自签名证书环境）。  
   - 替换镜像为国内源（如 `k8s.mirror.nju.edu.cn/metrics-server/metrics-server:v0.7.1`）。

3. **部署 Metrics Server**  
   ```bash
   kubectl apply -f components.yaml
   ```

4. **验证安装**  
   ```bash
   kubectl top nodes    # 查看节点资源使用
   kubectl top pods -A  # 查看所有 Pod 资源使用
   ```
#### 7.3 **基于 CPU 的 HPA 配置**
##### **示例配置**
1. **Deployment 定义**  
   ```yaml
   apiVersion: apps/v1
   kind: Deployment
   metadata:
     name: hpa-demo
   spec:
     template:
       spec:
         containers:
         - name: nginx
           image: nginx
           resources:
             requests:
               cpu: 50m  # 必须设置 requests 供 HPA 计算
   ```

2. **HPA 策略（命令行创建）**

   ```bash
   kubectl autoscale deployment hpa-demo --cpu-percent=10 --min=1 --max=10
   ```
   - `--cpu-percent=10`：CPU 利用率目标阈值（超过 10% 触发扩容）。HPA 会尝试保持所有 Pod 的平均 CPU 使用率接近 10%。 
   - `--min=1 --max=10`：副本数范围。

3. **压力测试**  

   前提：有一个svc资源代理Pod

   ```bash
   # 运行一个临时的 BusyBox 容器，并让它不断向 k8s 集群内的 Nginx 服务发起 HTTP 请求
   kubectl run -it --image busybox test-hpa --rm -- /bin/sh -c "while true; do wget -q -O- http://nginx.default.svc.cluster.local; done"
   # ctrl+C终止该命令会触发缩容至 1 Pod，因为CPU实际使用率变为了0
   ```
#### 7.4 **基于内存的 HPA 配置**
##### **示例配置**
1. **内存压力脚本（ConfigMap）**  
   ```yaml
   apiVersion: v1
   kind: ConfigMap
   metadata:
     name: increase-mem-config
   data:
     increase-mem.sh: |
       #!/bin/bash
       mkdir /tmp/memory
       mount -t tmpfs -o size=40M tmpfs /tmp/memory
       dd if=/dev/zero of=/tmp/memory/block
       sleep 60
       rm /tmp/memory/block
       umount /tmp/memory
   # 在系统中临时占用大约 40MB 的内存空间，持续 60 秒后释放这些内存。tmpfs 是一种基于内存的文件系统，占用的是内存空间
   ```
   
2. **Deployment 配置**  
   
   ```yaml
   apiVersion: apps/v1
   kind: Deployment
   metadata:
     name: hpa-mem-demo
   spec:
     template:
       spec:
         containers:
         - name: nginx
           image: nginx:1.18
           volumeMounts: # 脚本注入，需人为进入容器 sh 执行脚本
           - name: increase-mem-script
             mountPath: /etc/script
           resources:
             requests:
               memory: 50Mi  # 必须设置内存请求
           securityContext:
             privileged: true  # 允许执行挂载操作
   ```
   
3. **HPA 策略（yaml文件创建）**  
   
   ```yaml
   apiVersion: autoscaling/v2
   kind: HorizontalPodAutoscaler
   metadata:
     name: hpa-mem-demo
   spec:
     metrics:
     - type: Resource
       resource:
         name: memory
         target:
           type: Utilization
           averageUtilization: 30  # 平均内存利用率超过 30% 触发扩容
     scaleTargetRef:
       apiVersion: apps/v1
       kind: Deployment
       name: hpa-mem-demo
     minReplicas: 1
     maxReplicas: 5
   ```
#### 7.5 注意

HPA 可能有多个，可同时监控多个指标（CPU、内存、自定义等），最终副本数以**最大计算值**为准。
