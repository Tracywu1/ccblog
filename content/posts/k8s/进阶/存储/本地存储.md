## 一、emptyDir存储卷
### 1.1 特性与用途
- **特性**：
  - 在Pod调度到节点时创建空目录，Pod删除后数据随之删除。
  - 仅用于临时数据共享（如容器间日志采集）。
- **限制**：**不支持数据持久化**。
- **典型场景**：Pod内容器间共享文件或创建临时目录。

### 1.2 示例配置
```yaml
# emptydir_demo.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: emptydir
spec:
  selector:
    matchLabels:
      app: emptydir
  template:
    metadata:
      labels:
        app: emptydir
    spec:
      containers:
        - name: nginx
          image: nginx:1.18
          volumeMounts:
            - mountPath: /usr/share/nginx/html
              name: test-emptydir
        - name: busybox
          image: busybox:latest
          volumeMounts:
              name: test-emptydir
      volumes:
        - name: test-emptydir
          emptyDir: {}
```

### 1.3 验证步骤
1. 部署后查看Pod状态：
   ```bash
   kubectl get pods -o wide
   ```

2. 访问Pod IP验证数据共享：
   ```bash
   curl <Pod_IP>
   ```

3. 数据存储路径：  

   `/var/lib/kubelet/pods/<Pod_UID>/volumes/kubernetes.io~empty-dir/test-emptydir`

   ```bash
   # 查看pod的uid
   kubectl get pods <pod-name> -o jsonpath='{.metadata.uid}'
   ```
## 二、hostPath存储卷
- **特性**：
  - 直接使用宿主机本地目录。
  - 适合单节点测试或高性能存储需求（如SSD）。
- **限制**：
  - Pod调度受限（需固定节点）。
  - 节点故障导致数据不可用。
  - 目录资源隔离性差。若该目录的数据源不是一块完整独立的盘，即与他人共享的话，易被其他进程占用。

## 三、Local PV
### 3.1 特性与优势
- **特性**：
  - 基于hostPath + nodeAffinity，**绑定PV到特定节点**。
  - **推荐存储介质**：独立挂载的磁盘或块设备（避免资源争用）。
- **优势**：
  - 数据隔离性更好。
  - 支持动态调度（通过延迟绑定机制）。

### 3.2  创建Local PV

```yaml
# pv-local.yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-local
spec:
  capacity:
    storage: 5Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Delete
  storageClassName: local-storage # 延迟绑定的sc
  local: # local PV
    path: /data/k8s/localpv  # 建议挂载独立磁盘
  nodeAffinity: # 绑定PV到node-01
    required:
      nodeSelectorTerms:
        - matchExpressions:
            - key: kubernetes.io/hostname
              operator: In
              values:
                - k8s-node-01
```

### 3.3 延迟绑定（WaitForFirstConsumer）
- **问题背景**

  - 直接创建PVC可能绑定到资源不足（除存储空间外的资源，如cpu）的节点，导致Pod调度失败。
- **解决方案**

  - 通过`StorageClass`设置`volumeBindingMode: WaitForFirstConsumer`，延迟PVC和PV的绑定至Pod调度阶段。

- **配置StorageClass**：

  ```yaml
  # local-storageclass.yaml
  apiVersion: storage.k8s.io/v1
  kind: StorageClass # 可跨命名空间
  metadata:
    name: local-storage
  provisioner: kubernetes.io/no-provisioner # 说明是手动创建PV，无需动态生成
  volumeBindingMode: WaitForFirstConsumer # 延迟绑定
  ```

- 需要在PV和PVC中声明`storageClassName`

### 3.4 Rancher Local Path Provisioner

- **传统问题**：使用 `hostPath` 或手动创建 `Local PV` 需要人工预先配置存储资源，效率低且难以扩展。
- **解决方案**：
  - Rancher 开发的开源存储插件 Local Path Provisioner 允许 k8s 用户在每个节点上使用本地存储，它动态创建的PV相当于 hostPath + nodeAffinity；
  - 并且对外提供一个了 StorageClass 默认名为**`local-path`**，拥有延迟绑定的特性。

#### 3.4.1 安装
```bash
kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/master/deploy/local-path-storage.yaml
```
#### 3.4.2  动态创建PV示例

```yaml
# pvc-local.yaml（动态绑定）
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc-local
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: local-path  # 使用动态StorageClass

# Pod使用示例（自动绑定PV）
# Deployment即使有多个副本，每个Pod的PV和PVC也都是一样的，不会独一无二
apiVersion: apps/v1
kind: Deployment
metadata:
  labels:
    app: web-test
  name: web-test
spec:
  replicas: 1
  selector:
    matchLabels:
      app: web-test
  template:
    metadata:
      labels:
        app: web-test
    spec:
      containers:
        - image: nginx:1.18
          name: nginx
          volumeMounts:
            - name: wwwroot
              mountPath: /usr/share/nginx/html
      volumes:
        - name: wwwroot
          persistentVolumeClaim:
            claimName: pvc-local
```

#### 3.4.3 StatefulSet 使用示例

```yaml
# statefulset-local.yaml
# 每个Pod都有自己独立的PV和PVC
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: redis
spec:
  serviceName: "redis"
  replicas: 3
  selector:
    matchLabels:
      app: redis
  template:
    metadata:
      labels:
        app: redis
    spec:
      containers:
        - name: redis
          image: busybox:1.27
          command: ["tail", "-f", "/dev/null"]
          volumeMounts:
            - name: redis-data
              mountPath: /data/redis
  volumeClaimTemplates: # PVC模板，每个Pod创建的时候都会照着这个模板建PVC
    - metadata:
        name: redis-data
      spec:
        accessModes: ["ReadWriteOnce"]
        storageClassName: "local-path"
        resources:
          requests:
            storage: 1Gi
```

**验证**

1. 查看自动创建的PV和PVC：
   ```bash
   kubectl get pv,pvc
   ```
2. 验证数据持久化：
   ```bash
   kubectl exec -it <Pod_Name> -- ls /data/redis
   ```
## 四、总结
| **存储类型**  | **适用场景**                | **持久化** | **节点依赖**  | **生产推荐** |
| ------------- | --------------------------- | ---------- | ------------- | ------------ |
| emptyDir      | 临时数据共享                | ❌          | ❌             | ❌            |
| hostPath      | 单节点测试/高性能存储       | ✔️          | ✔️             | ❌            |
| Local PV      | 多节点隔离存储              | ✔️          | ✔️（可控）     | ✔️            |
| Local Path SC | 动态本地存储（StatefulSet） | ✔️          | ✔️（自动调度） | ✔️            |

**注意事项**：

- **hostPath**：避免用于生产环境，仅限测试。
- **Local PV**：推荐挂载独立磁盘，结合StorageClass实现动态调度。
- **延迟绑定**：解决资源与存储绑定的冲突，确保Pod调度灵活性。
