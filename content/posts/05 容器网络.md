# 容器网络
## 一、容器网络故障分析方法
### 1. 了解整个通信流程
> #### 手动构建容器基于Bridge网络模式的完整通信链路
>
> 通常使用docker命令创建容器的时候，这些都已经自动配置好了，手动构建仅为了解整个通信流程。
>
> ##### **（1） 创建使用None网络模式的容器**
> ```bash
> docker container run -d --network none --name test centos:7 tail -f /dev/null
> ```
> - **作用**：创建一个无网络配置的容器，仅保留本地回环接口（lo）。
> - **验证**：
>
>   ```bash
>   docker inspect test | grep -i pid  # 获取容器的进程ID（PID）
>   nsenter -t <PID> -n ip a          # 查看容器的网络接口（仅lo）
>   ```
>
> ---
>
> ##### **（2） 在宿主机创建veth对**
> ```bash
> ip link add name veth_host type veth peer name veth_container
> ```
> - **作用**：创建一对虚拟以太网设备（veth pair），`veth_host`留在宿主机，`veth_container`后续移入容器，连接两个命名空间。
> - **特性**：veth设备成对存在，数据从一端进，另一端出。
>
> ---
>
> ##### **（3） 将veth一端连接到容器网络命名空间**
> - **链接容器网络命名空间**：
>
>   ```bash
>   # ip netns默认会去/var/run/netns/目录下找名称空间的 id
>   # 必须用 ip netns 创建过命名空间，才会产生该目录
>   ipnetns add xxx
>   ls /var/run/netns # 可以看到有一个名为xxx的文件
>
>   # 为使ip命令可操作容器的网络命名空间，必须把容器名称空间链接到该目录下
>   ln -s /proc/<PID>/ns/net /var/run/netns/<PID>
>   ```
> - **移动veth设备到容器内**：
>
>   ```bash
>   ip link set veth_container netns <PID>
>   ```
>
> **配置容器内的veth设备，保证其可用**
>
> - **重命名与激活**：
>
>   ```bash
>   ip netns exec <PID> ip link set veth_container name eth0  # 重命名为eth0
>   ip netns exec <PID> ip link set eth0 up                  # 激活接口
>   ```
> - **分配IP并设置网关**：
>   ```bash
>   ip netns exec <PID> ip addr add 172.17.1.2/16 dev eth0    # 分配IP（需要与网关匹配，注意子网掩码）
>   ip netns exec <PID> ip route add default via 172.17.0.1  # 设置默认网关（指向docker0）
>   ```
>
> ---
>
> ##### **（4） 配置宿主机的veth_host端**
> ```bash
> ip link set veth_host up  # 激活宿主机端的veth设备
> ```
>
> 无需给veth_host分配ip和网关，因为它只需与docker0连接，并由docker0负责路由即可
>
> ---
>
> ##### **（5） 将veth_host连接到docker0网桥（虚拟交换机）**
> ```bash
> ip link set veth_host master docker0  # 将veth_host接入docker0网桥
> ```
>
> 容器引擎默认为docker0配置好了IP
>
> ---
>
> ##### **（6） docker0网桥转发到物理网卡**
>
> 容器引擎默认为docker0配置好了IP，以及以下配置
>
> - **开启路由转发**：
>
>   ```bash
>   echo 1 > /proc/sys/net/ipv4/ip_forward
>   ```
>
>   - **禁用（默认情况）**：当`ip_forward`被设置为0时，系统不会转发任何接收到的IP包。这意味着系统只处理目标地址是本机的IP包。
>   - **启用**：当`ip_forward`被设置为1时，系统会转发那些目标地址不是本机但需要通过本机路由的IP包。
> - **设置防火墙规则**：
>
>   ```bash
>   iptables -P FORWARD ACCEPT  # 允许所有转发流量
>   ```
>

### 2. 网络故障排查步骤
#### **1. 逐层抓包分析**
> **`tcpdump`**：一种常用的网络协议分析工具，主要用于捕获和解析网络上的数据包。
>
> **主要功能**：
>
> - **数据包捕获**：
>   - tcpdump可以捕获经过网络接口的数据包，包括进入和离开主机的数据包。
> - **协议解析**：
>   - 支持多种网络协议，如TCP、UDP、ICMP、IP等，并能解析这些协议的头部信息。
> - **过滤功能**：
>   - 提供强大的过滤表达式，允许用户根据源地址、目标地址、端口号、协议类型等条件过滤数据包。
> - **保存和读取**：
>   - 捕获的数据包可以保存到文件中，供后续分析使用；也可以读取之前保存的数据包文件。
> - **实时监控**：
>   - 可以实时监控网络流量，帮助网络管理员诊断网络问题
>
> **常用命令选项**：
>
> - `-i`：指定要监听的网络接口。
> - `-w`：将捕获的数据包保存到文件。
> - `-r`：读取保存的数据包文件。
> - `-nn`：不解析主机名和端口名，直接显示IP地址和端口号。
> - `-v`、`-vv`、`-vvv`：显示更详细的信息，级别逐渐增加。
> - `-c`：指定要捕获的数据包数量。
>
> 也可以使用`netperf`，这部分内容可自行查阅
- **容器内部（eth0）**：
  ```bash
  nsenter -t <PID> -n tcpdump -i eth0 icmp host www.baidu.com -nnv
  # nsenter: namespace enter
  # -n: network，进入容器的网络命名空间
  # icmp host www.baidu.com：过滤条件
  # 在进程ID为13892的进程的网络命名空间中，使用tcpdump捕获经过eth0接口的，与主机、www.baidu.com相关的ICMP数据包，并显示详细信息，同时不解析主机名和端口名
  ```
- **宿主机veth_host**：
  ```bash
  tcpdump -i veth_host icmp host ww.baidu.com -nnv
  ```
- **docker0网桥**：
  ```bash
  tcpdump -i docker0 icmp host ww.baidu.com -nnv
  ```
- **物理网卡（如ens32）**：
  会将容器 ip 地址通过 NAT 转换为宿主机的 ip 地址与外网通信
  ```bash
  tcpdump -i ens32 icmp host ww.baidu.com -nnv
  ```
#### **2. 检查防火墙与路由**
```bash
iptables -t nat -L -n                # 查看NAT规则（-t：指定目标表；-L：list，列出规则；-n：不要DNS解析）
cat /proc/sys/net/ipv4/ip_forward    # 确认路由转发已开启（值为1）
```
## 二、veth对乱序包问题
### 1. 核心原理
#### 1.1 TCP可靠传输机制
##### **可靠传输基础**
- **序列号与确认应答**：每个数据包有唯一序列号seq，接收方返回ACK确认。
##### **重传机制**
- **超时重传（RTO）**：
  - **RTT**：数据包往返时间。
  - **RTO**：动态计算的超时阈值，超时后重传，后续超时时间加倍。
- **快速重传**：
  - 收到3个重复ACK时立即重传，无需等待超时。
- **SACK（选择性确认）**：
  - 接收方通过SACK选项告知发送方已收到的数据块，帮助发送方仅重传丢失部分。
  - **触发条件**：
    - 默认需收到**3个重复ACK（DupAck）**才触发快速重传。
    - **例外场景**：接收方通过SACK反馈的数据间隙（已接收与未接收数据的差值）超过阈值（`reordering * mss_cache`），则立即重传，无需等待3个DupAck。
#### 1.2 滑动窗口与流量控制
- **滑动窗口**：发送方和接收方各维护一个窗口，表示允许传输的最大数据量。
  - **发送窗口**：已发送未确认的数据范围。
  - **接收窗口**：接收方剩余缓冲区大小。
- **流量控制**：通过动态调整窗口大小避免接收方过载。
### 2. **容器环境中的乱序与重传**
- **乱序包的影响**：
  - 若数据包乱序到达（如seq3早于seq2），接收方通过SACK反馈空白区间，触发发送方计算间隙大小。
  - **间隙过大**：若间隙超过内核阈值（`reordering`参数控制），立即重传，即使后续seq2正常抵达。
    <img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/image-20250402185222669.png" alt="image-20250402185222669" style="zoom: 40%;" align='left'/>
- **容器特性加剧问题**：
  - **veth机制**：容器通过veth pair与宿主机通信，veth_host接收数据包时模拟物理网卡的中断处理，可能导致**CPU调度频繁切换**，加剧乱序。
  - **验证命令**：
    ```sh
    nsenter -t $pid -n netstat -s | grep reordering
    # 输出示例：Detected reordering 933072 times using SACK （使用选择性确认（SACK）检测到 933072 次重新排序。）
    ```
  - **结论**：容器环境中SACK触发的重传概率远高于传统丢包场景。
### **3. RSS与RPS：降低乱序的核心机制**
<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/image-20250402185343489.png" alt="image-20250402185343489" style="zoom:45%;" />
- **RSS（接收端扩展）**
  - **硬件层实现**：
    - 物理网卡通过多RX队列并行收包，每个队列绑定独立CPU处理中断。
    - **哈希映射**：同一数据流（基于五元组哈希）固定分配到同一RX队列，保证**顺序性**。
- **RPS（接收包引导）**
  - **软件层实现**：
    - **工作原理**：
      1. 数据包到达后，内核计算哈希值，映射到目标CPU队列。
      2. 通过**IPI中断**（Inter-Processor Interrupt，处理器[CPU]间中断）通知目标CPU处理数据包，确保**同一数据流由同一CPU处理**。
    - **适用场景**：适用于无RSS支持的虚拟设备（如veth_host）。
    - 同一流数据包短暂被分配到不同CPU的场景（了解即可）：
      - 新数据流首包到达时内核会创建流表绑定记录，若前两个包间隔极短，此时流记录可能未就绪，导致第二个包在绑定生效前触发二次哈希计算，两个包被分配到不同CPU。
        ```
        时间线：
        0ns │ 包1到达 → 触发创建流记录（未完成）
        67ns│ 包2到达 → 流记录未就绪 → 重新哈希 → CPU_X
        200ns│ 流记录创建完成
        201ns│ 包3到达 → 使用流记录 → CPU_Y (与包2不同)
        ```
      - 若目标CPU软中断过载，后续包会临时分配到其他空闲CPU，导致同流包分散处理
      - 若绑定目标CPU因过热降频/休眠突然离线 → 流包被重分配到其他CPU
- **veth对乱序问题的根源**（既无RSS，也无RPS的情况）
  - **veth_host收包流程**：
    1. 容器通过veth_container发送数据至宿主机veth_host。
    2. veth_host模拟物理网卡中断，将数据包放入当前运行容器进程的CPU队列。
  - **乱序诱因**：
    - **CPU切换**：容器进程可能被调度到不同CPU，导致数据包分散到不同队列。
    - **无RSS支持**：veth设备默认无硬件队列分流能力，依赖RPS缓解乱序。
- **RPS的优化与代价**
  - **优化效果**：通过哈希绑定数据流与CPU，减少乱序。
  - **性能代价**：
    - **额外哈希计算**：增加CPU开销。
    - **IPI中断风暴**：高流量场景下软中断（softirq）激增，可能导致CPU利用率飙升。
### 4. 总结与调优建议
| **机制** | **层级** | **作用**             | **容器环境影响**       |
| :------- | :------- | :------------------- | :--------------------- |
| **SACK** | TCP协议  | 选择性重传丢失数据段 | 乱序间隙过大触发误重传 |
| **RSS**  | 硬件     | 多队列分流保序       | 不适用于虚拟设备       |
| **RPS**  | 内核     | 软件模拟多队列分流   | 缓解乱序但增加CPU负载  |
**调优建议**：
1. **启用RPS**：针对veth设备配置RPS，降低乱序概率。（具体的RPS脚本可参考博客）
2. **监控阈值**：调整`/proc/sys/net/ipv4/tcp_reordering`参数，平衡重传敏感度。
3. **CPU亲和性**：绑定容器进程到固定CPU，减少调度导致的队列切换。
4. **硬件升级**：使用支持RSS的SR-IOV虚拟网卡，彻底规避软件层乱序。
