# 容器内存
## 储备知识：
### 1. MMU与内存分页机制
#### 1.1 MMU核心作用
- **问题背景**：32位CPU寻址限制只能访问4G物理内存，实际可用物理内存不足时程序需降低内存占用（程序原本需要使用 4G 内存）
- **解决方案**：引入MMU（Memory Management Unit，内存管理单元）
  - CPU使用虚拟地址寻址，MMU负责虚拟地址→物理地址映射
  - 程序感知4G连续内存空间，突破物理内存限制
    - 即使物理内存不足4GB，或者物理内存碎片化，程序仍然可以感知到自己拥有一个连续的4GB内存空间。
    - 操作系统可以通过将部分硬盘空间作为交换区（swap space），将不常用的内存页交换到硬盘上，从而在物理内存有限的情况下，给程序提供超过物理内存大小的虚拟内存空间。（内存分页机制）
#### 1.2 内存分页（Paging）
- **基本机制**
  - 将虚拟地址和物理地址按固定大小（默认4KB）分割为虚拟页（Page）和物理页帧（Page Frame）。
  - 支持非连续内存分配，并实现虚拟内存（不常用页可换出到磁盘）。
- **页表（Page Table）**
  <img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/490bae94351d7ee7b67b1faec3d6f1b6.png" alt="img" style="zoom:30%;" align='left'/>
  - 存储虚拟地址与物理地址的映射关系，位于物理内存中。
  - CPU通过页表完成地址转换，但直接访问内存效率较低。
### 2. TLB优化机制
#### 2.1 TLB出现背景
- **性能瓶颈**：页表存储于内存，访问速度远慢于寄存器
- **解决方案**：TLB（Translation Lookaside Buffer，快表，是页表的Cache）
  - 缓存高频访问的页表项
  - 物理实现为CPU芯片上的高速缓存
#### 2.2 TLB工作流程
<img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/32e882e5bbc13832e7735b98c2cd9551.png" alt="img" style="zoom:33%;" align='left'/>
- **优化动机**
  TLB容量有限，频繁未命中会导致性能瓶颈。
  **解决方案**：增大内存分页大小（HugePages）。
### 3. 大页内存（HugePages）
#### 3.1 技术原理
- **核心思路**：增大分页尺寸
- **性能优势**：
  - 相同TLB容量下存储更多地址映射
  - 例：4MB页 vs 4KB页 → 多存储1000倍映射关系
#### 3.2 Linux配置实践
##### 3.2.1 环境检查
```bash
cat /proc/meminfo | grep Huge
# 输出示例：
HugePages_Total: 0
HugePages_Free: 0
Hugepagesize: 2048kB
```
##### 3.2.2 关键配置
1. **设置共享内存最大值**（需>JVM堆大小）
```bash
echo 4294967295 > /proc/sys/kernel/shmmax  # 4GB
```
2. **配置大页数量**
```bash
# 计算公式：总内存需求 / Hugepagesize
echo 154 > /proc/sys/vm/nr_hugepages  # 示例：1.5G内存/10MB页=>154页
```
3. **永久生效配置**
```bash
# 在/etc/sysctl.conf添加：
vm.nr_hugepages = 154
kernel.shmmax = 4294967295
# 执行生效
sysctl -p
```
#### 3.3 注意事项
- **OS版本要求**：
  - Linux ≥2.6内核 或 2.4内核+大页补丁
- **内存分配**：
  - 大页内存需连续物理内存，建议系统启动时预留
- **验证方法**：
  ```bash
  grep -i huge /proc/meminfo
  # 观察HugePages_Total是否生效
  ```
## 一、OOM介绍
### 1. 什么是OOM？
- **OOM**（Out of Memory）是Linux系统的内存保护机制，当系统或容器内存耗尽时触发，强制终止进程（选择性的）以释放内存。
### 2. 触发OOM的时机
- **系统级OOM**：物理机总内存耗尽，所有进程都可能被终止。
- **容器级OOM**：容器内存使用达到cgroup限制（`memory.limit_in_bytes`），直接终止容器。
### 3.为什么一定要终止进程（系统级OOM）
- **误解**：许多人认为在Linux中，内存申请是通过调用malloc()完成的，一旦内存不足，该方法调用失败即可，无需杀死进程。
- **实际情况**：Linux系统采用内存超配机制，即允许程序申请的内存超过实际物理内存大小。程序申请的内存实际上是虚拟内存，操作系统为程序分配的是一个地址范围，而不是直接的物理内存。只有当程序向这个地址范围写入数据时，才会真正占用物理内存。
  - **超配机制的优点**：Linux系统不会在内存申请时立即失败或杀死进程，而是允许程序在虚拟地址空间中进行操作，直到真正需要物理内存时才进行分配。这种机制提高了内存的使用效率。
- **原因**：当系统中的物理内存被耗尽，而又有进程需要更多的内存时，操作系统必须终止一些进程来释放内存，以保证系统的稳定运行和其它进程的正常执行。
### 4. 选择终止进程的机制（系统级OOM）
- **评分公式**：`评分 = 系统总可用页面数 × oom_score_adj + 进程已用物理页面数`
  - 系统总可用页面数：Linux系统中当前可用的物理内存页面总数。这些页面尚未被分配给任何进程或系统使用，可以随时被分配。
  - 进程已用物理页面数：特定进程当前占用的物理内存页面总数。
- **关键参数**：
  - `/proc/[pid]/oom_score_adj`：调整进程被终止的优先级（范围：-1000~1000）。
  - **评分越高**，进程被选中的概率越大。
## 二、Memory Cgroup核心参数
每个容器对应一个cgroup控制组（路径示例：`/sys/fs/cgroup/memory/system.slice/docker-xxx`）。
| 参数                    | 作用                                                         |
| ----------------------- | ------------------------------------------------------------ |
| `memory.limit_in_bytes` | 容器内所有进程可用的最大物理内存（单位：字节），**子组限制不能超过父组**。 |
| `memory.oom_control`    | 控制OOM机制开关：<br> - `0`（默认）：启用OOM <br> - `1`：禁用OOM（达到`limit_in_bytes`后容器不会被终止，但是会进入可中断睡眠S） |
| `memory.usage_in_bytes` | 只读参数，显示容器内所有进程占用的总物理内存（含RSS（进程实际占用的物理内存大小）和Page Cache）。 |
> **RSS**：当进程对内存地址开始做**真正读写**时，系统才把实际需要的物理内存分配给进程。这个过程中，进程真正得到的物理内存，就是 RSS。
>
> |       **操作方式**       | 是否触发物理分配 |        增加RSS的部分        |        Page Cache是否参与        |
> | :----------------------: | :--------------: | :-------------------------: | :------------------------------: |
> |  **写匿名内存** (堆/栈)  |   ✅ 首次写入时   |      写入的私有物理页       |              ❌ 无关              |
> |   **`mmap`文件后读取**   |   ✅ 首次访问时   |  文件数据所在的共享物理页   |        ✅ 同一页计入Cache         |
> | **`read()`读文件到缓存** |    ✅ 写缓存时    | 用户缓冲区`buf`的私有物理页 | ✅ 数据副本在Cache, **不**计入RSS |
>
> **Page Cache**：系统调用 read() 和 write() 的default行为都会把读过或者写过的页面存放在 Page Cache 里（只有仍有空闲内存）。
## 三、定位OOM问题与解决
```sh
# 查看内核日志
journal -k
# 或者
cat /var/log/messages
```

### 1. 日志关键信息
1. **进程内存分配**（`mem_alloc`）：
   - 查看`rss`（物理内存页数，1页=4KB）和`oom_score_adj`值。
2. **触发OOM的容器/控制组**：
   - 确认是系统级还是容器级OOM。
     - 如果罗列出的是整个操作系统的进程，那可以断定发生的是一次系统级的OOM
3. **被杀进程的PID**：
   - 定位具体进程。
### 2. 处理策略
- **调整内存限制**：若进程需更大内存，增加`memory.limit_in_bytes`。
- **修复内存泄漏**：若进程因Bug耗尽内存，需修复代码。
## 四、容器内存占满了就一定会OOM吗
### 1. 内存组成
- **公式**：`memory.usage_in_bytes = RSS + Page Cache`
  - `Page Cache`：Linux系统用于提升内存利用率的机制，可以**在内存不足时被释放，供RSS使用**。
    - 手动释放 `Page Cache` 的命令：`echo 3 > /proc/sys/vm/drop_caches`
    - `Page Cache` 相当于 `Free` 命令中的 `buffer(读缓冲)/cache(写缓存)`
- **查看真实内存使用**：
  - 通过`/sys/fs/cgroup/memory/<cgroup_path>/memory.stat`文件获取`rss`值（排除Page Cache干扰）。
  - `free` 命令中显示的 `available` 是真实可用内存。
### 2. 示例场景
由于 Page Cache 的特性，读写频繁的容器可能会经常处于最大内存限制，但只要 RSS 没有接近最大限制，就不用担心会触发 OOM。
- **初始状态**：
  - `rss=100M`，`Page Cache=899M`，`memory.usage_in_bytes=1000M`，`memory.limit_in_bytes=1000M`
- **内存增长**：
  - `rss`增至200M时，`rss=200M`，`Page Cache=799M`。
  - 若**`Page Cache`无法释放，触发OOM**。
## 五、Swap分区
**Swap分区**：作为物理内存的扩展，用于在内存不足时将部分数据交换到磁盘上，但大量使用 Swap 会降低系统的运行效率。
### 1. Swap分区的影响
#### 1.1 系统级OOM
- **可用内存** = 物理内存 + Swap。
- **触发条件**：物理内存和 Swap 均耗尽。
#### 1.2 容器级OOM
- **内存限制失效**：如果开启 SWAP，容器的内存限制（memory cgroup）可能会失去意义，因为容器在物理内存不足时总会交换到 SWAP 分区，即便容器内所有进程占用的内存量达到了容器最大内存限制也不会触发OOM。
- **启用Swap**：
  - 通过`swapon`命令启用指定的 SWAP 分区
  - 使用`-m`参数指定容器可用的物理内存（rss）大小，使用`--memory-swap`参数指定可用物理内存和SWAP分区的总大小。（如`docker run -m 200M --memory-swap=300M`）
#### 1.3 Swap分区的使用场景
**程序启动阶段的内存需求**：某些程序在启动阶段需要大量内存，但运行后内存需求减少。对于这类程序，SWAP分区可以防止其在启动阶段被OOM-Killer杀死。
### 2. Swap控制策略
#### 2.1 全局控制
- **`swappiness `参数**：权重值，控制匿名内存与Page Cache的释放比例（默认值60，范围0-100）。值越高，越积极使用 Swap 来回收内存。
  - 位置：`/proc/sys/vm/swappiness`
#### 2.2 容器级控制
- **禁用 Swap**：`docker run --memory-swappiness=0`。
- **注意**：
  - 容器级`memory.swappiness=0`表示完全禁用Swap，会覆盖全局的 `swappiness` ，即容器的优先级更高。
  - 系统级`swappiness=0`仅**减少**Swap使用，非彻底禁用。
#### 2.3 内存优化策略：Page Cache与Swap的平衡
##### 2.3.1 **Page Cache VS 匿名内存**
| **类型**       | **描述**                                                     | **释放方式**                         |
| :------------- | :----------------------------------------------------------- | :----------------------------------- |
| **Page Cache** | **加速文件访问**：当文件首次被读取时，数据从磁盘加载到内存的Page Cache中。后续再次访问同一文件时，直接从内存读取，避免磁盘I/O。<br />**延迟写入优化**：写入操作可能先写入Page Cache，由内核异步刷盘，减少直接写磁盘的等待时间。<br />可被内核直接释放，无需写入磁盘。 | 直接释放，不影响数据完整性。         |
| **匿名内存**   | 无对应磁盘文件的内存（如`malloc`申请的内存），需通过Swap分区换出到磁盘才能释放。 | 需写入Swap分区后释放，存在性能开销。 |
##### 2.3.2 **内存不足时的释放策略**
当物理内存不足时，Linux需平衡两种操作：
1. **释放Page Cache**：快速但可能影响文件I/O性能（需重新加载缓存）。
2. **换出匿名内存到Swap**：避免数据丢失但引入磁盘I/O延迟。
**核心矛盾**：
- **只释放Page Cache**：频繁文件操作时性能下降。
  - **高频访问文件失去缓存**：
    - 假设一个Web服务器日志文件被频繁读取，若其Page Cache被释放，每次读取需重新访问磁盘。
  - **写入缓冲失效**：
    - 若Page Cache中待写入磁盘的数据被释放，应用可能被迫同步写盘（如`fsync`），增加I/O等待。
- **只使用Swap**：匿名内存频繁换入/换出导致抖动（Thrashing）。
  - 换出（Swap Out）：当物理内存不足时，操作系统会将一些不常用的内存页从RAM中移到Swap空间，这个过程称为换出。
  - 换入（Swap In）：当需要访问那些已经被换出到Swap空间的内存页时，操作系统会将它们从Swap空间移回RAM，这个过程称为换入。
  - 抖动（Thrashing）：操作系统在内存管理过程中，频繁地进行换入/换出操作，导致系统性能急剧下降的现象。当系统陷入抖动状态时，大部分CPU时间都用于处理内存的换入/换出，而真正用于执行用户程序的时间很少。
    <img src="https://ccwu-1316557530.cos.ap-guangzhou.myqcloud.com/image-20250328201448680.png" alt="image-20250328201448680" style="zoom: 25%;"/>
    - **处理缺页中断：** CPU 被频繁地中断去处理 page fault。
    - **执行内核调度代码：** 内核需要决定哪个进程暂停、哪个进程运行、哪个页面被换出、哪个页面被换入。这些调度和管理工作非常耗费 CPU。
    - **驱动磁盘 I/O：** 换入换出操作本质上是大量的磁盘读写操作。CPU 需要花费大量时间等待磁盘响应（等待 I/O 完成）和处理磁盘控制器发出的中断（I/O 完成通知）。
    - **维护内存管理数据结构：** 跟踪哪些页在内存、哪些在 Swap、页的属性等信息也需要 CPU 周期。
##### 2.3.3 Swappiness 参数的作用
- **计算公式**：
  ```
  anon_prio = swappiness
  file_prio = 200 - anon_prio
  ```
  - **`anon_prio`**：匿名内存的释放权重（Swap优先级）。
  - **`file_prio`**：Page Cache的释放权重。
- **不同Swappiness值的行为**
  | **Swappiness** | **释放比例（匿名内存:Page Cache）** | **策略**                                                     |
  | :------------- | :---------------------------------- | :----------------------------------------------------------- |
  | **60（默认）** | 60:140                              | 优先释放Page Cache，适度使用Swap。                           |
  | **100**        | 100:100                             | 等比例释放匿名内存和Page Cache。                             |
  | **0**          | 0:200                               | **尽量不使用Swap**，优先释放Page Cache；内存极度不足时仍会触发Swap换出。 |
- **合理配置Swappiness**可避免性能抖动，需结合业务场景权衡：
  - **高I/O负载**：降低Swappiness（优先释放Page Cache）。
  - **匿名内存占用大**：提高Swappiness（利用Swap扩展内存）。
#### 2.4 总结
为了在一台宿主机上实现部分容器可以使用 swap  分区以防止内存突然暴涨而被杀死，同时禁止其他容器使用 swap 分区以严格限制其内存占用，可以采取以下措施：
1. 在宿主机开启 swap 分区
2.  在系统设置 `swappiness` 参数，控制 swap 分区，即匿名内存与 pagecache 释放的比重
3. 然后针对那些想禁止使用swap分区的容器，把它的memory.swappiness参数设置为0
   ```sh
   docker run -it --memory-swappiness=0 ubuntu:16.04 /bin/bash
   ```
## 七、资源类型与OOM
- 可压缩资源（如CPU）
  - **超限处理**：限制使用率，不终止进程（通过时间片调度）。
- 不可压缩资源（如内存、磁盘）
  - **超限处理**：触发OOM，强制终止进程。
- K8S关于容器的资源限制
  | 资源类型 | Request                  | Limit                                |
  | -------- | ------------------------ | ------------------------------------ |
  | CPU      | 设置`cpu.shares`         | 设置`cpu.cfs_quota_us/period_us`比值 |
  | 内存     | 调度依据（不修改cgroup） | 设置`memory.limit_in_bytes`          |
